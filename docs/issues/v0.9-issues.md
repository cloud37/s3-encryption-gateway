## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider ✅ DONE
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) — either an AES-256 master key or an RSA key pair — without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

### Affected
- `internal/crypto/keymanager_selfcontained.go` (new) — AES-256-GCM KEK wrap + RSA-OAEP KEK wrap implementations
- `internal/crypto/keymanager.go` — register new provider types
- `internal/config/config.go` — new `self_contained` KMS provider section
- `docs/KMS_COMPATIBILITY.md` — document new provider
- `docs/adr/ADR-XXXX-self-contained-envelope.md` (new)

### Implementation Guide

#### Phase 1 — AES-KEK Provider
- [x] Implement `AESKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: generate random DEK → wrap with AES-256-GCM using the KEK → store `{nonce, ciphertext}` in `KeyEnvelope.Ciphertext`.
  - `UnwrapKey`: decrypt `KeyEnvelope.Ciphertext` with KEK → return plaintext DEK.
  - `HealthCheck`: validate KEK is loaded and correct length (32 bytes).
  - `ActiveKeyVersion`: return configured key version integer.
  - `Close`: zeroize KEK from memory.
- [x] Accept KEK as base64-encoded 256-bit key from config or environment variable reference (`${ENV_VAR}` syntax consistent with existing config).
- [x] Support multiple versioned KEKs (map of version → key) to allow dual-read rotation without re-encrypting objects.

#### Phase 2 — RSA-KEK Provider
- [x] Implement `RSAKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: wrap DEK with RSA-OAEP (SHA-256) using the RSA public key.
  - `UnwrapKey`: decrypt envelope using RSA private key.
  - `HealthCheck`: validate key pair consistency (modulus match).
  - Minimum key size: 2048 bits; recommended: 4096 bits.
- [x] Accept PEM-encoded public + private key from config or environment variable references.
- [x] Support key versioning via fingerprint (SHA-256 of public key modulus).

#### Phase 3 — Key Rotation Support
- [x] Implement `RotatableKeyManager` for AES-KEK provider: `PrepareRotation` / `PromoteActiveVersion`.
- [x] Document rotation runbook: add new KEK version → promote → re-wrap objects on next write (lazy rotation).
- [x] Admin API endpoint to trigger rotation (reuse existing `internal/admin` rotation infrastructure).

#### Phase 4 — Configuration & Docs
- [x] Add `self_contained` section to config schema:
  ```yaml
  kms:
    provider: self_contained
    self_contained:
      type: aes        # or rsa
      key_version: 1
      keys:
        - version: 1
          aes_key: "${AES_KEK_KEY}"   # base64 256-bit
  ```
- [x] Update `docs/KMS_COMPATIBILITY.md` with provider matrix entry.
- [x] Write ADR documenting the design rationale, security properties, and comparison with external KMS.
- [x] Add getting-started example to `README.md`.

### Security Considerations
- The KEK must never be logged or included in error messages; enforce with linter rule or code review.
- Zeroize the KEK from memory on `Close()` and after each `WrapKey`/`UnwrapKey` operation (consistent with existing `zeroBytes` convention in `internal/crypto/engine.go`).
- If the KEK is lost, all wrapped DEKs are irrecoverable — document this clearly and recommend key backup procedures.
- RSA variant: use OAEP with SHA-256 (not PKCS#1 v1.5); enforce 2048-bit minimum.
- AES variant: derive a separate HMAC key from the KEK via HKDF for any integrity checks; do not reuse the encryption key for authentication.
- Support environment-variable-only key injection to avoid keys on disk in container deployments.

### Tests
- Unit: AES-KEK wrap/unwrap round-trip; wrong KEK returns `ErrUnwrapFailed`; Close zeroizes; version mismatch returns `ErrKeyNotFound`.
- Unit: RSA-KEK wrap/unwrap round-trip; key-pair mismatch detected at init; minimum key size enforced.
- Unit: dual-version read (old KEK version wraps, new version cannot unwrap, old version can).
- Integration (Tier 2): full encrypt/decrypt cycle via engine with AES-KEK provider against MinIO.
- Integration: key rotation promotion — objects written with v1 still decrypt after v2 promoted as active.
- Fuzz: `WrapKey`/`UnwrapKey` with malformed `KeyEnvelope.Ciphertext`.

### Definition of Done
- [x] `AESKEKManager` and `RSAKEKManager` implement `KeyManager` interface and pass all unit tests.
- [x] `AESKEKManager` implements `RotatableKeyManager`; rotation path tested end-to-end.
- [x] Tier 2 conformance tests pass with `self_contained` provider against MinIO and Garage.
- [x] Config schema updated; environment-variable injection works.
- [x] ADR authored and merged.
- [x] `docs/KMS_COMPATIBILITY.md` updated with provider entry.
- [x] No KEK material appears in any log output (verified by log-scraping test).

### Known Limitations
- RSA key rotation requires generating a new key pair and re-distributing the public key; no automated rotation path.
- Self-contained mode offers no HSM boundary — key material lives in process memory. Operators requiring HSM-grade isolation should use the existing Cosmian KMIP or HSM adapter.

## [V1.0-CRYPTO-2] Valkey At-Rest Encryption for Multipart Upload State ✅ DONE
- Owner: Crypto / Security
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`
- Status: **✅ VERIFIED COMPLETE**

### Summary
Add optional at-rest encryption for the full multipart-upload state blob persisted in Valkey.

Today, `UploadState` (including the wrapped DEK envelope, IV prefix, bucket/key names,
and per-part metadata) is stored in Valkey as plaintext JSON. The DEK itself is already
wrapped by the `KeyManager` before being written, so an attacker with Valkey read access
cannot recover the object-encryption key without also compromising the KMS/HSM or gateway
password. However, the remainder of the state — bucket names, object keys, IV prefixes,
and part-level metadata — is visible to anyone with access to Valkey memory dumps, RDB/AOF
files, or replication streams. This is an at-rest disclosure gap that TLS on the wire does
not close.

### Threat Model
- **Attacker with Valkey read access** (e.g., compromised Valkey replica, stolen backup,
  insider with `ACL` access) can enumerate in-flight multipart uploads, learn bucket/key
  names, and collect wrapped DEK envelopes for later offline analysis.
- **Attacker with Valkey write access** can tamper with part records or swap wrapped DEKs
  (existing tamper-detection in `UnwrapKey` AAD catches some variants, but not all).

Encrypting the entire state blob at rest removes this read-disclosure vector entirely.

### Affected
- `internal/mpu/state.go` — encrypt before `HSET`, decrypt after `HGET`
- `internal/config/config.go` — new `ValkeyEncryptionPasswordEnv` config field
- `internal/crypto/` — new `aesgcm.Seal` / `aesgcm.Open` helper or reuse existing
  `passwordKeyManager` machinery
- `docs/adr/0009-encrypted-multipart-uploads.md` — update threat-model and security
  considerations
- `docs/plans/V1.0-CRYPTO-2-plan.md` (new) — detailed implementation plan

### Implementation Guide

#### Phase 1 — Encryption Layer Design
- [x] Define a versioned ciphertext format for the at-rest blob:
  ```
  version(1 byte) || nonce(12 bytes) || ciphertext(... ) || tag(16 bytes)
  ```
  Version starts at `0x01` (AES-256-GCM). Leave room for future AEAD algorithms.
- [x] Derive a dedicated AEAD key from the configured password via HKDF-SHA256:
  ```
  key = HKDF-SHA256(salt="s3eg-mpu-state-v1", ikm=password)
  ```
  This isolates the Valkey encryption key from object-encryption keys.
- [x] Add `EncryptState([]byte) ([]byte, error)` and `DecryptState([]byte) ([]byte, error)`
  methods on `ValkeyStateStore`. Encrypt the full JSON blob before `HSET`; decrypt
  after `HGET` / `HGETALL`.

#### Phase 2 — Configuration
- [x] Add `EncryptionPasswordEnv string` to `ValkeyConfig` in `internal/config/config.go`:
  ```yaml
  multipart_state:
    valkey:
      # ... existing fields ...
      # Optional: dedicated password for at-rest encryption of multipart state in Valkey.
      # If unset, falls back to the gateway's main encryption password.
      # Recommended in production to achieve isolation between object-encryption keys
      # and Valkey storage encryption.
      encryption_password_env: "VALKEY_ENCRYPTION_PASSWORD"
  ```
- [x] Env var: `VALKEY_ENCRYPTION_PASSWORD` (optional). If absent, derive the AEAD key
  from `ENCRYPTION_PASSWORD` (gateway main password) with a distinct HKDF info string
  so the two key hierarchies never collide.
- [x] Config validation: if `EncryptionPasswordEnv` is set but the env var is empty,
  fail closed at startup with a clear error.

#### Phase 3 — Backwards Compatibility
- [x] **Unencrypted read path:** On `Get`/`List`, if decryption fails with an AEAD tag
  error, attempt to parse the blob as raw JSON (legacy unencrypted state). If that
  succeeds, treat it as unencrypted and log a one-time migration warning.
- [x] **Migration strategy:** Existing deployments can be migrated lazily:
  - Old state reads transparently (fallback path above).
  - New state writes are encrypted.
  - Orphaned old state expires via TTL naturally.
- [x] Add a config flag `valkey.encrypt_state` (default `true` in v1.0; `false` preserves
  backwards compatibility for operators who need time to rotate). Deprecate `false` in
  v1.1 and remove in v2.0.

#### Phase 4 — Metrics & Observability
- [x] Prometheus counter `gateway_mpu_state_encrypted_writes_total`
- [x] Prometheus counter `gateway_mpu_state_legacy_reads_total` (tracks fallback reads)
- [x] Log warning on first legacy read per gateway process: "Unencrypted Valkey state
  detected — enable valkey.encrypt_state=true"

#### Phase 5 — Documentation
- [x] Update ADR-0009 "Security Considerations" to reflect that at-rest encryption is
  now available and recommended.
- [x] Update runbook: `docs/RUNBOOK.md` — add "Valkey state at-rest encryption" section.
- [x] Update deployment docs: mention `VALKEY_ENCRYPTION_PASSWORD` in secret management
  checklist.

### Tests
- [x] Unit: round-trip encrypt/decrypt of `UploadState` JSON; ciphertext is non-deterministic
  (new nonce per write); tampered ciphertext returns error.
- [x] Unit: legacy unencrypted blob parses correctly via fallback path.
- [x] Unit: `HKDF` derivation produces different keys for `ENCRYPTION_PASSWORD` vs
  `VALKEY_ENCRYPTION_PASSWORD` even when the literal password is identical.
- [x] Unit: missing `VALKEY_ENCRYPTION_PASSWORD` env var fails startup when
  `EncryptionPasswordEnv` is configured.
- [x] Integration (miniredis): full `Create` → `Get` → `AppendPart` → `Delete` cycle with
  encryption enabled; raw hash value does not contain plaintext bucket/key names.
- [x] Fuzz: malformed ciphertext headers, wrong versions, truncated blobs — all rejected.

### Security Considerations
- **Key isolation:** Using a distinct `VALKEY_ENCRYPTION_PASSWORD` (recommended) ensures
  that a compromise of the Valkey encryption key does not automatically compromise
  object-encryption keys, and vice-versa.
- **Nonce uniqueness:** AES-GCM nonce is 96-bit random per write. Probability of collision
  is negligible (< 2^-32) for the expected lifetime state-write volume.
- **No key-derivation on hot path:** Derive the AEAD key once at `ValkeyStateStore`
  construction and cache it in the struct; do not re-run HKDF on every read/write.
- **Zeroize:** Clear the derived AEAD key from memory on `Close()` using `zeroBytes()`.
- **Version byte:** Future algorithms can be introduced by bumping the version byte;
  old gateways with a newer version will fail closed with a clear error rather than
  silently mis-decrypt.

### Definition of Done
- [x] `UploadState` blobs are encrypted at rest when `valkey.encrypt_state=true`.
- [x] Fallback read path supports legacy unencrypted state with a warning.
- [x] `VALKEY_ENCRYPTION_PASSWORD` is supported and documented.
- [x] All unit and integration tests pass.
- [x] ADR-0009 and runbook updated.
- [x] No plaintext bucket/key/IV metadata visible in Valkey `HGETALL` output when
  encryption is enabled (verified by test scraping raw miniredis values).

## [V1.0-CRYPTO-3] Encrypted Object Metadata ✅ DONE
- Owner: Crypto / Security
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
Encrypt gateway-generated encryption metadata before storing it in S3 object headers.

Today every encrypted object stores algorithm, salt, IV, KDF parameters, chunk manifest,
original size, ETag, and compression parameters as cleartext S3 user metadata. An attacker
with S3 read access can enumerate objects, observe encryption structure, and mount offline
dictionary attacks with full knowledge of the salt and KDF cost. Because the gateway is
open-source, PBKDF2 iteration counts and derivation logic are public; adding a second
password-derived layer would not increase brute-force resistance.

Encrypting the metadata blob with a **256-bit random key** (file-stored or KMS-wrapped)
removes this vector. An S3-only attacker sees only:
- `x-amz-meta-encrypted: true` (still needed for `IsEncrypted` detection)
- `x-amz-meta-enc-metadata: <opaque base64 blob>` (AES-256-GCM, 12-byte nonce per object)
- User-supplied `x-amz-meta-*` headers (left visible by design)

### Threat Model
- **Attacker with S3 read access** (compromised/minion key, insider, public bucket
  misconfiguration) learns nothing about encryption parameters, chunk structure, original
  sizes, or whether KMS was used.
- **Attacker with S3 read + gateway config** still needs the 256-bit metadata key (or KMS
  access) to decrypt the metadata blob.

### Affected
- `internal/crypto/engine.go` — encrypt metadata subset at write; decrypt at read
- `internal/crypto/metadata_encrypt.go` (new) — AES-256-GCM seal/open helpers
- `internal/crypto/engine_options.go` — `WithMetadataKey`, `WithWrappedMetadataKey`
- `internal/crypto/metadata_compaction.go` — add `x-amz-meta-em` compacted alias
- `internal/config/config.go` — `MetadataEncryptionKeyFile`, `MetadataEncryptionKey`
- `cmd/server/main.go` — load key, optionally wrap via `KeyManager`
- `docs/plans/V1.0-CRYPTO-3-plan.md` (new) — detailed implementation plan

### Key Format (exactly one must be set)

**Option 1 — File (preferred):**
```yaml
encryption:
  metadata_encryption_key_file: "/etc/s3eg/metadata-master.key"
```
File contents: base64-encoded 32 random bytes (exactly 32 bytes after decoding).
File should have mode `0600`.

**Option 2 — Inline (for Helm / Vault / sealed-secret deployments):**
```yaml
encryption:
  metadata_encryption_key: "gKf3zB0q..."   # ≥ 128 ASCII characters
```
The literal string is SHA-256 hashed to a 32-byte key. The 128-char minimum is a
policy guardrail to prevent low-entropy passwords; cryptographically the bottleneck
is always the 256-bit AES key.

**Option 3 — KMS wrapping (when `key_manager` is enabled):**
If a `KeyManager` is configured, the gateway generates or loads the metadata key,
wraps it via `KeyManager.WrapKey`, and persists the envelope. On startup the envelope
is unwrapped via `KeyManager.UnwrapKey`. The plaintext key never leaves gateway memory.

### Metadata Fallback Compatibility

The encrypted blob is just another metadata value. When metadata exceeds provider limits:
- **V1 fallback:** blob is embedded inside the object body and sealed under the data key
  → attacker needs both metadata key and data key.
- **V2 fallback:** blob is embedded in cleartext before chunked ciphertext → but the blob
  itself is AES-GCM encrypted, so still opaque without the metadata key.

No logic changes to fallback paths.

### Backwards Compatibility

- Old objects without `x-amz-meta-enc-metadata` / `x-amz-meta-em`: plaintext path forever.
- New objects with the option enabled: encrypted path.
- Can be toggled on/off per gateway instance without breaking existing objects.
- No migration of existing objects required.

### Security Considerations

- **Key loss = data loss:** Losing the metadata master key means losing access to the
  encryption parameters (salt, IV, algorithm, manifest). Without these, objects become
  undecryptable even if the main password is known. The key file must be backed up
  securely (offline, encrypted, geographically separate) and replicated to every HA node.
- **Key isolation:** When KMS wrapping is used, the plaintext metadata key only lives
  transiently in gateway memory. A host compromise reveals the key; an S3-only compromise
  does not.
- **Nonce uniqueness:** AES-GCM nonce is 96-bit random per object. Collision probability
  is negligible (< 2^-32) for the expected object volume.
- **No KDF on hot path:** The metadata key is derived once at engine construction
  (or unwrapped from KMS once at startup), not per-object.
- **Zeroize:** Clear the metadata key from memory on engine shutdown.
