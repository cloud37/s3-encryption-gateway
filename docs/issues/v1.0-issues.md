## [V1.0-S3-4] Protect Gateway-Owned MPU Metadata From Client Deletion
- Owner: S3 Compatibility
- Priority: P1
- Status: **OPEN — follow-up from issue #217**
- Labels: `milestone:v1.0`, `area:compat`, `area:mpu`

### Summary
Encrypted multipart objects depend on gateway-owned `.mpu-manifest` companion
objects. Client-facing listings no longer expose these objects, but direct
DeleteObject and DeleteObjects requests can still remove them and make the
associated encrypted object unreadable.

### Scope
- Reject or otherwise guard client deletion of a manifest while its main object exists.
- Preserve internal manifest cleanup when the main object is deleted.
- Define an explicit administrative/recovery path for intentional manifest cleanup.
- Add unit and Tier 2 conformance coverage for single and batch deletion.

## [V1.0-CONFIG-1] Per-Bucket Encryption Bypass + Flat ENV Policy Configuration ✅ DONE
- Owner: Config / DX / Crypto
- Priority: P2
- Status: **✅ VERIFIED COMPLETE**
- Labels: `milestone:v1.0`, `area:config`, `area:crypto`, `area:dx`
- Plan: `docs/plans/V1.0-CONFIG-1-plan.md`
- GitHub Issue: https://github.com/cloud37/s3-encryption-gateway/issues/196

### Summary
The gateway currently encrypts **every** object unconditionally. There is no mechanism to tell the gateway "do not encrypt data going into this bucket." The policy system (`PolicyConfig`) allows per-bucket key/algorithm selection and MPU opt-out, but has no `disable_encryption` flag. The only existing bypass-like mechanism, `encrypt_multipart_uploads: false`, only affects the Valkey/DEK MPU path — single-PUT objects are still encrypted.

The user in issue #196 wants to store Restic backups (already encrypted client-side) without a second encryption layer. Without a bypass, they must run two separate gateway instances.

This issue introduces:
1. **`disable_encryption: true`** in `PolicyConfig` — makes the gateway store and serve objects as plain bytes for matching buckets.
2. **`PassthroughEngine`** — a no-op `crypto.EncryptionEngine` (Encrypt = identity; Decrypt = identity for plaintext objects, error for encrypted objects).
3. **Flat indexed env vars** (`GW_POLICY_N_*`) covering **all** `PolicyConfig` fields, modelled after the existing `GW_CRED_N_*` pattern — no file mount required, works in Kubernetes/Docker/Compose.

### Affected
- `internal/config/policy.go` — `DisableEncryption bool` field; `BucketDisablesEncryption()`; updated `BucketEncryptsMultipart`; conflict validation; `Reset()`; `LoadPoliciesFromEnv()`
- `internal/crypto/passthrough.go` — new: `PassthroughEngine` + `ErrEncryptedObjectInBypassBucket`
- `internal/crypto/engine.go` — extract `IsEncryptedMetadata` as package-level function
- `internal/api/handlers.go` — `getEncryptionEngine` bypass branch; error translation for `ErrEncryptedObjectInBypassBucket`
- `internal/config/config.go` — `loadFromEnv` gains `GW_POLICY_N_*` parsing loop
- `cmd/server/main.go` — env-policy wiring at startup and hot-reload
- `config.yaml.example` — `disable_encryption` annotation; `GW_POLICY_N_*` usage example
- `helm/s3-encryption-gateway/values.yaml` — document `disable_encryption` and `GW_POLICY_N_*`

### ENV var contract (all PolicyConfig fields)
```
GW_POLICY_N_ID                        string   (required to define policy N)
GW_POLICY_N_BUCKETS                   comma-separated glob patterns
GW_POLICY_N_DISABLE_ENCRYPTION        true|false
GW_POLICY_N_REQUIRE_ENCRYPTION        true|false
GW_POLICY_N_DISALLOW_LOCK_BYPASS      true|false
GW_POLICY_N_ENCRYPT_MULTIPART_UPLOADS true|false
GW_POLICY_N_ENCRYPTION_PASSWORD       string
GW_POLICY_N_ENCRYPTION_ALGORITHM      string
GW_POLICY_N_RATE_LIMIT_ENABLED        true|false
GW_POLICY_N_RATE_LIMIT_REQUESTS       int
GW_POLICY_N_RATE_LIMIT_WINDOW         duration (e.g. 1m)
```
N is 0-based; iteration stops when `GW_POLICY_N_ID` is absent (same sentinel logic as `GW_CRED_N_*`).

### Implementation Guide
- [x] Add `DisableEncryption bool` to `PolicyConfig`.
- [x] Add `BucketDisablesEncryption(bucket string) bool` and `Reset()` to `PolicyManager`.
- [x] Add `LoadPoliciesFromEnv() error` to `PolicyManager`; iterates `GW_POLICY_N_*` vars.
- [x] Update `BucketEncryptsMultipart`: return `false` when `DisableEncryption` is true.
- [x] Reject `disable_encryption: true` + `require_encryption: true` in the same policy at load time.
- [x] Implement `PassthroughEngine` in `internal/crypto/passthrough.go`.
- [x] In `getEncryptionEngine`, add early-return for `policy.DisableEncryption`.
- [x] Translate `ErrEncryptedObjectInBypassBucket` → HTTP 409 `EncryptionConfigurationMismatch` in handlers.
- [x] In `loadFromEnv`, call `config.PolicyManager.LoadPoliciesFromEnv()` (or store raw env-derived `PolicyConfig` slice on `Config` for later loading).

### Tests

- [x] `TestPassthroughEngine_Decrypt_EncryptedObject_ReturnsError`
- [x] `TestBucketDisablesEncryption_MatchAndDefault`

## [V1.0-S3-3] ListObjects Plaintext Size Translation ✅ DONE

- Owner: API / S3 Compatibility
- Priority: P1
- Status: **✅ VERIFIED COMPLETE**
- Plan: `docs/plans/V1.0-S3-3-plan.md`

### Summary

Delivers correct `ListObjects` sizes (`listedSize == headSize`) for all
encrypted objects in steady-state warm-cache operation, with zero additional
backend HEAD calls in the warm-cache path. Backed by a per-bucket Valkey hash
(`plainsize:<bucket>`). Write-through index populated on `PutObject`,
`CompleteMultipartUpload`, `CopyObject`; evicted on `DeleteObject` /
`DeleteObjects`. Opt-in fallback HEAD batch for cache misses.
