# Milestone v1.0 — Issue Tracker and Implementation Guide

This document tracks issues for v1.0 with implementation guides and validation criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Full KMS production readiness (HA, caching, failure modes); sensitive data zeroization and constant-time comparisons audit
- S3 Compatibility: Broad feature parity (tagging, ACLs, lifecycle headers passthrough); SDK/tool compatibility certification matrix
- Performance & Scalability: Horizontal scale guidance; tuned HPA/SLOs; high-throughput benchmarks
- Observability: Turnkey dashboards (Grafana) and alerting rules
- Operations: Multi-arch images, SBOMs, provenance (SLSA); Helm chart on Artifact Hub with automated checks
- Ecosystem: Additional backends via shims (GCS, Azure Blob, filesystem); CLI for local testing (presign, quick put/get)

## Milestone Acceptance Criteria
- Compatibility certification matrix completed for targeted SDKs/tools.
- Published dashboards and alerting rules; documented SLOs and autoscaling guidance.
- Supply-chain artifacts for releases (multi-arch, SBOM, provenance) available.

Labels: `milestone:v1.0`, `area:kms`, `area:compat`, `area:perf`, `area:observability`, `area:ops`, `area:ecosystem`.

## [V1.0-CLI-2] Gateway-Aware Audit Tooling — `s3eg-cli` (Read-Only) + Remove Gateway-Bypassing Migration ✅ DONE
- Owner: DevEx
- Priority: P3
- Status: **✅ VERIFIED COMPLETE**
- Plan: `docs/plans/V1.0-CLI-2-plan.md`
- Supersedes: V1.0-CLI-1 (rejected)

### Summary

Reshape the project-owned binary into a **pure read-only audit tool** named
`s3eg-cli` (renamed from `s3eg-migrate`) and **remove the gateway-bypassing
migration write-path entirely**.

The binary's only justifiable unique value is reading the **backend ciphertext
envelope** — information the gateway deliberately hides and that no generic S3
tool can surface. The three audit sub-commands (`inspect`, `verify-key`,
`list-algorithm`) deliver exactly this. The decrypt→re-encrypt→rewrite
*migration* logic, by contrast, is a full re-implementation of the gateway's
crypto stack operating *directly against the backend*, bypassing the gateway.
It is removed in favour of the supported re-encryption pattern below.

### Rationale — Why remove migration

1. **It bypasses the gateway.** `s3eg-migrate` connects directly to the
   backend store (`s3.NewClient(&cfg.Backend)` in `cmd/migrate/main.go`),
   rebuilds the entire crypto stack locally (`buildEngines`), and rewrites
   objects in place. This duplicates and must forever track the gateway's
   crypto evolution — exactly the "maintenance burden without unique value"
   that rejected V1.0-CLI-1.
2. **The gateway already exposes legacy objects.** Every legacy class
   (A/B/C/D) is readable through the gateway's fallback decrypt paths. So
   re-encryption needs no bespoke binary: **GET through the gateway**
   (fallback-decrypts) **→ PUT through the gateway** (re-encrypts with current
   parameters) using any S3 tool (`awscli`, `s5cmd`, `mc`).
3. **Read-only audit is the unique value.** Inspecting the on-backend envelope
   is the one thing no S3 tool can do; that is what the binary should focus on.

### Supported Re-Encryption Pattern (replaces migration)

```
# Re-encrypt every object under a prefix with the gateway's current params:
aws s3 cp s3://bucket/key - --endpoint-url $GATEWAY | \
  aws s3 cp - s3://bucket/key --endpoint-url $GATEWAY
# (or use s5cmd / mc; loop over `aws s3 ls`)
```

The gateway decrypts on GET via its fallback paths and re-encrypts on PUT.
This is documented in `docs/MIGRATION.md` (rewritten) and `docs/S3_CLI_TOOLS.md`.

### No-AAD Recovery Gap (must be solved)

The gateway only attempts the no-AAD fallback when
`x-amz-meta-enc-legacy-no-aad == "true"` (`internal/crypto/engine.go:818`).
Objects encrypted before that marker existed and never backfilled currently
**fail to decrypt through the gateway**, so the GET→PUT pattern cannot recover
them once `backfill-legacy-no-aad` is removed.

Solution: add a **gateway server config flag**
`encryption.allow_unmarked_no_aad_fallback` (default `false`, fail-closed).
When an operator sets it `true` during a controlled recovery window, the
gateway permits the no-AAD fallback for unmarked objects, enabling GET→PUT
re-encryption to recover them. The default preserves the SEC-4 security
property (attacker with backend write access cannot strip AAD).

### New Sub-Commands (read-only only)

#### `inspect <bucket> <key>`
Decode and print the ciphertext envelope stored on the **backend** object
(not the plaintext the gateway returns). Shows:
- Encryption algorithm (`AES256-GCM`, `ChaCha20-Poly1305`, FIPS variants)
- KMS provider / key ID / key version that wrapped the DEK (when present)
- AAD scheme (`v2-aad` default, `v1-no-aad` when `MetaLegacyNoAAD=true`)
- Migration class (`modern`, `class_a_xor`, `class_b_no_aad`, …) via the
  existing `ClassifyObject` logic
- IV / nonce length and value (hex), salt (hex)
- Ciphertext length and first 32 bytes (hex, for fingerprinting)
- All `x-amz-meta-encryption-*` headers stored on the object

Read-only: issues `HeadObject` (+ a small ranged `GetObject` for the
ciphertext fingerprint). Never writes.

#### `verify-key <bucket> <key> [--key-version N]`
Confirm that an object's DEK unwraps with a specific key version, without a
full body decryption. Read-only; reports success/failure and the actual
recorded key version (`x-amz-meta-encryption-key-version`).

#### `list-algorithm <bucket> [--prefix P] [--workers N]`
Scan all objects in a bucket (optionally scoped to a prefix) and report a
breakdown of which encryption algorithm / class each object uses. Read-only
`ListObjects` + `HeadObject` scan with a worker pool. Output:

```
algorithm           count   pct
AES256-GCM         12 400  98.4%
ChaCha20-Poly1305     198   1.6%
(plaintext)             2   0.0%
```

### Binary Rename

Rename `s3eg-migrate` → `s3eg-cli` (full rename of `cmd/migrate` → `cmd/cli`).
Update Makefile targets, `.github/workflows/helm.yml` build steps, `README.md`,
CHANGELOG, and docs. Ship a `s3eg-migrate` compatibility shim/alias during a
transition period that prints a deprecation notice and points at `s3eg-cli`
and the GET→PUT re-encryption pattern.

## [V1.0-MAINT-2] Remove Compression Feature ✅ DONE

- Owner: Crypto / Maintenance
- Priority: P2
- Status: **✅ VERIFIED COMPLETE**
- Plan: docs/plans/V1.0-MAINT-2-plan.md

### Summary

Drop the built-in compression feature to reduce maintenance burden. Compression has
never seen significant production use, is a second-class feature with a structural
gap (MPU parts are always stored uncompressed), and compressing before encrypting is
a known cryptographic anti-pattern (CRIME/BREACH-style plaintext leakage via
ciphertext size).

Users who need compressed and encrypted objects can compose two purpose-built tools:

```
client → s4 (https://github.com/abyo-software/s4) → s3-encryption-gateway → storage
```

This keeps the gateway focused on its core responsibility (encryption) and delegates
compression to a dedicated tool.

### Scope

- Remove `internal/crypto/compression.go` and `CompressionEngine` interface.
- Remove `compressionEngine` parameter from all `NewEngine*` constructors.
- Remove `CompressionConfig` from `internal/config/config.go` and related
  hot-reload validation.
- Remove compression metadata constants (`MetaCompression*`) and `IsCompressionMetadata`.
- Remove compression handling in `internal/api/handlers.go`, `cmd/server/main.go`,
  and `test/harness/gateway.go`.
- Remove the decompression bomb guard (V1.0-SEC-M05) — no longer needed.
- Remove the `compression` block from the Helm chart values schema.
- Provide a migration note: any objects stored with `x-amz-meta-compression-enabled: true`
  must be re-uploaded (decrypt via old gateway version, re-upload uncompressed) before
  upgrading to the version that removes compression support.

### Definition of Done

- [x] All compression code removed; build and `go vet` clean.
- [x] All compression-related tests removed or converted.
- [x] Migration note added to `docs/MIGRATION.md` covering existing compressed objects.
- [x] Helm chart schema updated; `helm lint` clean.
- [x] `README.md` / `docs/` updated to remove references to compression and document
      the recommended `s4 → gateway` composition pattern.
