# Milestone v1.0 — Issue Tracker and Implementation Guide

This document tracks issues for v1.0 with implementation guides and validation criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Full KMS production readiness (HA, caching, failure modes); sensitive data zeroization and constant-time comparisons audit
- S3 Compatibility: Broad feature parity (tagging, ACLs, lifecycle headers passthrough); SDK/tool compatibility certification matrix
- Performance & Scalability: Horizontal scale guidance; tuned HPA/SLOs; high-throughput benchmarks
- Observability: Turnkey dashboards (Grafana) and alerting rules
- Operations: Multi-arch images, SBOMs, provenance (SLSA); Helm chart on Artifact Hub with automated checks
- Ecosystem: Additional backends via shims (GCS, Azure Blob, filesystem); CLI for local testing (presign, quick put/get)

## Milestone Acceptance Criteria
- Compatibility certification matrix completed for targeted SDKs/tools.
- Published dashboards and alerting rules; documented SLOs and autoscaling guidance.
- Supply-chain artifacts for releases (multi-arch, SBOM, provenance) available.

The proposed commands (`presign`, `put`, `get`, `head`) are fully covered by
existing S3-compatible tools (`awscli`, `s5cmd`, `mc`). The gateway is
S3-compatible by design; no wrapper binary is needed. Shipping a bespoke
binary for generic operations adds maintenance and release burden without
unique value.

## [V1.0-CLI-2] Gateway-Aware Tooling — Migrate Binary Improvements & Rename
- Owner: DevEx
- Priority: P3
- Status: **OPEN**
- Supersedes: V1.0-CLI-1 (rejected)

### Summary

Extend `s3eg-migrate` with sub-commands that provide gateway-specific
inspection and audit capabilities unavailable in any generic S3 tool. Explore
renaming the binary to `s3eg-cli` to reflect its multipurpose scope, while
retaining all existing migration functionality.

### Background

Standard S3 tools (`awscli`, `s5cmd`, `mc`) cover generic operations against
the gateway. The unique value of a project-owned binary lies in understanding
the gateway's internal encryption metadata format — things no S3-compatible
tool can surface. This issue extends `s3eg-migrate` along that axis rather
than reinventing generic operations.

### New Sub-Commands

#### `inspect <bucket> <key>`
Decode and print the ciphertext envelope stored on the **backend** object
(not the plaintext the gateway returns). Shows:
- Encryption algorithm (`AES-GCM-256`, `AES-GCM-256-FIPS`, …)
- Key ID / key version used to encrypt the DEK
- AAD scheme (`v1-no-aad`, `v2-aad`, …)
- IV / nonce length and value (hex)
- Ciphertext length and first 32 bytes (hex, for fingerprinting)
- Encryption metadata headers stored as S3 object metadata

Useful for: verifying that the gateway actually encrypted the object,
confirming which key was used, auditing AAD scheme before migration.

#### `verify-key <bucket> <key> [--key-version N]`
Confirm that an object was encrypted with a specific key version without
performing a full decryption. Loads the envelope header, attempts DEK
unwrap with the specified key, and reports success or failure.

Useful for: key rotation audits, confirming a particular key owns a
specific object, pre-migration validation.

#### `list-algorithm <bucket> [--prefix P] [--workers N]`
Scan all objects in a bucket (optionally scoped to a prefix) and report
a breakdown of which encryption algorithm each object uses. Output:

```
algorithm              count   pct
AES-GCM-256            12 400  98.4%
AES-GCM-256-FIPS          200   1.6%
legacy-v1-no-aad            2   0.0%
```

Useful for: pre-migration planning, verifying uniformity after migration,
compliance audits confirming no plaintext objects remain.

### Binary Rename Consideration

Rename `s3eg-migrate` → `s3eg-cli` to signal that the binary is a
multipurpose gateway companion tool, not solely a migration tool. The migrate
sub-commands are preserved verbatim; the default (no sub-command) behaviour
and all flags remain unchanged. Update Makefile targets, CI upload steps,
`docs/MIGRATION.md`, and `README.md` accordingly. Ship `s3eg-migrate` as a
compatibility symlink or alias during a transition period.

### Implementation Guide
- [ ] Add `inspect` sub-command to `cmd/migrate/main.go` (or `cmd/cli/main.go` post-rename).
- [ ] Add `verify-key` sub-command.
- [ ] Add `list-algorithm` sub-command with worker pool reusing existing migration pool pattern.
- [ ] All new sub-commands accept `--config`, `--log-level`, `--output` (text/json) global flags consistent with existing conventions.
- [ ] Decide on rename: open a discussion or PR; if approved update Makefile, CI, README, docs.
- [ ] Add unit tests for envelope parsing logic (mock backend objects).
- [ ] Add integration tests against MinIO using `//go:build integration` tag.
- [ ] Update `docs/MIGRATION.md` to reference new sub-commands.
- [ ] Add `docs/S3_CLI_TOOLS.md` cross-reference in the new sub-command help text.
