## [V1.0-MGMT-2] KDF Iteration Migration via s3eg-migrate ✅ DONE
- Owner: Crypto / Ops
- Priority: P2
- Labels: `area:crypto`, `area:ops`, `area:migration`

### Summary
Provide a migration path for objects encrypted before V1.0-SEC-H03 (PBKDF2-SHA256 at 100,000 iterations) to be re-encrypted at the operator's configured iteration count (default 600,000). Builds on top of the existing `s3eg-migrate` framework introduced in V1.0-MAINT-1.

### Background
V1.0-SEC-H03 raises the default PBKDF2-SHA256 iteration count from 100,000 to 600,000 (NIST SP 800-132 (2023) floor), introduces the `encryption.kdf.pbkdf2.iterations` config key, and stores the KDF algorithm + parameters in a new object metadata field `x-amz-meta-encryption-kdf-params`. Objects lacking this field are transparently decrypted using the legacy 100,000 iterations via a fallback path. V1.0-MGMT-2 provides the tooling to actively upgrade those legacy objects.

### Affected
- `internal/migrate/classifier.go` — new `ClassD_LegacyKDF` class
- `internal/migrate/migrator.go` — `FilterKDF` constant; `SourceIterations` / `TargetIterations` fields
- `docs/MIGRATION.md` — "Upgrading KDF parameters" runbook section

### Implementation Guide
- [x] Add `ClassD_LegacyKDF` to `classifier.go`: encrypted object with no `MetaKDFParams` field
- [x] Update `NeedsMigration` to include `ClassD_LegacyKDF`
- [x] Add `FilterKDF MigrationClassFilter = "kdf"` to `migrator.go`
- [x] Add `SourceIterations int` and `TargetIterations int` to `Migrator` struct
- [x] Document usage in `docs/MIGRATION.md`:
  ```
  s3eg-migrate --bucket my-bucket --filter kdf --dry-run
  s3eg-migrate --bucket my-bucket --filter kdf --workers 8 --verify
  ```

### Tests
- [x] `TestClassifyObject_ClassD_LegacyKDF` — encrypted + no `MetaKDFParams` → ClassD
- [x] `TestClassifyObject_Modern_WithKDFParams` — encrypted + `MetaKDFParams` present → ClassModern
- [x] `TestFilterKDF_AllowsDenyCorrectly` — ClassD passes filter; ClassModern is skipped

### Definition of Done
- [x] `s3eg-migrate --filter=kdf --dry-run` correctly counts ClassD objects
- [x] `s3eg-migrate --filter=kdf` re-encrypts ClassD objects with the target engine; all pass `--verify`
- [x] After migration, `--filter=kdf --dry-run` reports 0 ClassD objects
- [x] Runbook documented in `docs/MIGRATION.md`

## [V1.0-AUTH-1] Gateway-Managed Credential Store & Unified Authentication Refactor ✅ DONE
- Owner: Security / API
- Priority: P0
- Labels: `milestone:v1.0`, `area:security`, `area:api`, `area:config`, `breaking-change`

### Summary

Replace the current dual-mode (single-bucket / multi-bucket) authentication architecture with a unified model: the gateway maintains its own list of known access-key/secret-key pairs. Every request — whether it uses AWS Signature V4 (header or presigned URL) or AWS Signature V2/query-parameter style — must present a credential that the gateway recognises and can validate. Requests with no credentials, or with an unknown access key, are rejected before any backend interaction occurs.

The gateway's configured backend credentials (`backend.access_key` / `backend.secret_key`) determine the backend scope. When they grant access to a single bucket, the gateway naturally becomes single-bucket; when they grant access to multiple buckets, it is multi-bucket. The `proxied_bucket` option becomes a non-breaking optional filter (enforced if set, ignored if absent) rather than a mode switch. The `backend.use_client_credentials` passthrough mode is removed.

This resolves **CWE-306 (Missing Authentication for Critical Function)** identified in the data-plane security audit.

### Motivation

- Any client reachable at the network level can currently issue a bare `GET /bucket/key` — no credentials needed — and receive decrypted object content. The entire security perimeter is the network firewall.
- The original "network is the perimeter" design is now considered a liability, particularly as the gateway is increasingly deployed behind load-balancers and in multi-tenant Kubernetes clusters where pod-to-pod routes are less tightly controlled.
- The dual-mode model (single-bucket vs. multi-bucket) introduces unnecessary complexity and code that can be collapsed into a single, cleaner auth path.

### Affected Components

| Component | Change |
|-----------|--------|
| `internal/config/config.go` | Add `AuthConfig.Credentials []GatewayCredential`; add env/valueFrom support; deprecate `Backend.UseClientCredentials` |
| `internal/api/auth.go` | Add `CredentialStore` interface + `StaticCredentialStore` implementation |
| `internal/api/handlers.go` | Rewrite `getS3Client` to always validate against credential store (fail-closed) |
| `internal/api/credentials.go` | No structural change; `ExtractCredentials` is reused as-is |
| `internal/api/middleware.go` | New `AuthMiddleware` that runs credential lookup + signature validation before any handler |
| `cmd/server/main.go` | Wire `CredentialStore` into `Handler` at startup |
| `helm/s3-encryption-gateway/` | `values.yaml`, `values.schema.json`, Deployment template — credential list injection |
| `config.yaml.example` | Document new `auth.credentials` block |
| `docs/ARCHITECTURE.md` | Update auth-flow diagram |
| `docs/adr/` | New ADR for unified credential store |

### Implementation Guide

See `docs/plans/V1.0-AUTH-1-plan.md` for the comprehensive step-by-step implementation plan.

### Security Considerations

- Credential comparison (secret key lookup) MUST use `crypto/hmac.Equal` (constant-time) to prevent timing-based access-key enumeration.
- Credentials MUST NOT appear in any log output; sanitise `ExtractCredentials` result before logging.
- An empty credential list MUST cause a fail-closed startup error (no silent open-gateway mode).
- The `auth.credentials` list MUST support `valueFrom` references (env vars, mounted secrets) so that plaintext secrets never appear in `values.yaml` or `config.yaml` checked into version control.
- All three auth error classes (signature mismatch, unknown key, missing creds) continue to map to distinct S3 error codes — existing SDK retry/diagnostic behaviour is preserved.

### Tests

- Unit: `AuthMiddleware` rejects requests with no credentials.
- Unit: `AuthMiddleware` rejects requests with unknown access key.
- Unit: `AuthMiddleware` validates V4 signature using the stored secret; wrong signature returns `ErrSignatureMismatch`.
- Unit: `AuthMiddleware` validates V2/query-param signature; wrong signature returns `ErrSignatureMismatch`.
- Unit: constant-time comparison used for secret key validation (timing test).

- Unit: `proxied_bucket` filter — requests to other buckets return `AccessDenied`.
- Integration (Tier 2): full V4 PUT/GET/DELETE cycle with correct gateway credentials.

- Integration (Tier 2): bare unauthenticated `GET` returns `AccessDenied`.

## [V1.0-OPS-3] MPU Manifest Orphan Cleanup on Object Deletion ✅ DONE
- Owner: Ops / API
- Priority: P2
- Labels: `area:ops`, `area:s3`, `bug`

### Summary
When an encrypted multipart upload (MPU) object is deleted — either directly or via S3 lifecycle rotation — the corresponding `.mpu-manifest` metadata file is not removed from the backend. Over time, these orphaned manifest files accumulate in the bucket, consuming storage and polluting object listings.

### Background
Each MPU encrypted object is written alongside a companion manifest file stored at `<key>.mpu-manifest` in the same bucket. This manifest is used by the gateway during decryption to resolve DEK envelopes and part boundaries. The `DeleteObject` and `DeleteObjects` handlers in the gateway proxy the delete request to the backend but do not issue a corresponding delete for the manifest file.

### Affected
- `internal/api/handlers.go` — `DeleteObject` and `DeleteObjects` handlers
- `internal/s3/` — backend client delete calls
- Tests: `internal/api/` — delete handler tests

### Implementation Guide
- [x] In the `DeleteObject` handler: after a successful backend delete of the primary object, issue a second backend delete for `<key>.mpu-manifest`. Treat a `404` on the manifest delete as a no-op (non-MPU objects have no manifest).
- [x] In the `DeleteObjects` (batch delete) handler: for each successfully deleted object, collect manifest keys and issue a follow-up batch delete for the manifest files.
- [x] Log manifest cleanup at `debug` level; log unexpected errors (non-404) at `warn`.
- [x] Ensure the manifest cleanup does not cause the primary delete to appear failed if only the manifest delete errors.

### Tests
- [x] Unit: `DeleteObject` for an MPU-encrypted key also deletes `<key>.mpu-manifest`.
- [x] Unit: `DeleteObject` for a non-MPU key does not error when manifest 404s.
- [x] Unit: `DeleteObjects` batch cleans up all manifest files for deleted MPU objects.
- [x] Integration (Tier 2): upload encrypted MPU object → delete via gateway → verify manifest is absent from backend bucket.

### Definition of Done
- [x] `DeleteObject` and `DeleteObjects` handlers clean up companion `.mpu-manifest` files.
- [x] Manifest cleanup failures (non-404) are logged as warnings but do not propagate as primary delete errors.
- [x] All unit and Tier 2 integration tests pass.

### Cross-References
- Issue #135 (reporter: tihomal) — first observed during CNPG backup rotation
- `internal/api/handlers.go` — `DeleteObject` / `DeleteObjects` handler implementations

## [V1.0-S3-2] S3 API Feature Parity ✅ DONE
- Owner: API
- Priority: P1
- Labels: `milestone:v1.0`, `area:compat`, `area:api`

### Summary
The gateway currently handles ~18 of the S3 REST API operation surface. The vast majority
of the missing ~35+ operations are pure metadata/configuration subresources that can be
**proxied verbatim** to the backend using the existing `forwardSignatureV4Request`
infrastructure. A small subset (BucketEncryption, SelectObjectContent) require gateway-
specific interception. Without these operations, common tools (Terraform AWS provider,
rclone, boto3 lifecycle management, browser CORS preflight) fail with `405 Method Not
Allowed` or `404 Not Found`.

### Full Plan
See `docs/plans/V1.0-S3-2-plan.md` for the complete gap analysis, routing design,
proxy helper design, and phase breakdown.

### Affected
- `internal/api/handlers.go` — ~35 new handler methods + `RegisterRoutes` additions
- `internal/api/utils.go` — `copyProxyResponse`, `handlePassthrough` helpers
- `internal/api/handlers_test.go` — unit tests for all new handlers
- `test/conformance/s3_compat_test.go` — Tier 2 conformance tests
- `docs/S3_API_IMPLEMENTATION.md` — updated coverage matrix

### Security Considerations
- All proxy handlers must use `forwardSignatureV4Request` — never strip the `Authorization`

- `BucketEncryption` (`?encryption`) is proxied verbatim. The subresource controls

- `DeleteBucket` must emit an audit event regardless of backend response status.
- ACL handlers proxy verbatim; the gateway does not enforce S3 ACLs at the proxy layer.
