## 1. Context & Current State

### Existing KeyManager Infrastructure

The `KeyManager` interface is defined in `internal/crypto/keymanager.go`. It
exposes six methods: `Provider()`, `WrapKey()`, `UnwrapKey()`,
`ActiveKeyVersion()`, `HealthCheck()`, and `Close()`. The `KeyEnvelope` struct
(`internal/crypto/keymanager.go`) carries wrapped DEK bytes alongside
versioning metadata (`KeyID`, `KeyVersion`, `Provider`, `Ciphertext`,
`CreatedAt`).

Two production adapters exist today: the Cosmian KMIP adapter
(`internal/crypto/keymanager_cosmian.go`) and the in-memory AES-256 key-wrap
adapter (`internal/crypto/keymanager_memory.go`). Both delegate encryption to
an external server or use RFC 3394 key-wrap with a pre-shared master key
copied into memory at startup. Neither is suitable for fully air-gapped
deployments that want AEAD-based wrapping (AES-256-GCM) or asymmetric wrapping
(RSA-OAEP) with only local key material.

The `RotatableKeyManager` interface (`internal/crypto/keymanager.go`) extends
`KeyManager` with `PrepareRotation(ctx, *int) (RotationPlan, error)` and
`PromoteActiveVersion(ctx, RotationPlan) error`. The in-memory adapter already
implements this interface (`internal/crypto/keymanager_memory.go`), providing
the reference pattern for `AddVersion` / `PrepareRotation` /
`PromoteActiveVersion`.

### Registry

`internal/crypto/keymanager_registry.go` provides `Register(name, Factory)`
and `Open(ctx, name, cfg)`. The `init()` function there registers the `memory`
adapter; the `hsm` adapter is registered via build-tag-gated files. A new
`self_contained` adapter must follow this same pattern.

### Configuration

`internal/config/config.go` defines `KeyManagerConfig` with provider-specific
sub-structs (`Cosmian CosmianConfig`, `Memory MemoryKMConfig`). The comment at
`internal/config/config.go` reads:
```
// TODO(v1.0): Add AWS and Vault config fields when adapters are implemented
```
A new `SelfContained SelfContainedKMConfig` field must be added there.
`loadFromEnv()` in the same file handles env-var overrides explicitly; new
`SELF_CONTAINED_*` vars must be wired there.

### Why the Existing State Is Insufficient

The `memory` adapter (`internal/crypto/keymanager_memory.go`) uses RFC 3394
AES key-wrap. While cryptographically sound, it is optimized for the
single-node testing scenario: it auto-generates an ephemeral key if none is
provided and does not expose AES-GCM-based wrapping, which is more widely
implemented and provides authenticated encryption of the wrapped key. There is
no RSA-OAEP variant at all today, blocking operators who wish to use existing
PKI infrastructure to control DEK access without a network KMS call.

The issue (V1.0-KMS-4) targets the gap: provide a drop-in `KeyManager`
implementation that is fully self-contained (no external calls), supports both
symmetric (AES-256-GCM) and asymmetric (RSA-OAEP/SHA-256) wrapping, and
plugs directly into the existing engine and admin rotation infrastructure.

## 2. Design Goals & Non-Goals

### Goals

1. **Interface compliance** — `AESKEKManager` and `RSAKEKManager` both satisfy
   `KeyManager` at compile time. `AESKEKManager` additionally satisfies
   `RotatableKeyManager`. All contract tests in `ConformanceSuite` pass.

2. **AEAD wrapping** — `AESKEKManager` uses AES-256-GCM (not RFC 3394) for
   DEK wrapping, producing an authenticated ciphertext per wrap call with a
   fresh random 12-byte nonce. Tampering with the wrapped DEK returns
   `ErrUnwrapFailed`.

3. **Asymmetric wrapping** — `RSAKEKManager` uses RSA-OAEP with SHA-256. Minimum
   accepted key size is 2048 bits; the constructor returns an error for smaller
   keys. This allows operators to leverage existing PKI.

4. **Version-aware multi-key storage** — Both managers store a map of
   `version → key material` and try the envelope's declared version first on
   `UnwrapKey`, falling back to all known versions. This enables dual-read
   rotation without re-encrypting historical objects.

5. **Zeroization** — All key material is zeroized via `zeroBytes()` on
   `Close()`. AES KEK copies are also zeroized after each `UnwrapKey` call
   (stack copies only; the map copy persists until `Close()`).

6. **Env-var injection** — Both managers accept key material as
   `"env:VAR_NAME"` references, never requiring key bytes on disk in container
   deployments. The base64-encoded form is the canonical wire format.

7. **Registry integration** — Both adapters register under the name
   `"self_contained"` via `init()`. The factory selects `AESKEKManager` or
   `RSAKEKManager` based on a `"type"` field in the config map.

8. **No KEK exposure** — No KEK material appears in log output, error messages,
   or `KeyEnvelope` fields. A log-scraping test enforces this invariant.

### Non-Goals

1. **RFC 3394 key-wrap for the self-contained adapter** — The `memory` adapter
   already provides this. This plan introduces GCM-based wrapping only; RFC
   3394 is not replicated.

2. **RSA rotation automation** — RSA key rotation requires a new key pair and
   re-distribution; there is no automated `PromoteActiveVersion` path for RSA.
   Manual rotation documentation is sufficient (Phase D).

3. **FIPS-gated code paths** — FIPS-compliant Go crypto already uses the same
   `crypto/aes` and `crypto/rsa` primitives; no separate build-tag file is
   required for AES-GCM or RSA-OAEP. Tracked separately if FIPS certification
   is needed.

4. **AWS KMS / Vault Transit** — Covered by V1.0-KMS-2 and V1.0-KMS-3
   respectively.

5. **Automatic lazy re-wrapping of historical DEKs** — Post-rotation objects
   retain their original wrapped DEK; re-wrapping happens only on the next
   write. This is by design (lazy rotation) and is documented in the runbook.

## 3. Target Interface / Semantics / Behaviour

### 3.1 `AESKEKManager`

```go
// AESKEKManager wraps DEKs using AES-256-GCM with a locally-held KEK.
//
// Invariants:
//   - Thread-safe: all exported methods acquire m.mu (RLock for reads, Lock
//     for Close/PromoteActiveVersion).
//   - Context propagation: ctx.Err() is checked before acquiring any lock.
//   - Zeroization: all KEK bytes in m.keys are zeroed on Close. Stack copies
//     produced during WrapKey/UnwrapKey are zeroed via defer.
//   - Idempotency: Close is idempotent; second call is a no-op (nil error).
//   - Nil envelope: UnwrapKey returns ErrInvalidEnvelope for nil or zero-
//     ciphertext envelopes.
//   - Post-close: WrapKey/UnwrapKey/ActiveKeyVersion/HealthCheck return
//     ErrProviderUnavailable after Close.
//   - Ciphertext format: [12-byte nonce || GCM ciphertext || 16-byte GCM tag]
//     stored verbatim in KeyEnvelope.Ciphertext.
type AESKEKManager struct {
    mu            sync.RWMutex
    providerName  string
    keys          map[int][]byte  // version → 32-byte AES-256 KEK
    activeVersion int
    closed        bool
}
```

**`WrapKey(ctx, plaintext, metadata)`**
1. Check `ctx.Err()`.
2. RLock; check `closed → ErrProviderUnavailable`.
3. Retrieve `keys[activeVersion]`; if absent, return `ErrKeyNotFound`.
4. Generate 12-byte nonce via `crypto/rand.Read`.
5. `aes.NewCipher(kek)` → `cipher.NewGCM(block)` → `gcm.Seal(nonce, nonce, plaintext, nil)`.
6. Return `KeyEnvelope{KeyID: fmt.Sprintf("self-contained-aes-v%d", activeVersion), KeyVersion: activeVersion, Provider: providerName, Ciphertext: sealed, CreatedAt: time.Now()}`.

**`UnwrapKey(ctx, envelope, metadata)`**
1. Check `ctx.Err()`.
2. Validate envelope not nil and `Ciphertext` not empty (→ `ErrInvalidEnvelope`).
3. RLock; check `closed → ErrProviderUnavailable`.
4. Build `candidateVersions` list: declared version first, then all others.
5. For each version, copy KEK, attempt `gcm.Open`, zeroize copy on exit; on success return plaintext.
6. On exhaustion: `fmt.Errorf("%w: all versions failed: %w", ErrUnwrapFailed, lastErr)`.

**`ActiveKeyVersion(ctx)`** — RLock, return `activeVersion`.

**`HealthCheck(ctx)`** — RLock, verify `len(keys[activeVersion]) == 32`.

**`Close(ctx)`** — Lock, set `closed = true`, call `zeroBytes(key)` and `delete(keys, ver)` for all versions.

**`PrepareRotation(ctx, target *int) (RotationPlan, error)`** — identical semantics to `inMemoryKeyManager.PrepareRotation` (`internal/crypto/keymanager_memory.go:261`). If `target` is nil, auto-select the highest non-active version; if none exists, return `ErrRotationAmbiguous`.

**`PromoteActiveVersion(ctx, plan RotationPlan) error`** — identical semantics to `inMemoryKeyManager.PromoteActiveVersion` (`internal/crypto/keymanager_memory.go:295`). Validates `plan.CurrentVersion == activeVersion` and `plan.TargetVersion` exists.

### 3.2 `RSAKEKManager`

```go
// RSAKEKManager wraps DEKs using RSA-OAEP (SHA-256) with a locally-held key pair.
//
// Invariants (additional to KeyManager base):
//   - Minimum key size: 2048 bits (enforced at construction time).
//   - Public key must match private key; mismatch → error at construction.
//   - WrapKey uses the public key only (encryption); no private key access.
//   - UnwrapKey uses the private key; private key is not exported or logged.
//   - Ciphertext format: raw RSA-OAEP output stored in KeyEnvelope.Ciphertext.
//   - KeyID: hex-encoded SHA-256 of the public key's DER representation,
//     truncated to 16 bytes, prefixed "self-contained-rsa-".
//   - Versioning: version is set at construction; all wrapped envelopes carry
//     that version. Multi-version support requires instantiating multiple
//     managers (no AddVersion path for RSA).
//   - Close: zeroizes the private key's primes (d, p, q, dp, dq, qinv) by
//     overwriting with zeros via reflect or manual field zeroing.
type RSAKEKManager struct {
    mu           sync.RWMutex
    providerName string
    privateKey   *rsa.PrivateKey  // includes public key
    keyID        string           // fingerprint
    keyVersion   int
    closed       bool
}
```

**`WrapKey(ctx, plaintext, metadata)`**
1. Check `ctx.Err()`.
2. RLock; check `closed → ErrProviderUnavailable`.
3. `rsa.EncryptOAEP(sha256.New(), rand.Reader, &privateKey.PublicKey, plaintext, nil)`.
4. Return `KeyEnvelope{KeyID: keyID, KeyVersion: keyVersion, ...}`.

**`UnwrapKey(ctx, envelope, metadata)`**
1. Check `ctx.Err()`, nil/empty envelope checks.
2. RLock; check `closed`.
3. `rsa.DecryptOAEP(sha256.New(), rand.Reader, privateKey, envelope.Ciphertext, nil)`.
4. On error: `fmt.Errorf("%w: %w", ErrUnwrapFailed, err)`.

**`HealthCheck(ctx)`** — RLock; verify modulus of public key matches private key.

**`Close(ctx)`** — Lock; zero primes and CRT parameters of `privateKey`; set `closed = true`.

### 3.3 Factory Configuration Map

The `"self_contained"` factory (`selfContainedFactory`) inspects the `"type"` key:
- `"aes"` → calls `newAESKEKManagerFromConfig(cfg)`
- `"rsa"` → calls `newRSAKEKManagerFromConfig(cfg)`
- anything else → error

**AES config keys:**
| Key | Type | Meaning |
|-----|------|---------|
| `"type"` | `string` | Must be `"aes"` |
| `"provider"` | `string` | Override `Provider()` name (default: `"self_contained"`) |
| `"keys"` | `[]map[string]any` | List of `{"version": int, "key_source": "env:VAR"\|"base64:..."\|"file:PATH"}` |
| `"active_version"` | `int` | Version to use for wrapping (default: highest version number) |

**RSA config keys:**
| Key | Type | Meaning |
|-----|------|---------|
| `"type"` | `string` | Must be `"rsa"` |
| `"provider"` | `string` | Override `Provider()` name |
| `"private_key_source"` | `string` | `"env:VAR"` (PEM), `"file:PATH"`, or literal PEM |
| `"key_version"` | `int` | Version integer for this key pair (default: 1) |

### 3.4 Go Config Struct

```go
// SelfContainedKMConfig captures settings for the self-contained KEK adapter.
type SelfContainedKMConfig struct {
    // Type selects the wrapping algorithm: "aes" or "rsa".
    Type string `yaml:"type" env:"SELF_CONTAINED_TYPE"`

    // AES holds settings for AES-256-GCM KEK wrapping.
    AES SelfContainedAESConfig `yaml:"aes"`

    // RSA holds settings for RSA-OAEP KEK wrapping.
    RSA SelfContainedRSAConfig `yaml:"rsa"`
}

// SelfContainedAESConfig holds AES-256-GCM KEK settings.
type SelfContainedAESConfig struct {
    // Keys is the list of versioned KEK entries.
    Keys []SelfContainedAESKeyEntry `yaml:"keys"`
    // ActiveVersion selects the wrapping version (default: highest).
    ActiveVersion int `yaml:"active_version" env:"SELF_CONTAINED_AES_ACTIVE_VERSION"`
}

// SelfContainedAESKeyEntry is a single versioned AES KEK reference.
type SelfContainedAESKeyEntry struct {
    Version   int    `yaml:"version"`
    KeySource string `yaml:"key_source"` // "env:VAR", "base64:...", "file:PATH"
}

// SelfContainedRSAConfig holds RSA-OAEP KEK settings.
type SelfContainedRSAConfig struct {
    PrivateKeySource string `yaml:"private_key_source" env:"SELF_CONTAINED_RSA_PRIVATE_KEY_SOURCE"`
    KeyVersion       int    `yaml:"key_version" env:"SELF_CONTAINED_RSA_KEY_VERSION"`
}
```

**Rationale:** Matching the `MemoryKMConfig` pattern (a single struct per provider with typed fields and env tags) keeps the config layer consistent and allows `loadFromEnv` to handle the common env-var injection pattern without reflection magic. Separate `AES` / `RSA` sub-structs mirror the `CosmianConfig` nesting style and avoid a stringly-typed grab-bag.

### 3.5 YAML Shape

```yaml
encryption:
  key_manager:
    enabled: true
    provider: self_contained
    self_contained:
      type: aes
      aes:
        active_version: 2
        keys:
          - version: 1
            key_source: "env:AES_KEK_V1"   # base64-encoded 32-byte key
          - version: 2
            key_source: "env:AES_KEK_V2"
```

```yaml
encryption:
  key_manager:
    enabled: true
    provider: self_contained
    self_contained:
      type: rsa
      rsa:
        private_key_source: "file:/run/secrets/kek_rsa.pem"
        key_version: 1
```

**Validation rules:**
- `type` must be `"aes"` or `"rsa"` when provider is `"self_contained"`.
- For `"aes"`: at least one key entry required; all `key_source` values must be non-empty; versions must be unique positive integers.
- For `"rsa"`: `private_key_source` must be non-empty; resolved PEM must parse to a private key ≥ 2048 bits.

- Functional option `AESKEKOption`: `WithAESProvider(string)`.
- Compile-time assertion: `var _ KeyManager = (*AESKEKManager)(nil)`.

Code sketch for `WrapKey`:
```go
func (m *AESKEKManager) WrapKey(ctx context.Context, plaintext []byte, _ map[string]string) (*KeyEnvelope, error) {
    if err := ctx.Err(); err != nil {
        return nil, fmt.Errorf("keymanager/self-contained-aes: %w", err)
    }
    m.mu.RLock()
    defer m.mu.RUnlock()
    if m.closed {
        return nil, ErrProviderUnavailable
    }
    kek, ok := m.keys[m.activeVersion]
    if !ok {
        return nil, fmt.Errorf("%w: no active key version %d", ErrKeyNotFound, m.activeVersion)
    }
    nonce := make([]byte, 12)
    if _, err := rand.Read(nonce); err != nil {
        return nil, fmt.Errorf("keymanager/self-contained-aes: nonce generation: %w", err)
    }
    block, err := aes.NewCipher(kek)
    if err != nil {
        return nil, fmt.Errorf("keymanager/self-contained-aes: cipher init: %w", err)
    }
    gcm, err := cipher.NewGCM(block)
    if err != nil {
        return nil, fmt.Errorf("keymanager/self-contained-aes: gcm init: %w", err)
    }
    sealed := gcm.Seal(nonce, nonce, plaintext, nil) // nonce prepended
    return &KeyEnvelope{
        KeyID:      fmt.Sprintf("self-contained-aes-v%d", m.activeVersion),
        KeyVersion: m.activeVersion,
        Provider:   m.providerName,
        Ciphertext: sealed,
        CreatedAt:  time.Now(),
    }, nil
}
```

### Phase B — Core RSA-KEK Implementation

**B1.** `internal/crypto/keymanager_selfcontained_rsa.go` — **new**

Implement `RSAKEKManager` struct:
- `NewRSAKEKManager(privateKey *rsa.PrivateKey, version int, opts ...RSAKEKOption) (*RSAKEKManager, error)` — constructor; validates `privateKey != nil`; calls `privateKey.Precompute()`; checks `privateKey.Size() >= 256` (2048 bits); computes `keyID` as `hex.EncodeToString(sha256sum(x509.MarshalPKCS1PublicKey(&pk.PublicKey))[:8])` prefixed by `"self-contained-rsa-"`.

- Handles `"env:VAR"` (PEM from env), `"file:PATH"` (PEM from file).

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| AES-GCM nonce collision: if `rand.Read` fails silently, two objects could share a nonce under the same KEK | Use `rand.Read` with explicit error check; abort `WrapKey` on failure (`fmt.Errorf("nonce generation: %w", err)`); nonce is 96 bits which is safe for up to ~2^32 objects before collision probability becomes non-negligible — document the per-key object limit in the ADR |
| KEK bytes appear in structured log fields via `%v` formatting on error wrapping | Add `TestAESKEKManager_NoKEKInLogs` (see §6); add linter rule or code review checklist item: no `fmt.Errorf` may reference `kek`, `masterKey`, or key slice variables |
| RSA private key not zeroed completely: `big.Int` uses internal `nat` arrays; simple slice clearing may not zero the backing array if the GC has made copies | Use `bi.SetInt64(0)` followed by `bi.SetBytes(make([]byte, bi.BitLen()/8+1))` pattern; note limitation in ADR and recommend HSM adapter for higher-assurance requirements |
| Operator loses KEK → all DEKs permanently irrecoverable | Document clearly in ADR, README, and `KMS_COMPATIBILITY.md`; suggest key backup procedures (split-key backup with `shamir-secret-sharing` or stored in Vault); this is a promoted DoD item |
| `"self_contained"` registry name collision with future adapters | Name is registered in `init()` via `Register` which panics on duplicate; collision will fail fast in tests |
| Config validation gap: `type` field absent defaults to empty string | `Validate()` must reject `provider == "self_contained"` with empty `Type`; add negative test `TestConfig_SelfContained_MissingType_Invalid` |
| Integration test flakiness due to MinIO startup timing | Use `testcontainers-go` `WaitForLog` strategy already used in other integration tests in `test/` |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **`zeroBytes` on `big.Int` primes** — The current `zeroBytes()` in
   `internal/crypto/engine.go` operates on `[]byte`. Zeroing `rsa.PrivateKey`
   CRT parameters requires calling `big.Int.SetInt64(0)` and filling the
   backing array. A helper `zeroBigInt(*big.Int)` should be added to
   `engine.go`; this is a small, safe change but is a separate concern from the
   `RSAKEKManager` scope and could be independently reviewed.

2. **Conformance suite extension for RSA** — `ConformanceSuite_Rotation` currently
   assumes `AddVersion(version int, material []byte)` which is AES-specific.
   RSA does not support `AddVersion`. A future issue should add a
   `RotatableKeyManager_RSA_Skip` helper or a separate conformance tier for
   asymmetric providers.

3. **FIPS build tag** — AES-256-GCM and RSA-OAEP both use `crypto/aes`,
   `crypto/cipher`, and `crypto/rsa` from the standard library. No separate
   FIPS build file is needed now. If FIPS certification is pursued, a
   `createaead_fips.go`-style file may be required; tracked separately.

4. **Lazy re-wrap background job** — Post-rotation objects continue using old
   DEKs until the next write. An optional background scanner could re-wrap DEKs
   proactively; this is deferred and would require a separate issue.

## 11. References

1. NIST SP 800-38D, *Recommendation for Block Cipher Modes of Operation: Galois/Counter Mode (GCM) and GMAC* (NIST, 2007) — AES-GCM nonce uniqueness requirements and authentication tag size guidance.
   <https://csrc.nist.gov/publications/detail/sp/800-38d/final>

2. NIST FIPS PUB 186-5, *Digital Signature Standard (DSS)* (NIST, 2023) — RSA key size requirements (minimum 2048 bits for new keys through 2030).
   <https://csrc.nist.gov/publications/detail/fips/186/5/final>

3. RSA Laboratories, *PKCS #1 v2.2: RSA Cryptography Standard* — RSAES-OAEP encoding specification; use of SHA-256 as hash function.
   <https://www.rfc-editor.org/rfc/rfc8017>

4. R. Housley, *RFC 3394: Advanced Encryption Standard (AES) Key Wrap Algorithm* (IETF, 2002) — Background on key-wrap alternatives; explains why GCM is chosen over RFC 3394 for this adapter (GCM is more widely understood and provides authenticated encryption with standard AEAD interface).
   <https://www.rfc-editor.org/rfc/rfc3394>

5. H. Krawczyk and P. Eronen, *RFC 5869: HMAC-based Extract-and-Expand Key Derivation Function (HKDF)* (IETF, 2010) — Referenced in security considerations for AES variant: if an HMAC key is ever needed, HKDF from the KEK must be used rather than reusing the KEK directly.
   <https://www.rfc-editor.org/rfc/rfc5869>

6. `docs/adr/0007-admin-api-and-key-rotation.md` — Defines `RotatableKeyManager` interface and the drain-and-cutover rotation state machine used by the admin API.

7. `internal/crypto/keymanager_memory.go` — Reference implementation for `RotatableKeyManager`, `candidateVersions`, zeroization pattern, and mutex discipline.
