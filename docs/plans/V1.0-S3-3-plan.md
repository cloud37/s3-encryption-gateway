## 1. Context & Current State

### The broken invariant

The S3 de-facto contract that all sync clients depend on is:

```
ListObjects[i].Size == HeadObject(key).Content-Length
```

The gateway breaks this for every encrypted object when `max-keys > 10`.
`handleListObjects` (`internal/api/handlers.go:2228`) calls
`s3Client.ListObjects`, which returns raw backend sizes — ciphertext sizes,
not plaintext sizes. The per-object `HeadObject` translation that correctly
resolves plaintext sizes is intentionally gated behind `maxKeys <= 10`
(`handlers.go:2340`) to avoid N-fold latency amplification on large listing
pages.

For a 124,804-byte plaintext object encrypted with 64 KiB chunks and
AES-256-GCM, the listing reports `124836` (2 × 16-byte AEAD tag overhead).
rclone, restic, Duplicati, and s5cmd compare listing sizes against source
sizes; they flag every encrypted object as modified on every run and
re-transfer or error-report indefinitely.

### Why the HeadObject-per-object path cannot be the general solution

The S3 `ListObjectsV2` XML response body never contains user-defined metadata
(`x-amz-meta-*`). Plaintext size is stored in `x-amz-meta-encryption-original-size`
(`crypto.MetaOriginalSize`, `internal/crypto/engine.go:40`), which is only
retrievable via `HeadObject` or `GetObject`. Issuing one `HeadObject` per
listed key creates:

- **N-fold latency:** 1000 listed keys → 1000 serial HEAD calls before the
  first XML byte is sent (or, with parallelism, ~50 serial rounds at N=20
  workers). Backends that charge per API call (Wasabi, Cloudflare R2,
  Backblaze B2) accrue unbounded cost proportional to listing volume.
- **Backend amplification:** a `rclone sync` over a 50,000-object bucket
  issues 50,000 extra HEADs per sync run. Running nightly for a year produces
  ~18 million extra HEAD calls.

### Current workarounds and their limits

The gateway already translates sizes correctly in two narrow cases:

1. **`maxKeys <= 10` path** (`handlers.go:2340`): issues a synchronous
   `lookupListObjectPlaintextSize` (`handlers.go:2372`) per object. Correct
   but deliberately bounded to the Docker Distribution `statList()` pattern.
2. **In-progress MPU (Valkey Phase 1)** (`handlers.go:2302`): resolves
   plaintext sizes from the Valkey MPU state store using `listActiveMPUPlainSizes`
   (`handlers.go:4182`) — a single `List` call, no per-object HEAD. This is
   the architectural precedent for the cache approach.

The conformance test `testListObjectsV1/V2_SizeAccuracy`
(`test/conformance/listobjects_size_test.go:305`) currently asserts
`listedSize > plaintextSize`, i.e. it validates the broken behavior as
correct. This must be inverted.

### Objective

Deliver correct `ListObjects` sizes (`listedSize == headSize`) for all
encrypted objects in steady-state operation, with zero additional backend
HEAD calls in the warm-cache path, and predictable, opt-in HEAD cost for
cache misses.

## 2. Design Goals & Non-Goals

### Goals

1. **Correctness in steady state:** after the cache warms up through normal
   `PutObject`/`CompleteMultipartUpload` traffic, `ListObjects` returns
   `listedSize == headSize` for every encrypted object — zero extra backend
   calls.
2. **Graceful degradation:** if Valkey is unavailable, the gateway returns
   ciphertext sizes (current behavior) rather than crashing or returning 5xx.
   No new hard dependency on Valkey for correctness.
3. **Cache coherence:** `DeleteObject`, `DeleteObjects`, and `CopyObject`
   maintain the index automatically with no operator action.
4. **Opt-in fallback HEAD batch:** a configuration flag
   (`list_size_translate.fallback_head_enabled`) enables bounded concurrent
   `HeadObject` calls for cache misses. Disabled by default to prevent
   unintended billing amplification.
5. **Single Valkey round-trip per listing page:** the entire listing page is
   resolved with one `HMGET` call, not one `GET` per key.
6. **Observability:** Prometheus counters expose cache hit rate, miss rate,
   and fallback HEAD call volume so operators can track warm-up progress and
   billing exposure.
7. **No regression for Docker Distribution:** the existing `maxKeys <= 10`
   path is preserved and runs before the new cache lookup; Harbor/Docker
   Distribution continues to work as today.

### Non-Goals

1. **Retroactive warm-up of legacy objects** — objects uploaded before this
   feature is deployed will not be automatically indexed. A future admin
   endpoint (`POST /admin/warm-size-cache`) is noted as a follow-up (§10).
2. **Valkey as a required dependency** — the size cache is additive; the
   gateway must boot and serve traffic without Valkey configured.
3. **ETag correction in listings** — ETags in `ListObjects` responses remain
   ciphertext ETags. Fixing listing ETags is tracked separately.
4. **GetObjectAttributes implementation** — a correct `ObjectSize` in
   `GetObjectAttributes` would also fix rclone, but that API is not
   implemented and is out of scope for this plan.
5. **MPU in-progress listing fix** — Phase 1 (Valkey MPU state) already
   handles in-progress MPU objects. No changes to that path.

## 3. Target Interface / Semantics / Behaviour

### 3.1 `SizeCache` interface

```go
// SizeCache is a write-through index mapping (bucket, key) → plaintextSize.
// It is used by handleListObjects to resolve encrypted object sizes without
// issuing per-object HeadObject calls to the backend.
//
// Invariants:
//   - All methods are safe for concurrent use.
//   - Context cancellation must be respected; implementations must not block
//     past ctx.Done().
//   - Set and Delete are best-effort: callers MUST NOT treat an error as fatal.
//     The index is advisory; stale or absent entries fall back to ciphertext sizes.
//   - GetBatch returns a map keyed on the requested keys. Missing keys are
//     absent from the map (not present with zero value).
//   - plaintextSize values MUST be > 0; implementations may reject ≤ 0 sizes.
type SizeCache interface {
    // Set records bucket/key → plaintextSize. Overwrites any existing entry.
    Set(ctx context.Context, bucket, key string, plaintextSize int64) error

    // SetBatch records multiple entries atomically where possible.
    // Partial failure is permitted; the error describes the first failure.
    SetBatch(ctx context.Context, bucket string, sizes map[string]int64) error

    // GetBatch returns the plaintext sizes for all requested keys that exist
    // in the cache. Keys not found are absent from the returned map.
    GetBatch(ctx context.Context, bucket string, keys []string) (map[string]int64, error)

    // Delete removes the entry for bucket/key. Safe to call for absent keys.
    Delete(ctx context.Context, bucket, key string) error

    // DeleteBatch removes entries for multiple keys in the same bucket.
    DeleteBatch(ctx context.Context, bucket string, keys []string) error

    // Close releases resources. Idempotent.
    Close() error
}
```

**Rationale:** Small interface, single responsibility. Separating `SizeCache`
from `mpu.StateStore` keeps MPU lifecycle concerns isolated from the size
index. Both can share the same Valkey connection pool but use distinct key
namespaces.

### 3.2 Valkey key schema

```
plainsize:<bucket>  →  Hash field: <key> → "<plaintextSize>"
```

One Valkey hash per bucket. `HMGET plainsize:<bucket> key1 key2 ... keyN`
resolves an entire listing page in a single round-trip. `HDEL` removes
individual keys on delete. No TTL — entries are permanent until explicitly
evicted.

**Rationale:** A Valkey hash per bucket maps directly to `HMGET`/`HSET`/`HDEL`
semantics with O(N) complexity for batch operations. Avoids N independent
`GET` calls.

### 3.3 `ListSizeTranslateConfig` stanza

```yaml
list_size_translate:
  enabled: true                   # master switch; default true if Valkey configured
  fallback_head_enabled: false    # opt-in: issue HEAD for cache misses; default false
  fallback_head_concurrency: 10   # max concurrent HeadObject goroutines; default 10
  fallback_head_timeout: 5s       # per-page HEAD batch deadline; default 5s
```

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | bool | `true` | Enable size-cache lookup on ListObjects. Automatically false when Valkey is not configured. |
| `fallback_head_enabled` | bool | `false` | Issue bounded concurrent HEADs for cache misses. Disabled by default to avoid billing amplification. |
| `fallback_head_concurrency` | int | `10` | Maximum concurrent HeadObject goroutines per listing page. |
| `fallback_head_timeout` | duration | `5s` | Per-page deadline for the HEAD batch. Keys that do not resolve within the deadline return ciphertext sizes. |

**Validation rules:** `fallback_head_concurrency` must be in [1, 100].
`fallback_head_timeout` must be in [100ms, 60s]. If `enabled` is true but
Valkey is not configured, log a warning and silently disable.

### 3.4 `handleListObjects` resolution order

After `s3Client.ListObjects` returns the page of objects, the handler applies
size resolution in this order:

1. **Existing Docker Distribution path** (`maxKeys <= 10`): unchanged —
   runs first, may resolve all objects before the cache is consulted.
2. **Valkey size cache (`GetBatch`):** for all objects not yet resolved,
   call `cache.GetBatch(ctx, bucket, keys)`. Substitute hits.
3. **Fallback HEAD batch** (only if `fallback_head_enabled: true`): for
   remaining misses, dispatch bounded concurrent `HeadObject` calls. Populate
   cache from results (`Set` each resolved key). Substitute hits.
4. **Unresolved keys:** return ciphertext size (current behavior, no change).

This order ensures the fast path (cache) is always attempted before the
expensive path (HEAD), and the expensive path is never triggered unless
explicitly opted in.

### 3.5 Error codes

No new S3 error codes. All cache operations are fail-soft. Valkey errors are
logged at `WARN` level and the listing proceeds with ciphertext sizes.

**A2. New file: `internal/sizecache/valkey.go`**

Implement `ValkeySizeCache` backed by `go-redis/v9` (already a dependency via
`internal/mpu/state.go`):

```go
type ValkeySizeCache struct {
    client redis.UniversalClient
}

func bucketKey(bucket string) string {
    return "plainsize:" + bucket
}

func (c *ValkeySizeCache) Set(ctx context.Context, bucket, key string, size int64) error {
    return c.client.HSet(ctx, bucketKey(bucket), key, strconv.FormatInt(size, 10)).Err()
}

func (c *ValkeySizeCache) SetBatch(ctx context.Context, bucket string, sizes map[string]int64) error {
    if len(sizes) == 0 {
        return nil
    }
    args := make([]interface{}, 0, len(sizes)*2)
    for k, v := range sizes {
        args = append(args, k, strconv.FormatInt(v, 10))
    }
    return c.client.HSet(ctx, bucketKey(bucket), args...).Err()
}

func (c *ValkeySizeCache) GetBatch(ctx context.Context, bucket string, keys []string) (map[string]int64, error) {
    if len(keys) == 0 {
        return nil, nil
    }
    vals, err := c.client.HMGet(ctx, bucketKey(bucket), keys...).Result()
    if err != nil {
        return nil, err
    }
    result := make(map[string]int64, len(keys))
    for i, v := range vals {
        if v == nil {
            continue
        }
        if s, ok := v.(string); ok {
            if n, parseErr := strconv.ParseInt(s, 10, 64); parseErr == nil && n > 0 {
                result[keys[i]] = n
            }
        }
    }
    return result, nil
}

func (c *ValkeySizeCache) Delete(ctx context.Context, bucket, key string) error {
    return c.client.HDel(ctx, bucketKey(bucket), key).Err()
}

func (c *ValkeySizeCache) DeleteBatch(ctx context.Context, bucket string, keys []string) error {
    if len(keys) == 0 {
        return nil
    }
    return c.client.HDel(ctx, bucketKey(bucket), keys...).Err()
}

func (c *ValkeySizeCache) Close() error { return c.client.Close() }
```

Constructor `NewValkeySizeCache(client redis.UniversalClient) *ValkeySizeCache`.
The caller (wiring in `cmd/server/main.go`) passes in the same `redis.UniversalClient`
already created for the MPU state store, sharing the connection pool.

```go
// Phase 3: size-cache resolution for general listings.
// For each object not yet resolved by phases 1–2 (MPU Valkey state and
// maxKeys<=10 HEAD path), look up plaintext size from the write-time index.
if h.sizeCache != nil && h.config.ListSizeTranslate.Enabled && len(listResult.Objects) > 0 {
    // Collect keys that still carry ciphertext sizes.
    missIdxs := make([]int, 0, len(listResult.Objects))
    missKeys := make([]string, 0, len(listResult.Objects))
    for i := range listResult.Objects {
        missIdxs = append(missIdxs, i)
        missKeys = append(missKeys, listResult.Objects[i].Key)
    }

    cached, cacheErr := h.sizeCache.GetBatch(ctx, bucket, missKeys)
    if cacheErr != nil {
        h.logger.WithError(cacheErr).Warn("handleListObjects: size cache GetBatch failed; returning ciphertext sizes")
    } else {
        stillMissIdxs := missIdxs[:0]
        stillMissKeys := missKeys[:0]
        for i, idx := range missIdxs {
            if ps, ok := cached[missKeys[i]]; ok {
                listResult.Objects[idx].Size = ps
                h.metrics.RecordListSizeCacheHit(ctx, bucket)
            } else {
                stillMissIdxs = append(stillMissIdxs, idx)
                stillMissKeys = append(stillMissKeys, missKeys[i])
                h.metrics.RecordListSizeCacheMiss(ctx, bucket)
            }
        }

        // Fallback HEAD batch for unresolved misses (opt-in).
        if h.config.ListSizeTranslate.FallbackHeadEnabled && len(stillMissIdxs) > 0 {
            h.resolveListSizesByHead(ctx, bucket, listResult.Objects, stillMissIdxs, stillMissKeys, s3Client)
        }
    }
}
```

**E2. New helper `resolveListSizesByHead` in `internal/api/handlers.go`**

```go
// resolveListSizesByHead issues bounded concurrent HeadObject calls for the
// given object indices, substitutes plaintext sizes, and populates the size
// cache. It is fail-soft: objects whose HEAD fails or times out retain their
// ciphertext sizes.
func (h *Handler) resolveListSizesByHead(
    ctx context.Context,
    bucket string,
    objects []s3.ObjectInfo,
    idxs []int,
    keys []string,
    s3Client s3.Client,
) {
    cfg := h.config.ListSizeTranslate
    batchCtx, cancel := context.WithTimeout(ctx, cfg.FallbackHeadTimeout)
    defer cancel()

    sem := make(chan struct{}, cfg.FallbackHeadConcurrency)
    type result struct {
        idx  int
        key  string
        size int64
        err  error
    }
    results := make(chan result, len(idxs))
    var wg sync.WaitGroup

    for i, idx := range idxs {
        wg.Add(1)
        go func(i, idx int, key string) {
            defer wg.Done()
            sem <- struct{}{}
            defer func() { <-sem }()

            ps, ok, err := h.lookupListObjectPlaintextSize(batchCtx, bucket, key, objects[idx].Size, s3Client)
            if err != nil || !ok {
                h.metrics.RecordListSizeFallbackHead(ctx, bucket, "error")
                results <- result{idx: idx, key: key, err: err}
                return
            }
            h.metrics.RecordListSizeFallbackHead(ctx, bucket, "hit")
            results <- result{idx: idx, key: key, size: ps}
        }(i, idx, keys[i])
    }

    go func() { wg.Wait(); close(results) }()

    cacheUpdates := make(map[string]int64)
    for r := range results {
        if r.err == nil && r.size > 0 {
            objects[r.idx].Size = r.size
            cacheUpdates[r.key] = r.size
        }
    }
    if h.sizeCache != nil && len(cacheUpdates) > 0 {
        _ = h.sizeCache.SetBatch(ctx, bucket, cacheUpdates)
    }
}
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Valkey unavailability causes listing failures | `GetBatch` errors are swallowed with a `WARN` log; listing proceeds with ciphertext sizes. Conformance test `TestListObjects_SizeCache_ValkeyUnavailable` verifies this path. |
| `FallbackHeadEnabled: true` causes unintended API cost on high-traffic paths | Disabled by default. Requires explicit opt-in in config. `fallback_head_concurrency` and `fallback_head_timeout` cap worst-case amplification. Document billing risk in `config.yaml.example`. |
| `resolveListSizesByHead` goroutine leak if context cancelled early | `batchCtx` with deadline is derived from request ctx; goroutines check `batchCtx.Done()` via `lookupListObjectPlaintextSize` (which calls `HeadObject` with the context). `wg.Wait()` guarantees all goroutines exit before results channel is closed. |
| Two Valkey connection pools (MPU state + size cache) | Share the `redis.UniversalClient` instance. Extract connection construction into a helper in `cmd/server/main.go`. Promoted to DoD. |
| `plainSize` not in scope at cache injection point in `handlePutObject` | Lift `plainSize` variable to function scope before the streaming-PUT block (Phase D2). Verified during implementation with `go vet`. |
| Cache poisoning: wrong size written on failed PUT | `sizeCache.Set` is called only after `w.WriteHeader(http.StatusOK)` is reached — i.e., after the backend confirmed success. A failed PUT never reaches the injection point. |
| CopyObject does not fetch source metadata in all code paths | `handleCopyObject` (`handlers.go:4431`) already calls `HeadObject` on the source (`handlers.go:4490`). The existing metadata map is reused. Verify no early-return path skips the source HEAD before the cache Set. |
| Legacy objects (uploaded before this feature) are not in cache | Cache miss returns ciphertext size — same as current behavior, no regression. Operator can enable `fallback_head_enabled` temporarily to warm the cache via normal listing traffic. Future admin endpoint (§10) for proactive warm-up. |
| `DeleteObjects` batch path collects deletions inside a loop; not all keys may succeed | Only insert into `deletedKeys` when the individual delete in the batch response is `Code == ""` (success). Do not evict keys for items the backend reported as failed. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

- **Admin warm-up endpoint** (`POST /admin/warm-size-cache?bucket=X`): trigger
  the audit-style parallel HEAD scanner (`internal/audit/listalgorithm.go`) to
  pre-populate the size index for existing objects uploaded before this feature
  was deployed. One-time operator action, not automatic. Tracked as a follow-up
  issue.
- **ETag correction in ListObjects**: `ListObjects` also returns ciphertext ETags.
  The gateway restores `MetaOriginalETag` in `HeadObject` and `GetObject` but not
  in `ListObjects`. This is a separate correctness issue; fix is deferred to avoid
  scope creep here.
- **`GetObjectAttributes` implementation**: would provide a standards-compliant
  `ObjectSize` field that rclone and other clients can query without relying on
  listing sizes. Out of scope for this plan.
- **Valkey connection refactor**: if the shared-client extraction (Phase F) reveals
  that `ValkeyStateStore` has a tightly coupled constructor, a broader Valkey
  connection-management refactor may be warranted. Tracked as a tech-debt item.
