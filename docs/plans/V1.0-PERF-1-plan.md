## 1. Context & Current State

### 1.1 Existing scaling infrastructure

The gateway ships basic autoscaling infrastructure today but it lacks
published guidance, empirically derived thresholds, and structured load
profiles:

| Component | File | Current state |
|---|---|---|
| Raw k8s HPA manifest | `k8s/hpa.yaml:1` | CPU 70 %, memory 80 %; min=2, max=10; scale-up window 0 s, scale-down stabilisation 300 s |
| Helm HPA template | `helm/s3-encryption-gateway/templates/hpa.yaml:1` | Mirrors `autoscaling.*` values; not enabled by default (`autoscaling.enabled: false`) |
| `values.yaml` autoscaling stanza | `helm/s3-encryption-gateway/values.yaml:724` | `minReplicas: 2`, `maxReplicas: 10`, `targetCPUUtilizationPercentage: 70`, `targetMemoryUtilizationPercentage: 80` |
| Production overlay | `helm/s3-encryption-gateway/values.prod.yaml:36` | Enables HPA with `minReplicas: 3`, `maxReplicas: 20`, `targetCPUUtilizationPercentage: 65`; no custom-metrics variant |
| Resource limits | `helm/s3-encryption-gateway/values.yaml:698` | Default: `requests: cpu=100m, memory=64Mi`; `limits: cpu=500m, memory=256Mi` |
| Production resource limits | `helm/s3-encryption-gateway/values.prod.yaml:57` | `requests: cpu=200m, memory=256Mi`; `limits: cpu=2000m, memory=1Gi` |
| Load harness | `test/conformance/load_test.go:1` | Range-read and multipart soak, env-driven (3 workers / 5 s CI; 10 workers / 60 s soak) |
| Soak env vars | `Makefile:124` | `SOAK_WORKERS=10`, `SOAK_DURATION=60s`, `SOAK_QPS=25`, `SOAK_OBJECT_SIZE=52428800`, `SOAK_PART_SIZE=10485760` |
| Perf baselines | `docs/perf/v0.6-qa-1/` | micro-baseline.txt, macro JSON per provider, slo-summary.md (TBD values) |
| PERFORMANCE.md | `docs/PERFORMANCE.md:1` | Describes micro/macro methodology; thresholds table at §6; no horizontal-scaling guidance |

Three gaps explain why the current state is insufficient:

**Gap 1 — No high-concurrency load profile.** The existing soak runs
10 workers at 25 QPS against 50 MiB objects. This exercises the streaming
crypto path but does not simulate the throughput regime that triggers the
HPA — i.e. dozens of concurrent workers sustaining hundreds of small-object
requests per second alongside large-object transfers. There is no load
profile that drives CPU to the HPA threshold and measures how quickly the
cluster responds.

**Gap 2 — No empirical basis for resource requests/limits.** The production
defaults (`cpu=200m request`, `cpu=2000m limit`) were not derived from
load-test measurements. Operators cannot answer "how many replicas do I
need for N MB/s sustained throughput?" without running their own
experiments.

**Gap 3 — No horizontal-scaling documentation or SLO definitions.** The
`PERFORMANCE.md` document describes benchmark methodology but says nothing
about Kubernetes autoscaling, capacity planning, or service-level objectives
that operators should commit to. The `OPS_DEPLOYMENT.md` covers progressive
delivery (blue/green, canary) but not steady-state capacity management.

The objective of V1.0-PERF-1 is to close all three gaps by producing
documented, reproducible load profiles; published sizing guidance; tuned
Helm HPA examples; and a `docs/SCALING.md` playbook that operators can
follow without further research.

### 1.2 Load test harness — relevant existing code

`test/conformance/load_test.go` contains the in-process gateway harness:
- `resolveLoadParams` (`load_test.go:57`) reads soak env vars.
- `TestConformance_RangeRead` and `TestConformance_Multipart` (`load_test.go:80+`) execute the two load profiles.
- `loadtest_summary.go` emits per-run NDJSON (throughput, latency percentiles,
  heap high-water-mark).

`scripts/bench-macro.sh` drives the soak against each provider and collects
NDJSON output into `docs/perf/v0.6-qa-1/macro-<provider>.json`.

There is no equivalent script or test function for high-concurrency,
mixed-operation load (simulate 50+ concurrent clients doing a mix of small
PutObject + GetObject + MPU).

### 1.3 Helm autoscaling — existing template

`helm/s3-encryption-gateway/templates/hpa.yaml:1-32` renders an
`autoscaling/v2` HPA with CPU + memory utilisation targets. The template
references `autoscaling.behavior` via `toYaml`, meaning arbitrary
`behavior` stanzas are already supported. Missing capabilities:
- No KEDA `ScaledObject` variant for custom-metrics HPA (e.g. scale on
  `http_requests_total` rate via Prometheus adapter).
- No `PodDisruptionBudget` coordination guidance (PDB exists in the chart
  at `helm/.../templates/poddisruptionbudget.yaml` but the HPA and PDB
  values are not documented together).
- No guidance on `terminationGracePeriodSeconds` sizing for large in-flight
  uploads that should not be interrupted mid-part.

### 1.4 ADRs and prior plans relevant to scaling

- ADR 0010 (`docs/adr/0010-backend-retry-policy.md`) defers circuit-breaker
  configuration to "real-world data from V0.6-QA-1 baselines" — the V0.6-QA-1
  SLO annex (`docs/perf/v0.6-qa-1/slo-summary.md`) feeds that decision.
  V1.0-PERF-1 adds high-concurrency data that informs the same circuit-breaker
  choices.
- V0.6-PERF-1 (`docs/plans/V0.6-PERF-1-plan.md`) introduced streaming
  encrypt/decrypt to bound per-request heap. The memory resource limits in
  this plan depend on those allocations being bounded.
- V1.0-OBS-1 (`docs/plans/V1.0-OBS-1-plan.md`) delivers Grafana dashboards
  and PrometheusRule alerting. V1.0-PERF-1 depends on those dashboards being
  available to visualise the load profiles and inform the SLO thresholds.

## 2. Design Goals & Non-Goals

### Goals

1. **Reproducible load profiles** — Define at least three named load profiles
   (`smoke`, `soak`, `spike`) with explicit env-var sets and expected outcome
   ranges. Any operator can run `make test-load-profile PROFILE=soak` and
   reproduce the reference numbers.
2. **Empirical sizing table** — Publish a `docs/SCALING.md` table mapping
   concurrent clients → throughput (MB/s) → recommended replicas → resource
   requests/limits, derived from the profiled load runs.
3. **Tuned HPA Helm values** — Ship a `helm/.../examples/values-hpa-tuned.yaml`
   overlay with HPA settings derived from the empirical data, ready for
   operators to adopt or adapt.
4. **SLO definitions** — Define three gateway SLOs (availability, latency,
   throughput) with explicit error budgets and populate the existing
   `docs/perf/v0.6-qa-1/slo-summary.md` table with measured baselines.
5. **KEDA alternative** — Document a KEDA `ScaledObject` example that scales
   on `http_requests_total` rate for workloads where CPU utilisation is a poor
   proxy for request pressure (e.g. small-object, high-IOPS workloads).
6. **Graceful-shutdown sizing** — Specify `terminationGracePeriodSeconds` and
   `lifecycle.preStop` values for common large-upload scenarios.
7. **Makefile integration** — Add named profile targets (`test-load-smoke`,
   `test-load-spike`, `test-load-high-concurrency`) to `Makefile`, matching
   the existing `test-load`, `test-load-soak` pattern.

### Non-Goals

- **New streaming or crypto optimisations.** The streaming rewrite is complete
  (V0.6-PERF-1); this plan documents scale-out, not scale-up.
- **Multi-region active/active.** S3 backend replication is out of scope;
  the gateway is stateless beyond the Valkey MPU store, and cross-region Valkey
  topology is a separate operations concern.
- **Kubernetes cluster autoscaling (Cluster Autoscaler / Karpenter).** This plan
  covers pod-level HPA, not node-level autoscaling. Node autoscaler configuration
  depends on cloud provider and is operator-specific.
- **KEDA implementation.** The plan ships an annotated KEDA `ScaledObject`
  YAML example, not a Helm template; full KEDA integration is tracked as a
  v1.x follow-up.
- **SLO alerting rules.** Prometheus alerting rules for SLO burn-rate are
  delivered by V1.0-OBS-1; this plan defines the SLO thresholds that feed them.

## 3. Target Interface / Semantics / Behaviour

### 3.1 Named load profiles

Three profiles are defined; each is a named set of env-var overrides applied
on top of the existing `resolveLoadParams` function (`load_test.go:57`):

| Profile | `SOAK_WORKERS` | `SOAK_DURATION` | `SOAK_QPS` | `SOAK_OBJECT_SIZE` | `SOAK_PART_SIZE` | Purpose |
|---|---|---|---|---|---|---|
| `smoke` | 3 | 10s | 10 | 102400 (100 KiB) | 5242880 (5 MiB) | Quick CI gate; passes in < 30 s |
| `soak` | 10 | 300s | 25 | 52428800 (50 MiB) | 10485760 (10 MiB) | Sustained steady-state; baseline for sizing table |
| `spike` | 50 | 60s | 50 | 102400 (100 KiB) | 5242880 (5 MiB) | High-concurrency burst; drives CPU to HPA threshold |

A fourth profile `high-throughput` (10 workers, 300 s, 10 QPS, 500 MiB
objects, 50 MiB parts) exercises large-object bandwidth-bound throughput and
feeds the `terminationGracePeriodSeconds` guidance.

Each profile is wired to a Makefile target (§4, Phase A).

### 3.2 SLO definitions

Three gateway SLOs, measured over a 30-day rolling window:

| SLO | Indicator | Target | Error budget (30 d) |
|---|---|---|---|
| **Availability** | `1 - (5xx_errors / total_requests)` as measured by `http_requests_total{status=~"5.."}` | ≥ 99.9 % | 43.2 min |
| **Latency (p99)** | `http_request_duration_seconds` p99 for `PutObject` ≤ 1 MiB | ≤ 500 ms | — |
| **Throughput** | Sustained aggregate MB/s across all replicas | ≥ 80 % of single-replica peak × replica count | — |

These thresholds are populated into `docs/perf/v0.6-qa-1/slo-summary.md`
after the profiled load runs; the table structure already exists.

**Rationale:** These SLOs follow the "four golden signals" approach [1] and
are deliberately conservative. The latency SLO covers only small-object PUT
because large-object latency is dominated by network/backend throughput, not
gateway overhead. The throughput SLO acknowledges that horizontal scaling is
not perfectly linear (Valkey coordination overhead, connection pool
saturation) and caps the target at 80 % of linear.

### 3.3 HPA configuration contract

The tuned HPA Helm overlay (`values-hpa-tuned.yaml`) defines:

```yaml
autoscaling:
  enabled: true
  minReplicas: 2        # Never below 2 for HA
  maxReplicas: 20       # Sized for 10× baseline traffic
  targetCPUUtilizationPercentage: 60   # empirically tuned; see SCALING.md §3
  targetMemoryUtilizationPercentage: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 25            # max 25% pods removed per minute — conservative
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 15   # fast scale-up for spike profiles
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
      - type: Pods
        value: 4
        periodSeconds: 15
      selectPolicy: Max

podDisruptionBudget:
  enabled: true
  minAvailable: 2       # Always keep 2 pods during node drain

terminationGracePeriodSeconds: 120   # ≥ p99 upload duration for 50 MiB parts
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "sleep 15"]   # kube-proxy propagation tail
```

**Rationale:** `targetCPUUtilizationPercentage: 60` (vs the current 65 in
`values.prod.yaml`) gives a 10 % headroom buffer before hitting the limit,
reducing request queuing during scale-out latency. The stabilisation window
of 15 s on scale-up allows faster reaction to spike traffic. The conservative
25 %/min scale-down policy protects against oscillation under bursty workloads,
following guidance from [2].

### 3.4 Resource sizing table (to be populated from profiled runs)

The `docs/SCALING.md` document will include the following table, populated
after the `spike` and `soak` profiles run against the MinIO Testcontainer
backend:

| Concurrent clients | Throughput (MB/s) | p99 PutObject latency | Recommended replicas | CPU request | Memory request |
|---|---|---|---|---|---|
| 10 | TBD | TBD | 2 | 200m | 256Mi |
| 25 | TBD | TBD | 3–4 | 200m | 256Mi |
| 50 | TBD | TBD | 6–8 | 200m | 256Mi |
| 100 | TBD | TBD | 12–16 | 200m | 256Mi |

### Phase B — Load profile execution and data collection

These steps produce the empirical data used in §3.4 and SCALING.md. They are
executed locally or in CI; the results are committed.

- **B1.** Run `make bench-load-capture` against a representative MinIO
  Testcontainer; collect NDJSON output.
- **B2.** Parse NDJSON from each profile to extract: `throughput_mbps`,
  `latency_ns.p50`, `latency_ns.p95`, `latency_ns.p99`, `errors`,
  `heap_inuse_max_bytes`.
- **B3.** Populate the sizing table in `docs/SCALING.md` (Phase D) from the
  `spike` profile data — number of concurrent workers maps to "concurrent
  clients" in the table.
- **B4.** Record peak CPU and memory usage from `pprof` or `kubectl top pods`
  during the `spike` run to validate resource limit recommendations.

  The `spike` profile with 50 workers is the relevant signal because that is
  the regime that triggers the HPA. The `soak` profile (10 workers, 300 s)
  provides the steady-state baseline for the latency SLO.

- **B5.** Commit NDJSON output files to `docs/perf/v1.0-perf-1/` (new
  directory). These are reference outputs analogous to `docs/perf/v0.6-qa-1/`.

### Phase C — Helm HPA and scaling examples

- **C1.** `helm/s3-encryption-gateway/examples/values-hpa-tuned.yaml` — new

- **C2.** `helm/s3-encryption-gateway/examples/values-keda-example.yaml` — new

  A standalone KEDA `ScaledObject` manifest (not a Helm-rendered resource)
  that shows how to scale on `http_requests_total` rate via Prometheus adapter:

  ```yaml
  apiVersion: keda.sh/v1alpha1
  kind: ScaledObject
  metadata:
    name: s3-encryption-gateway
  spec:
    scaleTargetRef:
      name: s3-encryption-gateway
    minReplicaCount: 2
    maxReplicaCount: 20
    cooldownPeriod: 300
    triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus-operated.monitoring.svc:9090
        metricName: s3_gateway_requests_per_second
        threshold: "100"      # scale when avg req/s per replica exceeds 100
        query: >
          sum(rate(http_requests_total{
            job="s3-encryption-gateway"
          }[2m])) / count(kube_pod_info{
            pod=~"s3-encryption-gateway-.*"
          })
  ```

  The file includes inline comments explaining each field and a note on
  when KEDA is preferable to CPU-based HPA.

## 6. Test Strategy

| Layer | Tests |
|---|---|
| Smoke profile CI gate | `TestConformance_RangeRead` + `TestConformance_Multipart` under `SMOKE_ENV` via `make test-load-smoke`; must complete in < 120 s with 0 errors |
| Soak profile (nightly) | `TestConformance_RangeRead` + `TestConformance_Multipart` under `SOAK_ENV` (existing); must maintain p99 latency ≤ SLO threshold defined in §3.2 |
| Spike profile (manual / release gate) | Same tests under `SPIKE_ENV`; validates HPA-triggering conditions; 0 errors expected; heap high-water-mark ≤ `maxReplicas × memory limit` projected capacity |
| High-throughput profile (manual) | Large-object MPU under `HIGH_THROUGHPUT_ENV`; validates `terminationGracePeriodSeconds` sizing; 0 objects corrupted (byte-for-byte verified via existing hash assertion in `loadtest_summary.go`) |
| Helm schema validation | `helm lint helm/s3-encryption-gateway` passes with `values-hpa-tuned.yaml` applied; `helm template ... -f values-hpa-tuned.yaml` renders valid KUBE objects |
| NDJSON output contract | `TestLoadtestSummary_JSONFields` — assert all required fields (`throughput_mbps`, `latency_ns.p50`, `latency_ns.p95`, `latency_ns.p99`, `errors`, `heap_inuse_max_bytes`) are present and non-negative in every emitted record |
| SLO annex completeness | `TestSLOSummaryNoTBD` — CI-gated test that greps `docs/perf/v0.6-qa-1/slo-summary.md` for the string "TBD" and fails if any is found (ensures Phase B numbers are committed before merge) |

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Testcontainer-backed spike profile (50 workers) produces CPU-throttled numbers on shared CI runners, invalidating the sizing table | Run Phase B on a dedicated machine (developer workstation or a `c5.2xlarge`-class CI runner) and document the runner class in `docs/perf/v1.0-perf-1/README.md`; note that relative percentages are more useful than absolute MB/s for sizing guidance |
| `terminationGracePeriodSeconds` recommendation too short for 500 MiB MPU in the high-throughput profile | Use 95th-percentile upload duration + 30 s safety margin (not p99); document the formula and the input measurement so operators can re-derive it for their object sizes |
| `values-hpa-tuned.yaml` diverges from `values.schema.json` constraints and fails `helm lint` | Add schema test to CI as part of the existing `helm lint` check; validate the example in Phase C3 before committing |
| KEDA `ScaledObject` example becomes stale as KEDA API evolves | Add a `# Tested with KEDA v2.14` comment and a renovate-compatible comment for the KEDA API version; document in `SCALING.md` §4.2 that operators must verify against their installed KEDA version |
| SLO annex TBD rows not populated before merge (Phase B skipped) | `TestSLOSummaryNoTBD` CI gate fails the PR if any "TBD" string remains; this is a hard DoD requirement |
| Valkey sizing for high-concurrency MPU not adequately specified | Phase D `SCALING.md` §8 quantifies the key-per-upload storage footprint using the known `UploadState` struct size; add a conservative headroom multiplier of 2× and document the TTL-based eviction behaviour |
| `values.prod.yaml` `maxReplicas: 20` may be insufficient for very high traffic | Document that `maxReplicas` should be operator-specific; the example file uses 20 as a safe default for up to ~200 concurrent clients based on the sizing table; operators must raise it for larger workloads |

## 10. Follow-ups / Out-of-scope items surfaced during planning

- **KEDA Helm template.** A first-class `ScaledObject` Helm template (vs the
  standalone example YAML) would allow operators to manage KEDA via the chart.
  This requires adding KEDA CRDs as a chart dependency and is non-trivial.
  Deferred to v1.x as a separate issue.
- **Multi-provider spike profiles.** Phase B runs the spike profile only against
  MinIO; Garage and RustFS may behave differently under 50-worker load due to
  their concurrency models. Adding spike runs for those providers adds ~1 d and
  is deferred as a follow-up.
- **Cluster Autoscaler (CA) / Karpenter guidance.** Node-level autoscaling
  interacts with HPA (over-provisioning for scale-up speed vs bin-packing).
  This is cloud-provider-specific and belongs in a separate operations topic.
- **Circuit-breaker configuration from load data.** ADR 0010 defers
  circuit-breaker defaults to QA-1 baselines. The high-concurrency data from
  this plan feeds that decision; a follow-up issue should revisit ADR 0010
  after V1.0-PERF-1 merges.
