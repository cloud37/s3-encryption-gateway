# V1.0-MAINT-2 — Remove Compression Feature — Implementation Plan

Status: Draft
Owner: Crypto / Maintenance
Priority: P2
Labels: `milestone:v1.0`, `area:crypto`, `area:maintenance`

> Scope clarified relative to the tracker entry at
> `docs/issues/v1.0-issues.md:364`. This plan expands the issue's bullet-list
> scope into a full, deterministic specification. Two design decisions were
> confirmed during planning:
> 1. **Full constructor removal** — the `compressionEngine` parameter is
>    removed from *every* `NewEngine*` constructor (positional and
>    functional-options), all ~30+ call sites are updated, and
>    `NewEngineWithCompression` is deleted outright.
> 2. **Full metadata-handling removal** — compaction/expansion of the
>    `ce`/`ca`/`cos` keys and `IsCompressionMetadata` are removed entirely;
>    backward read compatibility for already-compressed objects is delegated
>    to a mandatory pre-upgrade migration note (decrypt + re-upload).

## 1. Context & Current State

The gateway ships an optional "compress-then-encrypt" feature implemented as a
standalone `CompressionEngine` that is threaded through the encryption engine.
The feature is opt-in (`compression.enabled: false` by default) and has three
structural and security problems that motivate removal:

1. **Cryptographic anti-pattern.** Compressing plaintext *before* encrypting it
   makes the ciphertext length a function of plaintext redundancy. When an
   attacker controls part of the plaintext, this leaks information about the
   secret part via observable ciphertext size — the CRIME/BREACH class of
   attacks (RFC 7457 §2.6) [1][4]. The gateway encrypts arbitrary
   user-supplied object bodies, so this is a real, not theoretical, exposure.
2. **Structural gap.** Compression is only applied on the legacy single-AEAD
   path. MPU parts and the chunked path are always stored uncompressed, so the
   feature is second-class and inconsistent.
3. **Maintenance burden with no production use.** The feature carries a
   dedicated engine (`internal/crypto/compression.go`, 203 lines), a config
   stanza, metadata constants, compaction/expansion logic, a decompression-bomb
   guard (V1.0-SEC-M05), Helm wiring, and a test suite — all for a path that
   has seen no significant production use.

the recommended external composition (`s4 → gateway`) for users who genuinely
need compression.

## 2. Design Goals & Non-Goals

### Goals

1. **Zero compression surface in crypto** — `internal/crypto/compression.go`
   deleted; the `CompressionEngine` interface, the `compressionEngine` struct,
   `NewCompressionEngine`, and `IsCompressionMetadata` no longer exist in the
   package.
2. **Clean constructor signatures** — every `NewEngine*` constructor loses its
   `compressionEngine` positional parameter; `NewEngineWithCompression` is
   deleted; `NewEngineWithOpts` becomes `NewEngineWithOpts(password []byte,
   opts ...Option)`. All call sites updated. Build and `go vet` clean.
3. **Config removed** — `CompressionConfig`, the `Config.Compression` field,
   its defaults, and its hot-reload immutability checks are gone; remaining
   config tests pass.
4. **Metadata constants removed** — `MetaCompression*` constants and all
   compaction/expansion of `ce`/`ca`/`cos` are removed; metadata round-trip
   tests for non-compression keys still pass.
5. **Tests converted, not weakened** — every compression-specific test is
   deleted; every shared test (constructor callers, metadata compaction,
   policy) is updated to the new signatures and still asserts its original
   non-compression intent.
6. **Operator surface cleaned** — Helm values/schema/template/README,
   `config.yaml.example`, and end-user docs no longer mention compression and
   instead document the `s4 → gateway` pattern.
7. **Safe upgrade path documented** — `docs/MIGRATION.md` gains a mandatory
   pre-upgrade section instructing operators to decrypt + re-upload any object
   carrying `x-amz-meta-compression-enabled: true` before upgrading.

### Non-Goals

1. **No replacement compression mechanism** — the gateway will not gain any
   built-in compression; composition with `s4` is the documented answer.
   (Out of scope by issue intent.)
2. **No automated re-encryption tooling for compressed objects** — detection
   and re-upload of legacy compressed objects is operator-driven via the
   migration note, not a new `s3eg-migrate` sub-command. (Could be a follow-up;
   see §10.)
3. **No change to the chunked/MPU encryption format** — those paths never
   compressed, so removal does not touch their wire format. (Avoids risk to the
   primary data path.)
4. **No removal of the V1.0-SEC-M05 CHANGELOG history** — the historical
   CHANGELOG entry stays; only live code is removed and a new CHANGELOG entry
   records the removal. (Preserves audit trail.)

## 3. Target Interface / Semantics / Behaviour

### 3.1 Engine constructors (post-removal)

The `compressionEngine` positional parameter is removed everywhere. The single
real constructor and the public wrappers become:

```go
// NewEngine creates a new encryption engine with the given password.
func NewEngine(password []byte) (EncryptionEngine, error)

// NewEngineWithOptions creates an engine with algorithm options.
func NewEngineWithOptions(password []byte, preferredAlgorithm string, supportedAlgorithms []string) (EncryptionEngine, error)

// NewEngineWithProvider adds provider-specific settings.
func NewEngineWithProvider(password []byte, preferredAlgorithm string, supportedAlgorithms []string, provider string) (EncryptionEngine, error)

// NewEngineWithChunking adds chunked-mode support.
func NewEngineWithChunking(password []byte, preferredAlgorithm string, supportedAlgorithms []string, chunkedMode bool, chunkSize int) (EncryptionEngine, error)

// NewEngineWithChunkingAndProvider is the single underlying constructor.
func NewEngineWithChunkingAndProvider(password []byte, preferredAlgorithm string, supportedAlgorithms []string, chunkedMode bool, chunkSize int, provider string, pbkdf2Iterations int) (EncryptionEngine, error)
```

`NewEngineWithCompression` is **deleted** (no replacement). The functional
options constructor becomes:

```go
// NewEngineWithOpts is the preferred constructor for new callers.
func NewEngineWithOpts(password []byte, opts ...Option) (EncryptionEngine, error)
```

**Rationale:** Removing the always-`nil` parameter follows the Go idiom of not
carrying dead parameters; the functional-options form is already the preferred
shape (`engine_options.go:139-147`), and dropping the vestigial 2nd arg makes
the option list the sole extension point. [3]

### 3.2 Engine struct & Encrypt/Decrypt semantics

- The `compressionEngine CompressionEngine` field (`engine.go:119`) is removed,
  along with its assignment (`engine.go:232`).
- `Encrypt` no longer compresses: the block at `engine.go:451-473` is removed;
  `toEncryptReader` is initialised directly from `bytes.NewReader(plaintext)`
  and that is what is sealed. The `compressionMetadata` map and its merge
  (`engine.go:548-552`) are removed.
- `Decrypt` no longer decompresses: the block at `engine.go:869-887` (including
  the V1.0-SEC-M05 `io.LimitReader` guard) is removed; `finalReader` is the
  decrypted plaintext directly.
- The metadata-stripping loops that call `IsCompressionMetadata`
  (`engine.go:587`, `:893`, `:971`, `:1278`, `:1739`, `:1955`) drop the
  `|| IsCompressionMetadata(k)` clause and keep only `IsEncryptionMetadata(k)`.
- The fallback paths (`engine.go:1629-1652`, `:1934-1955`) lose their
  compression/decompression blocks identically.

**Behavioural contract after removal:** for any object the *current* gateway
would write, output is byte-for-byte identical except that no
`x-amz-meta-compression-*` headers are ever emitted (they never were unless
`compression.enabled: true`, which is non-default). For an object written by an
*older* gateway with compression enabled, `Decrypt` will AEAD-verify and return
the still-gzip-compressed bytes **without** decompressing — this is why the
pre-upgrade migration note (§3.5) is mandatory.

### 3.3 Metadata compaction (post-removal)

`metadata_compaction.go` loses all `ce`/`ca`/`cos` handling:

- Compaction block `:136-145` removed.
- Expansion block `:217-225` removed.
- `IsCompressionMetadata` fallthrough at `:150` and `:229` reduced to
  `IsEncryptionMetadata(key)` only.
- `"x-amz-meta-ce", "x-amz-meta-ca", "x-amz-meta-cos"` removed from the
  `compactedKeys` list (`:248-249`).

**Rationale:** With no decompression on read, retaining read-side expansion
would surface compressed bytes to clients silently. Full removal plus the
migration contract is the chosen, confirmed approach.

### 3.4 Config (post-removal)

The `compression:` YAML stanza is removed from the schema. `Config` no longer
has a `Compression` field. The three hot-reload immutability checks
(`config.go:2273-2282`) are deleted. No new config keys are introduced.

### 3.5 Migration note contract

`docs/MIGRATION.md` gains a new `## Removing Compression (v1.0+)` section
stating:

> Objects written with `compression.enabled: true` carry the
> `x-amz-meta-compression-enabled: true` marker. Gateway versions ≥ the release
> that removes compression **will not decompress them** and will return the raw
> gzip bytes to clients. Before upgrading, for each affected object: download
> (decrypt) it through the *old* gateway, then re-upload it through a gateway
> with compression disabled (or any version, since default is off). Use
> `s3eg-migrate` to enumerate objects; filter on the compression marker.

For users who still need compression, document the composition:

```
client → s4 (https://github.com/abyo-software/s4) → s3-encryption-gateway → storage
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| A missed `NewEngineWith*(... , nil, ...)` call site breaks the build. | `go build ./...` after Phase H is the gate; the compiler enumerates every missed site. The §4 grep list (`NewEngineWith`) is the authoritative caller inventory. |
| Removing the SEC-M05 `io.LimitReader` reintroduces a decompression-bomb path. | Removal is safe **only because** the decompression code is removed entirely — there is no `gzip.NewReader` left to bomb. DoD checks that no `gzip`/`Decompress` symbol remains in `internal/crypto`. |
| Old compressed objects silently return gzip bytes to clients after upgrade. | Mandatory pre-upgrade migration note (§3.5) in `docs/MIGRATION.md`; promoted to DoD. Default config has compression off, so most deployments are unaffected. |
| `metadata_compaction.go` edits leave a dangling short-key (`ce/ca/cos`) in `compactedKeys`, causing inconsistent round-trips. | Remove from `compactedKeys` (`:248-249`) and from both compaction/expansion in the same phase (D1); `metadata_compaction_test.go` round-trip tests verify no orphan keys. |
| Unused-import / unused-variable compile errors after stripping branches (e.g. `strconv`, harness option fields). | `go vet`/`golangci-lint` in the verify block catch these; Phase B6 explicitly checks engine imports and Phase C5 removes now-unused harness options. |
| Helm `values.schema.json` JSON becomes invalid after block removal (trailing comma). | `helm lint` gate; validate JSON with `python -m json.tool` on the file before commit. |
| Per-bucket policy YAML in user configs still contains a `compression:` block after upgrade. | Policy parsing ignores unknown keys (no schema enforcement at runtime); document in the migration note that stray `compression:` blocks are inert and may be deleted. |
