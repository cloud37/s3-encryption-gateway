## 1. Context & Current State

V1.0-SEC-H03 laid the complete infrastructure for selectable KDFs:

- `internal/crypto/kdf.go:32-40` — `KDFParams` struct already carries
  `Time`, `Memory`, and `Threads` fields for argon2id.
- `internal/crypto/kdf.go:42-52` — `FormatKDFParams` already emits the
  `argon2id:t:m:p` wire format.
- `internal/crypto/kdf.go:54-107` — `ParseKDFParams` already parses the
  `argon2id:t:m:p` wire format and validates `memory > 0`.
- `internal/crypto/kdf_argon2_default.go:1-13` (`//go:build !fips`) —
  `deriveKeyArgon2id` is a stub that returns `ErrArgon2NotEnabled`.
- `internal/crypto/kdf_argon2_fips.go:1-7` (`//go:build fips`) —
  `deriveKeyArgon2id` returns `ErrAlgorithmNotApproved`.
- `internal/crypto/engine.go:349-365` — `deriveKeyWithParams` already has a
  `switch` arm for `KDFAlgArgon2id` that delegates to `deriveKeyArgon2id`.
- `internal/config/config.go:107-113` — `KDFConfig` only contains a
  `PBKDF2 PBKDF2Config` field; there is no `Algorithm` selector and no
  `Argon2id Argon2idConfig` sub-struct.
- `internal/config/config.go:384` — `EncryptionConfig.KDF` defaults to
  `PBKDF2{Iterations: 600000}` with no algorithm field.
- `helm/s3-encryption-gateway/values.yaml:339-347` — `kdf.pbkdf2.iterations`
  exists; no `kdf.algorithm` or `kdf.argon2id` stanza.
- `helm/s3-encryption-gateway/values.schema.json:872-900` — Schema has the
  `kdf.pbkdf2.iterations` object; no `algorithm` or `argon2id` properties.

**What is missing:**
1. A real `deriveKeyArgon2id` body in the `!fips` build.
2. A config-level `algorithm` selector so operators can choose argon2id.
3. Config validation that rejects `algorithm: argon2id` in FIPS builds.
4. Engine wiring so that *new* encryptions use argon2id when configured,
   while *decrypt* automatically handles both PBKDF2 and argon2id from
   `MetaKDFParams` metadata.
5. KAT tests, conformance tests, migration runbook updates, and Helm schema
   updates.

**Why argon2id:** OWASP 2024 rates argon2id as the #1 password-hashing
recommendation. It is memory-hard, which raises the cost of hardware-based
offline attacks far beyond what PBKDF2 can achieve at any practical iteration
count. For non-FIPS deployments this is a meaningful security upgrade.

## 2. Design Goals & Non-Goals

### Goals

1. **Selectable KDF at runtime** — operators set
   `encryption.kdf.algorithm: argon2id` (non-FIPS) or keep the default
   `pbkdf2-sha256`.
2. **Backwards-compatible decrypt** — objects written with PBKDF2 (100k or
   600k) continue to decrypt transparently; objects written with argon2id
   decrypt transparently on non-FIPS gateways.
3. **FIPS gate** — FIPS builds (`-tags fips`) reject `algorithm: argon2id`
   at startup with `ErrAlgorithmNotApproved`.
4. **OWASP-recommended defaults** — argon2id defaults to `m=19456, t=2, p=1`
   (OWASP 2024 minimum).
5. **Operator-tunable parameters** — `time`, `memory`, and `threads` are
   configurable via `encryption.kdf.argon2id.*`.
6. **Migration path** — `s3eg-migrate --filter=kdf` supports
   PBKDF2→argon2id and argon2id→PBKDF2 re-encryption because the algorithm
   is encoded in `MetaKDFParams`.
7. **Zero material leakage** — the derived key is zeroized after use via the
   existing `zeroBytes()` convention.

### Non-Goals

1. **FIPS-validated argon2id** — out of scope; no FIPS 140-3 validated
   implementation exists. FIPS builds continue to use PBKDF2 exclusively.
2. **Auto-tuning argon2id parameters** — deferred; operators must set
   parameters manually. Future work may add a calibration CLI.
3. **Default algorithm change** — the default remains `pbkdf2-sha256` so that
   existing deployments are not surprised by a memory-hard KDF that may
   exhaust small containers.
4. **Pre-emptive re-encryption of all objects** — not required; migration is
   operator-driven via `s3eg-migrate`.

## 3. Target Interface / Semantics / Behaviour

### 3.1 Config Schema

```go
// KDFConfig is the top-level key-derivation configuration block.
type KDFConfig struct {
    // Algorithm selects the KDF for new objects.
    //   "pbkdf2-sha256" (default) — FIPS-compliant
    //   "argon2id"                — memory-hard, non-FIPS only
    Algorithm string `yaml:"algorithm" env:"ENCRYPTION_KDF_ALGORITHM"`

    // PBKDF2 holds PBKDF2-SHA256 parameters (active when Algorithm == "pbkdf2-sha256").
    PBKDF2 PBKDF2Config `yaml:"pbkdf2"`

    // Argon2id holds argon2id parameters (active when Algorithm == "argon2id").
    Argon2id Argon2idConfig `yaml:"argon2id"`
}

// Argon2idConfig holds operator-tunable argon2id parameters.
//
// Recommended production values (OWASP 2024):
//   Time    = 2
//   Memory  = 19456  (19 MiB)
//   Threads = 1
//
// These are the minimums; operators may raise them for higher attack cost.
type Argon2idConfig struct {
    Time    uint32 `yaml:"time"    env:"ENCRYPTION_KDF_ARGON2ID_TIME"`
    Memory  uint32 `yaml:"memory"  env:"ENCRYPTION_KDF_ARGON2ID_MEMORY"`
    Threads uint8  `yaml:"threads" env:"ENCRYPTION_KDF_ARGON2ID_THREADS"`
}
```

**Rationale:** A single `Algorithm` string keeps the config flat and mirrors
the `KDFAlgorithm` type used in `KDFParams`. The nested `PBKDF2` and
`Argon2id` structs hold provider-specific parameters without polluting the
top-level namespace.

### 3.2 Engine Behaviour

**Encrypt (new objects):**
- If `e.kdfAlgorithm == KDFAlgArgon2id`:
  - `MetaKDFParams` is set to `FormatKDFParams(KDFParams{Algorithm: KDFAlgArgon2id, Time: t, Memory: m, Threads: p})`.
  - The per-object key is derived via `argon2.IDKey(password, salt, t, m, p, aesKeySize)`.
- Otherwise (default):
  - `MetaKDFParams` is set to `FormatKDFParams(DefaultKDFParams(e.pbkdf2Iterations))`.
  - Derivation uses PBKDF2 as today.

**Decrypt (existing objects):**
- `ParseKDFParams` reads `MetaKDFParams` from metadata.
- If `Algorithm == KDFAlgArgon2id`, `deriveKeyWithParams` calls
  `deriveKeyArgon2id(salt, params)`.
- In FIPS builds this returns `ErrAlgorithmNotApproved`.
- In non-FIPS builds with argon2id compiled in, this returns the derived key.
- In non-FIPS builds without argon2id (stub), this returns `ErrArgon2NotEnabled`.

### 3.3 FIPS Build Rejection

At `config.Validate()` time (in FIPS builds only):

```go
if cfg.Encryption.KDF.Algorithm == "argon2id" {
    return fmt.Errorf("%w: argon2id is not approved in FIPS builds", ErrAlgorithmNotApproved)
}
```

This uses a compile-time gate (`//go:build fips`) for the validation
function or a runtime check against a `buildtags.IsFIPS()` helper.

### 3.4 Wire Format (already implemented by V1.0-SEC-H03)

| Algorithm | `MetaKDFParams` value |
|---|---|
| PBKDF2-SHA256 | `pbkdf2-sha256:600000` |
| argon2id | `argon2id:2:19456:1` |

## 4. Work Breakdown

### Phase A — Core argon2id implementation
- **A1.** Replace stub body in `internal/crypto/kdf_argon2_default.go`.
  - Nature: edit
  - Import `golang.org/x/crypto/argon2`.
  - Implement `deriveKeyArgon2id(password []byte, params KDFParams) ([]byte, error)`:
    ```go
    func deriveKeyArgon2id(password []byte, params KDFParams) ([]byte, error) {
        if params.Memory == 0 {
            return nil, fmt.Errorf("argon2id: memory must be > 0")
        }
        key := argon2.IDKey(password, nil, params.Time, params.Memory, params.Threads, aesKeySize)
        // argon2.IDKey accepts a nil salt because the caller already
        // incorporates the per-object salt into the password or the
        // parameters. In our architecture the salt is passed separately
        // to deriveKeyWithParams; we pass it as the salt parameter.
        // Corrected implementation:
        //   key := argon2.IDKey(password, salt, params.Time, params.Memory, params.Threads, aesKeySize)
        return key, nil
    }
    ```
  - Note: `deriveKeyArgon2id` must accept the salt as a parameter.
    Update the signature from `deriveKeyArgon2id(_ []byte, _ KDFParams)` to
    `deriveKeyArgon2id(password, salt []byte, params KDFParams) ([]byte, error)`.
    Update the call site in `engine.go:deriveKeyWithParams` to pass `salt`.

- **A2.** Add `kdfAlgorithm KDFAlgorithm` and `argon2idParams Argon2idConfig`
  fields to the `engine` struct (`internal/crypto/engine.go:133`).
  - Nature: edit
  - Default `kdfAlgorithm = KDFAlgPBKDF2SHA256`.

- **A3.** Update `deriveKeyWithParams` (`engine.go:349-365`) to pass `salt`
  to `deriveKeyArgon2id`:
  ```go

      return deriveKeyArgon2id(e.password, salt, params)
  ```

- **A4.** Update `deriveKey` (`engine.go:369-371`) to respect the configured
  algorithm:
  ```go
  func (e *engine) deriveKey(salt []byte) ([]byte, error) {
      switch e.kdfAlgorithm {
      case KDFAlgArgon2id:
          return e.deriveKeyWithParams(salt, KDFParams{
              Algorithm: KDFAlgArgon2id,
              Time:      e.argon2idParams.Time,
              Memory:    e.argon2idParams.Memory,
              Threads:   e.argon2idParams.Threads,
          })
      default:
          return e.deriveKeyWithParams(salt, DefaultKDFParams(e.pbkdf2Iterations))
      }
  }
  ```

- **A5.** Update all encrypt paths (`Encrypt`, `encryptChunked`, etc.) so that
  `MetaKDFParams` is written using the active algorithm and parameters.
  - Replace every `FormatKDFParams(DefaultKDFParams(e.pbkdf2Iterations))`
    with `FormatKDFParams(e.defaultKDFParams())`.
  - Add helper:
    ```go
    func (e *engine) defaultKDFParams() KDFParams {
        switch e.kdfAlgorithm {
        case KDFAlgArgon2id:
            return KDFParams{
                Algorithm: KDFAlgArgon2id,
                Time:      e.argon2idParams.Time,
                Memory:    e.argon2idParams.Memory,
                Threads:   e.argon2idParams.Threads,
            }
        default:
            return DefaultKDFParams(e.pbkdf2Iterations)
        }
    }
    ```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| argon2id memory exhaustion in small containers (e.g. 256 MiB limit) | Default `memory=19456` (19 MiB) is conservative. Document that operators in tiny containers should keep the default PBKDF2 algorithm. Validation warns if `memory > 65536` (64 MiB). |
| FIPS build accidentally accepts argon2id config | Startup validation in `config.Validate()` gated by `buildtags.IsFIPS()` rejects `algorithm: argon2id` before the engine is constructed. Unit test `TestConfig_Validate_Argon2id_FIPS_Rejected` guards regression. |
| Cross-build function signature drift (`kdf_argon2_default.go` vs `_fips.go`) | Both files are edited in the same PR; CI compiles both `go test ./...` and `go test -tags=fips ./...` to catch signature mismatches. |
| Existing objects with `MetaKDFParams` absent cannot be read after algorithm change | Decrypt reads the algorithm from metadata; absent metadata falls back to legacy PBKDF2 100k as before. The engine's *configured* algorithm only affects *new* writes. |
| go.mod drift — `golang.org/x/crypto/argon2` may not be present | The project already depends on `golang.org/x/crypto` (ChaCha20-Poly1305). argon2 is in the same module; no new dependency required. Verify with `go mod tidy`. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **Auto-tuning CLI for argon2id parameters** — a small tool that benchmarks
   `argon2.IDKey` with varying `m` and `t` to find the highest parameters
   that complete within a target latency (e.g. 100 ms). Deferred because
   operator-set parameters are sufficient for v1.0.
2. **Argon2id GPU-resistance analysis** — document the trade-offs between
   `memory` and `threads` for multi-core vs GPU attackers. Can be added to
   the ADR later; not blocking implementation.
