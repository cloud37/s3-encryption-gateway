## 1. Context & Current State

### KeyManager Interface

The `KeyManager` interface (`internal/crypto/keymanager.go:24`) exposes six
methods: `Provider()`, `WrapKey()`, `UnwrapKey()`, `ActiveKeyVersion()`,
`HealthCheck()`, and `Close()`. The `KeyEnvelope` struct
(`internal/crypto/keymanager.go:72`) carries `KeyID`, `KeyVersion`,
`Provider`, `Ciphertext`, and `CreatedAt`. The `RotatableKeyManager`
interface (`internal/crypto/keymanager.go:91`) extends `KeyManager` with
`PrepareRotation(ctx, *int) (RotationPlan, error)` and
`PromoteActiveVersion(ctx, RotationPlan) error`.

Sentinel errors `ErrProviderUnavailable`, `ErrKeyNotFound`,
`ErrUnwrapFailed`, `ErrInvalidEnvelope`, `ErrRotationNotSupported`, and
`ErrRotationAmbiguous` are all defined in `internal/crypto/keymanager.go`.

### Existing Adapters

Two production adapters exist today: Cosmian KMIP
(`internal/crypto/keymanager_cosmian.go`) and in-memory AES key-wrap
(`internal/crypto/keymanager_memory.go`). The `self_contained` adapter
(V1.0-KMS-4) provides AES-256-GCM and RSA-OAEP wrapping entirely in-process.
No adapter today delegates to HashiCorp Vault Transit.

### Registry

`internal/crypto/keymanager_registry.go:69` registers `memory` in `init()`.
The `cosmian` adapter is registered from `internal/api/crypto_factory.go`.
The `self_contained` adapter registers via its own `init()`. A new `vault`
(and alias `vault-transit`) adapter must follow the same pattern.

### Configuration

`internal/config/config.go:303` defines `KeyManagerConfig` with a comment
at line 299–301 explicitly listing `"vault" or "vault-transit": HashiCorp
Vault Transit (see V1.0-KMS-3)` as a planned v1.0 provider. Lines 311–312
contain commented-out stub fields:

```go
// AWS        AWSKMSConfig  `yaml:"aws"`
// Vault      VaultConfig   `yaml:"vault"`
```

`config.go:1694-1695` (the `Validate()` switch) currently rejects `"vault"`
and `"vault-transit"` as unsupported. The `loadFromEnv()` function has no
`VAULT_*` wiring.

`config.yaml.example:102` mentions `"vault"` as a planned provider with
`# Planned (v1.0): "aws", "vault"`.

### Dependency Gap

`go.mod` does not include `github.com/hashicorp/vault/api`. The module must
be added before the adapter compiles.

### Why the Existing State Is Insufficient

The gateway has no mechanism to call Vault Transit `encrypt` / `decrypt`
endpoints. Operators who manage keys in HashiCorp Vault (including Vault OSS
with the Transit secrets engine available since v0.6.2) must currently use
the `self_contained` adapter with a locally-held key, which does not benefit
from Vault's audit log, key versioning server, or centralized access policy.
This plan implements the missing adapter end-to-end.

## 2. Design Goals & Non-Goals

### Goals

1. **Interface compliance** — `vaultTransitManager` satisfies `KeyManager`
   at compile time. `ConformanceSuite` passes. A `var _ KeyManager`
   compile-time assertion is present.

2. **Transit encrypt/decrypt** — `WrapKey` calls the Vault Transit `encrypt`
   endpoint; ciphertext is the Vault-formatted `vault:v{N}:...` base64
   string stored verbatim in `KeyEnvelope.Ciphertext`. `UnwrapKey` calls
   `decrypt` and returns the plaintext DEK bytes.

3. **Key versioning / dual-read** — `ActiveKeyVersion` returns the current
   Vault key version obtained from the key metadata endpoint. `UnwrapKey`
   passes the ciphertext envelope as-is; Vault's Transit engine handles
   version dispatch internally. The adapter additionally exposes a
   `RotatableKeyManager` implementation that calls Vault's key rotation API.

4. **Multiple auth methods** — Supports token, AppRole, AWS IAM, and
   Kubernetes auth methods selectable via config. The factory resolves the
   correct `vault/api` login path at construction time.

5. **mTLS and CA verification** — Vault client is always constructed with a
   configurable `tls.Config`; `InsecureSkipVerify` is possible but requires
   explicit opt-in.

6. **No credential logging** — Vault tokens, AppRole secrets, and AWS
   credentials are never written to any log field, error string, or
   `KeyEnvelope` field. A log-scraping unit test enforces this invariant.

7. **DEK zeroization** — Plaintext bytes returned by Vault's Transit decrypt
   are zeroized after the `UnwrapKey` caller processes them; internal copies
   in the adapter are cleared before the method returns.

8. **Health check** — `HealthCheck` calls the Vault `/v1/sys/health`
   endpoint (lightweight; does not perform encryption).

9. **Context propagation** — All HTTP calls to Vault use `req.WithContext(ctx)`;
   cancellation is honoured.

10. **Registry integration** — The adapter registers under names `"vault"`
    and `"vault-transit"` via `init()` in the new file.

### Non-Goals

1. **Vault Enterprise-only features** — Namespaces, performance replicas,
   and HSM auto-unseal are not exercised. The adapter sends requests to the
   configured path; operators may prefix `namespace/` in the transit path.

2. **Lease renewal / token renewal daemon** — Token renewal is delegated to
   a Vault Agent or the operator's deployment tooling. The adapter will fail
   closed when the token expires; a prominent log.Warn fires when remaining
   TTL falls below a configurable threshold.

3. **FIPS-gated build path** — TLS and crypto operations use the standard
   library. No separate `_fips.go` file is needed for this adapter.

4. **AWS KMS adapter** — Covered by V1.0-KMS-2 in a sibling issue.

5. **Lazy re-wrap of historical DEKs** — Post-rotation objects retain their
   original Vault ciphertext; re-wrapping happens on the next write.

## 3. Target Interface / Semantics / Behaviour

### 3.1 `vaultTransitManager` struct

```go
// vaultTransitManager wraps DEKs using HashiCorp Vault Transit secrets engine.
//
// Invariants:
//   - Thread-safe: all exported methods acquire mu (RLock for reads, Lock
//     for Close). The Vault API client is safe for concurrent use.
//   - Context propagation: every Vault HTTP request is decorated with the
//     caller's ctx via client.AddHeader on a per-request clone, or via
//     a context-aware request helper. Returns ctx.Err() wrapped on cancellation.
//   - DEK zeroization: plaintext bytes received from Vault Transit decrypt
//     are zeroed within the adapter after copying to the caller-owned slice.
//     The caller is then responsible for zeroizing the returned slice.
//   - No credential logging: tokens, AppRole secrets, AWS creds are never
//     formatted into any log message, error string, or KeyEnvelope field.
//   - Post-close: all methods return ErrProviderUnavailable after Close.
//   - Ciphertext format: the Vault Transit ciphertext string
//     "vault:v{N}:base64payload" stored as []byte in KeyEnvelope.Ciphertext.
//   - KeyID: "vault-transit:{transitPath}/{keyName}" — stable across versions.
//   - Key version: parsed from the "vault:v{N}:" prefix of a fresh WrapKey
//     response. For UnwrapKey, version is extracted from the envelope's
//     existing ciphertext; no version lookup is performed.
type vaultTransitManager struct {
    mu           sync.RWMutex
    client       *api.Client          // authenticated Vault API client
    transitPath  string               // e.g. "transit"
    keyName      string               // e.g. "s3gw-dek"
    providerName string               // defaults to "vault-transit"
    closed       bool
    opts         vaultTransitOptions  // immutable after construction
}
```

**Rationale:** A single struct with an `*api.Client` follows the same mutex
discipline as `cosmianKMIPManager` (`internal/crypto/keymanager_cosmian.go:41`).
The Vault API client is goroutine-safe per its documentation, so the mutex
protects only the `closed` flag and hot-path guard; no per-request locking
is needed for the actual HTTP call.

### 3.2 `WrapKey` semantics

```
WrapKey(ctx, plaintext []byte, metadata map[string]string) (*KeyEnvelope, error)
```

1. Check `ctx.Err()`.
2. RLock; check `closed → ErrProviderUnavailable`.
3. Base64-encode `plaintext` as the `"plaintext"` field of the Transit
   encrypt request body.
4. POST to `{transitPath}/encrypt/{keyName}` with context-bound request.
5. Extract `data.ciphertext` from the JSON response.
6. Parse `v{N}:` from the ciphertext prefix to populate `KeyVersion`.
7. Return `KeyEnvelope{KeyID: keyID, KeyVersion: N, Provider: providerName, Ciphertext: []byte(ciphertext), CreatedAt: time.Now()}`.
8. On any Vault error: `fmt.Errorf("keymanager/vault-transit: wrap: %w", err)`.

### 3.3 `UnwrapKey` semantics

```
UnwrapKey(ctx, envelope *KeyEnvelope, metadata map[string]string) ([]byte, error)
```

1. Check `ctx.Err()`.
2. Validate `envelope != nil && len(envelope.Ciphertext) > 0` (→ `ErrInvalidEnvelope`).
3. RLock; check `closed → ErrProviderUnavailable`.
4. POST to `{transitPath}/decrypt/{keyName}` with `"ciphertext"` set to the
   envelope ciphertext string.
5. Extract and base64-decode `data.plaintext` from the response.
6. Make a caller-owned copy; zeroize the adapter-held decode buffer.
7. On Vault `400`/`404` error (invalid ciphertext, unknown key version):
   `fmt.Errorf("%w: %s", ErrUnwrapFailed, vaultErrMsg)`.

### 3.4 `ActiveKeyVersion` semantics

1. GET `{transitPath}/keys/{keyName}`.
2. Parse `data.latest_version` (integer) from the response.
3. Return it.

### 3.5 `HealthCheck` semantics

1. GET `/v1/sys/health` with `standbyok=true&perfstandbyok=true` query params.
2. Accept HTTP 200, 429 (standby), 472 (DR replication), 473 (perf standby)
   as healthy.
3. Return `ErrProviderUnavailable` for any other status or network error.

### 3.6 `Close` semantics

1. Lock; set `closed = true`.
2. If any token material is held in memory (e.g., AppRole token), zeroize it.
3. Second call is a no-op returning nil.

### 3.7 `PrepareRotation` semantics (RotatableKeyManager)

```
PrepareRotation(ctx, target *int) (RotationPlan, error)
```

1. Call `ActiveKeyVersion(ctx)` to get `currentVersion`.
2. If `target != nil` and `*target <= currentVersion` → return
   `ErrRotationAmbiguous` (can only rotate forward in Vault).
3. Return `RotationPlan{CurrentVersion: currentVersion, TargetVersion: currentVersion + 1}`.
   (Vault auto-increments the version on rotate; the target is always N+1.)

### 3.8 `PromoteActiveVersion` semantics (RotatableKeyManager)

```
PromoteActiveVersion(ctx, plan RotationPlan) error
```

1. POST to `{transitPath}/keys/{keyName}/rotate` (no body required).
2. Call `ActiveKeyVersion(ctx)` and verify it equals `plan.TargetVersion`.
3. If mismatch: `fmt.Errorf("%w: expected %d got %d", ErrRotationConflict, …)`.

**Compile-time assertions (placed at package level):**
```go
var _ KeyManager           = (*vaultTransitManager)(nil)
var _ RotatableKeyManager  = (*vaultTransitManager)(nil)
```

### 3.9 Auth Method Resolution

The factory resolves auth at construction time via `vaultAuthStrategy`:

| `auth.method` | Vault login path | Required config fields |
|---|---|---|
| `"token"` (default) | n/a — token set directly on client | `auth.token` or `env:VAULT_TOKEN` |
| `"approle"` | `/auth/{auth.mount}/login` | `auth.role_id`, `auth.secret_id` or `"env:..."` |
| `"aws"` | `/auth/{auth.mount}/login` | AWS credentials from environment (standard SDK chain) |
| `"kubernetes"` | `/auth/{auth.mount}/login` | `auth.role`, `auth.jwt_path` (default `/var/run/secrets/kubernetes.io/serviceaccount/token`) |

Each auth strategy returns an `*api.Secret` from the login call, from which
`Secret.Auth.ClientToken` is extracted and set via `client.SetToken()`. The
token is stored only in the Vault client's internal state.

### 3.10 Go Config Struct

```go
// VaultConfig captures settings for the HashiCorp Vault Transit adapter.
//
// Supported auth methods: "token" (default), "approle", "aws", "kubernetes".
// See docs/KMS_COMPATIBILITY.md for per-method configuration details.
type VaultConfig struct {
    // Address is the Vault server address (e.g. "https://vault.internal:8200").
    Address string `yaml:"address" env:"VAULT_ADDR"`

    // TransitPath is the mount path of the Transit secrets engine.
    // Default: "transit".
    TransitPath string `yaml:"transit_path" env:"VAULT_TRANSIT_PATH"`

    // KeyName is the name of the Transit key used for DEK wrapping/unwrapping.
    // Required.
    KeyName string `yaml:"key_name" env:"VAULT_TRANSIT_KEY_NAME"`

    // Namespace is an optional Vault Enterprise namespace prefix
    // (e.g. "admin/"). Leave empty for OSS Vault.
    Namespace string `yaml:"namespace" env:"VAULT_NAMESPACE"`

    // Timeout is the per-request timeout for Vault API calls.
    // Default: 5s.
    Timeout time.Duration `yaml:"timeout" env:"VAULT_TIMEOUT"`

    // Auth holds authentication configuration.
    Auth VaultAuthConfig `yaml:"auth"`

    // TLS holds TLS configuration for the Vault client.
    TLS VaultTLSConfig `yaml:"tls"`
}

// VaultAuthConfig selects the Vault authentication method and its parameters.
type VaultAuthConfig struct {
    // Method selects the auth backend: "token", "approle", "aws", "kubernetes".
    // Default: "token".
    Method string `yaml:"method" env:"VAULT_AUTH_METHOD"`

    // Token is the Vault token for "token" auth method.
    // Prefer TokenEnv to avoid inline secrets.
    Token string `yaml:"token" env:"VAULT_TOKEN"`

    // TokenEnv is the environment variable holding the Vault token.
    TokenEnv string `yaml:"token_env" env:"VAULT_TOKEN_ENV"`

    // Mount is the auth backend mount path.
    // Defaults: "approle" for approle, "aws" for aws, "kubernetes" for kubernetes.
    Mount string `yaml:"mount" env:"VAULT_AUTH_MOUNT"`

    // RoleID is the AppRole role ID (approle auth).
    RoleID string `yaml:"role_id" env:"VAULT_ROLE_ID"`

    // SecretID is the AppRole secret ID (approle auth).
    // Prefer SecretIDEnv.
    SecretID string `yaml:"secret_id" env:"VAULT_SECRET_ID"`

    // SecretIDEnv is the environment variable holding the AppRole secret ID.
    SecretIDEnv string `yaml:"secret_id_env" env:"VAULT_SECRET_ID_ENV"`

    // Role is the role name for Kubernetes and AWS auth methods.
    Role string `yaml:"role" env:"VAULT_AUTH_ROLE"`

    // JWTPath is the path to the Kubernetes service account JWT file.
    // Default: "/var/run/secrets/kubernetes.io/serviceaccount/token".
    JWTPath string `yaml:"jwt_path" env:"VAULT_K8S_JWT_PATH"`
}

// VaultTLSConfig holds TLS settings for the Vault API client.
type VaultTLSConfig struct {
    // CAFile is the path to the CA certificate file.
    CAFile string `yaml:"ca_file" env:"VAULT_CACERT"`

    // CertFile is the path to the client certificate file.
    CertFile string `yaml:"cert_file" env:"VAULT_CLIENT_CERT"`

    // KeyFile is the path to the client key file.
    KeyFile string `yaml:"key_file" env:"VAULT_CLIENT_KEY"`

    // InsecureSkipVerify disables TLS certificate verification.
    // WARNING: never set true in production.
    InsecureSkipVerify bool `yaml:"insecure_skip_verify" env:"VAULT_SKIP_VERIFY"`

    // MinVersion is the minimum TLS version: "1.2" or "1.3". Default: "1.2".
    MinVersion string `yaml:"min_version" env:"VAULT_TLS_MIN_VERSION"`
}
```

**Rationale:** Follows the `CosmianConfig` pattern
(`internal/config/config.go:374`) — a typed sub-struct per provider with
explicit env tags, consistent with `loadFromEnv` conventions. Separating
`VaultAuthConfig` and `VaultTLSConfig` into dedicated structs mirrors
`ValkeyTLSConfig` and `SinkTLSConfig` precedents in the same file.

- For `"approle"`: `role_id` must be non-empty; `secret_id` or `secret_id_env` must be non-empty.
- For `"kubernetes"`: `role` must be non-empty.

   func (m *vaultTransitManager) WrapKey(ctx context.Context, plaintext []byte, _ map[string]string) (*KeyEnvelope, error) {
       if err := ctx.Err(); err != nil {
           return nil, fmt.Errorf("keymanager/vault-transit: %w", err)
       }
       m.mu.RLock()
       defer m.mu.RUnlock()
       if m.closed {
           return nil, ErrProviderUnavailable
       }
       encoded := base64.StdEncoding.EncodeToString(plaintext)
       path := fmt.Sprintf("%s/encrypt/%s", m.transitPath, m.keyName)
       secret, err := m.client.Logical().WriteWithContext(ctx, path, map[string]any{
           "plaintext": encoded,
       })
       if err != nil {
           return nil, fmt.Errorf("keymanager/vault-transit: wrap: %w", err)
       }
       ciphertext, ok := secret.Data["ciphertext"].(string)
       if !ok || ciphertext == "" {
           return nil, fmt.Errorf("keymanager/vault-transit: wrap: missing ciphertext in response")
       }
       version := parseVaultKeyVersion(ciphertext) // parses "vault:v{N}:" prefix
       return &KeyEnvelope{
           KeyID:      m.keyID(),
           KeyVersion: version,
           Provider:   m.providerName,
           Ciphertext: []byte(ciphertext),
           CreatedAt:  time.Now(),
       }, nil
   }

**F2.** `docs/adr/0014-vault-transit-adapter.md` — **new**

ADR documenting:
- **Context:** HashiCorp Vault is the most widely-deployed secrets management
  platform in cloud-native environments. The Vault Transit engine provides
  envelope-encryption semantics (encrypt/decrypt a payload) without exposing
  the KEK to the gateway.
- **Decision:** Implement a `KeyManager` adapter using `github.com/hashicorp/vault/api`
  calling Transit encrypt/decrypt directly. Vault handles key versioning
  server-side; the gateway stores the `vault:v{N}:...` ciphertext opaquely.
- **Alternatives considered:** Wrapping Vault KV v2 (stores the DEK
  encrypted in Vault KV — rejected: exposes the DEK in Vault's storage,
  defeats the purpose); DIY HTTPS client (rejected: `vault/api` provides
  auth method helpers, retry logic, and lease awareness).
- **Consequences:** Gateway depends on `github.com/hashicorp/vault/api`.
  Token expiry causes 403 errors until restart or Vault Agent is used. Vault
  must be unsealed and reachable on the encrypt/decrypt path.
- **Security properties:** DEKs are never stored in Vault; only the
  ciphertext produced by Transit is persisted. All Vault credentials
  (tokens, AppRole secrets) are zeroized on `Close()` and never logged.

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Vault token expires mid-operation causing 403 errors | Add `slog.Warn` at construction time if `Auth.Method == "token"` and no renewal daemon is configured; document Vault Agent sidecar pattern in ADR and KMS_COMPATIBILITY.md |
| `github.com/hashicorp/vault/api` brings in indirect deps with CVEs | Run `govulncheck ./...` as part of CI; pin the version to a specific tag; reject upgrade PRs that introduce new CVEs |
| TLS `InsecureSkipVerify: true` misconfigured in production | Add `slog.Error` in factory when `InsecureSkipVerify` is set to `true`; document the risk prominently in config.yaml.example comment; promoted to DoD |
| Token/AppRole secret ID appears in structured log via `%v` on error wrap | Add `TestVaultTransit_NoTokenInLogs` (§6); review all error format strings to never include credential variables |
| Vault DEV server uses HTTP not HTTPS; integration test may skip TLS | Integration test (`TestVaultTransit_PutGetRotate`) uses dev mode which listens on HTTP; test explicitly configures `InsecureSkipVerify=false` is skipped, and a separate TLS integration sub-test documents the mTLS path |
| `vault:v{N}:` prefix parse silently returns 0 for unexpected formats | `parseVaultKeyVersion` must never silently succeed with version 0; add `TestParseVaultKeyVersion` table covering malformed inputs; treat version 0 in `WrapKey` response as an error |
| Registry double-registration if `init()` is called twice (impossible in Go, but aliasing `"vault"` and `"vault-transit"` might conflict) | Both names are registered in a single `init()` call in the same file; `Register` panics on duplicate — verified in `keymanager_registry_test.go` |
| `PromoteActiveVersion` rotates in Vault but the gateway's in-process active version diverges | `PromoteActiveVersion` re-fetches `ActiveKeyVersion` post-rotate and compares to `plan.TargetVersion`; mismatch returns `ErrRotationConflict` |
| Kubernetes JWT file path unset defaults to wrong value | Default `"/var/run/secrets/kubernetes.io/serviceaccount/token"` is well-known; add config validation that the file exists at construction time for `kubernetes` auth, not at startup |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **Token renewal daemon** — Vault tokens have a TTL; the adapter fails
   closed on expiry. A background goroutine that renews the token lease
   (`/v1/auth/token/renew-self`) would allow long-running deployments without
   restart. Deferred to a follow-up issue; Vault Agent sidecar is the
   recommended alternative in v1.0.

2. **Vault Agent auto-auth sink** — When Vault Agent is present it writes
   a renewable token to a file. A `"file"` auth method sub-variant that
   re-reads the token file on each request would integrate cleanly. Deferred.

3. **Vault namespace support for Enterprise** — The `Namespace` field is
   wired but enterprise namespaces are not tested in CI. A follow-up issue
   can add a dedicated conformance tier gated by `//go:build vault_enterprise`.

4. **AWS auth method unit test** — `TestVaultTransit_Auth_AWS` requires an
   AWS-signed request which is difficult to mock cleanly. Integration with
   real AWS credentials is deferred; the config validation path is covered
   by `TestConfig_Vault_AWSAuth_Valid`.

5. **Concurrent token refresh** — If two goroutines simultaneously detect
   token expiry and attempt re-authentication, both will succeed but one
   token will be silently discarded. A `sync.Once` per-token-epoch pattern
   can fix this; deferred pending observability data on frequency.

### GAP-KMS3-1 — HIGH — `ErrRotationConflict` is not defined in `keymanager.go`

**Source:** Go specification — `errors.Is` contract requires the sentinel error to be accessible to callers.

**Description:** `ErrRotationConflict` is used in `PromoteActiveVersion` (§3.8) and asserted in `TestVaultTransit_PromoteActiveVersion_VersionMismatch`, but is NOT listed in the §1 sentinel errors inventory which enumerates all errors from `keymanager.go`. If the implementor adds this sentinel only to `keymanager_vault.go` rather than `keymanager.go`, the `errors.Is` contract breaks across package boundaries. If it is not added at all, the code will not compile.

**Required fix:** Add `ErrRotationConflict = errors.New("keymanager: rotation version conflict")` to `internal/crypto/keymanager.go`. Update the §1 inventory. Add a DoD checkbox: "`ErrRotationConflict` is defined in `internal/crypto/keymanager.go` and referenced from `PromoteActiveVersion`."

---

### GAP-KMS3-2 — HIGH — `zeroString(s *string)` cannot overwrite Go string bytes without `unsafe`

**Source:** *Serious Cryptography, 2nd Ed.* (Aumasson, No Starch Press, 2024), Ch. 3 — residual key material in memory is a tangible threat.

**Description:** Plan §B1 point 16 specifies `zeroString` as "overwrites underlying string bytes with zeros." In Go, strings are immutable — their backing `[]byte` array cannot be modified through any safe Go operation. The only mechanism is `unsafe.Pointer` with `unsafe.StringData(s)` (Go 1.20+). Without `unsafe`, `zeroString` is effectively a no-op, leaving Vault tokens and AppRole secret IDs in memory — a security-relevant gap.

**Required fix:** Explicitly document in §B1 that `zeroString` requires `import "unsafe"` and uses `unsafe.StringData(s)` (Go 1.20+) to obtain a pointer to the backing array. Alternatively, store Vault tokens as `[]byte` internally and only convert to `string` at API call sites, then zero the intermediate `[]byte` immediately. Add a DoD checkbox: "`zeroString` zeroing is verified by a test that checks memory contents after `Close()`."

### GAP-KMS3-3 — MEDIUM — Token expiry not detected by HealthCheck; `gateway_kms_healthy` gauge is inaccurate

**Source:** *API Security in Action* (Madden, Manning 2021), Ch. 11 — liveness vs. readiness divergence for token-authenticated systems.

**Description:** `HealthCheck` probes only `/v1/sys/health`. A Vault instance that is healthy but whose token has expired returns HTTP 200 from `/v1/sys/health` while returning HTTP 403 on all `WrapKey`/`UnwrapKey` calls. The `gateway_kms_healthy` gauge shows `1` (healthy) while all encryption operations fail. The DoD has no item requiring `HealthCheck` to detect token expiry.

**Required fix:** Extend `HealthCheck` to also call `/v1/auth/token/lookup-self` after the `/v1/sys/health` check. If this returns 403 (token invalid/expired), return `ErrProviderUnavailable` with a message indicating token expiry. Add a DoD checkbox: "`HealthCheck` returns `ErrProviderUnavailable` when the configured token is expired (`TestVaultTransit_HealthCheck_ExpiredToken` passes)."

### GAP-KMS3-4 — MEDIUM — Unit tests using `httptest.NewTLSServer` will fail TLS verification without explicit Vault client trust configuration

**Source:** Go `net/http/httptest` documentation — TLS test servers use self-signed certificates not trusted by default HTTP clients.

**Description:** Phase E1 states tests use `httptest.NewTLSServer` but provides no guidance on injecting the test server's self-signed certificate into the Vault `api.Client`. The `vault/api` package's default `http.Client` does not trust arbitrary test CAs, causing all HTTPS calls in unit tests to fail with `x509: certificate signed by unknown authority`.

**Required fix:** Add an explicit helper `newMockVaultClient(t, srv *httptest.Server) *api.Client` that calls `api.DefaultConfig()`, sets `cfg.Address = srv.URL`, and replaces `cfg.HttpClient` with `srv.Client()` (which trusts the test certificate). Document this in Phase E1. Alternatively, use `httptest.NewServer` (plain HTTP) for unit tests and reserve TLS for integration tests.
