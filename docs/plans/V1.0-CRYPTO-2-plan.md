## 1. Context & Current State

The multipart-upload state store (`internal/mpu/state.go`) persists
`UploadState` as plaintext JSON in Valkey. `Create` at `state.go:207`
marshals the struct and writes it via `HSETNX`. `Get` at `state.go:229`
reads the hash field and unmarshals it directly. `AppendPart` at
`state.go:265` writes `PartRecord` JSON in the same hash. `List` at
`state.go:302` iterates over all `mpu:*` keys and unmarshals their meta
fields. None of these paths perform any encryption.

The `WrappedDEK` field inside `UploadState` is already a ciphertext blob
produced by `KeyManager.WrapKey`, so an attacker with Valkey read access
cannot recover the object-encryption DEK without also compromising the
KMS/HSM or gateway password. However, the rest of the state — bucket
names (`Bucket`, line 43), object keys (`Key`, line 44), IV prefixes
(`IVPrefixHex`, line 51), part-level metadata (`PartRecord`, line 32),
and KMS provider identifiers (`KMSProvider`, line 56) — is visible to
anyone who can read Valkey memory dumps, RDB/AOF files, or replication
streams. TLS on the wire (`buildTLSConfig`, `state.go:162`) does not
close this at-rest disclosure gap.

ADR 0009 (`docs/adr/0009-encrypted-multipart-uploads.md:616`) explicitly
notes this under "Threat Model Expansion → Valkey read disclosure" and
calls it an accepted risk for v0.6. V1.0-CRYPTO-2 closes the gap by
encrypting the entire state blob at rest.

## 2. Design Goals & Non-Goals

### Goals

1. **Opaque at-rest blobs** — when enabled, the raw Valkey `HGET` value
   for any `UploadState` meta field must be indistinguishable from random
   bytes to an attacker without the derived AEAD key.
2. **Key isolation** — the Valkey encryption key must be derived from a
   dedicated password (or from the gateway main password with a distinct
   HKDF info string) so that a compromise of one key hierarchy does not
   automatically compromise the other.
3. **Zero-downtime migration** — existing deployments with unencrypted
   state must continue to work without a manual migration step. Old state
   reads transparently; new state writes are encrypted.
4. **Fail-closed configuration** — if `EncryptionPasswordEnv` is set but
   the referenced environment variable is empty, the gateway must refuse
   to start with a clear error.
5. **Zeroize on shutdown** — the derived AEAD key must be cleared from
   memory on `ValkeyStateStore.Close()` using the existing `zeroBytes()`
   convention.
6. **Versioned ciphertext format** — a one-byte version prefix leaves
   room for future AEAD algorithms; unknown versions fail closed with a
   clear error rather than silently mis-decrypting.
7. **Observability** — Prometheus counters track encrypted writes and
   legacy fallback reads, plus a one-time log warning when unencrypted
   state is detected.

### Non-Goals

1. **Re-encrypting existing state in-place** — not required; old state
   expires via TTL naturally and the fallback read path handles it.
2. **Changing the Valkey wire protocol or hash layout** — the hash
   key (`mpu:<hash>`) and field names (`meta`, `part:<N>`) remain
   unchanged; only the *value* of the `meta` field becomes an encrypted
   blob. (Part records are left plaintext; see §10 Follow-ups.)
3. **HSM-bound key storage for Valkey encryption** — out of scope; the
   key is password-derived. Deployments requiring HSM-grade isolation
   should use the external KMS path for the DEK itself, which is already
   supported.
4. **Backwards compatibility beyond one major version** — the
   `encrypt_state=false` toggle is a v1.0-only escape hatch, deprecated
   in v1.1 and removable in v2.0.

## 3. Target Interface / Semantics / Behaviour

### Versioned Ciphertext Format

```go
// Ciphertext layout for the at-rest blob:
//   version(1 byte) || nonce(12 bytes) || ciphertext(... ) || tag(16 bytes)
//
// Version 0x01 = AES-256-GCM.
// Future versions use the same overall framing with a different version byte.
const stateEncryptionVersionV1 byte = 0x01

const (
    stateEncryptionVersionLen = 1
    stateEncryptionNonceLen   = 12 // AES-GCM standard nonce
    stateEncryptionTagLen     = 16 // AES-GCM standard tag
)
```

### Key Derivation

```go
// deriveStateAEADKey derives a dedicated 32-byte AES-256 key from the
// configured password using HKDF-SHA256.
//
// Invariants:
//   - The same password + salt always produces the same key (deterministic).
//   - Using a distinct salt isolates this key from object-encryption keys.
//   - The derived key is cached in the ValkeyStateStore struct; HKDF is
//     NOT re-run on every read/write.
func deriveStateAEADKey(password string) []byte {
    // salt is a fixed, public string — HKDF security does not rely on salt
    // secrecy, only on salt uniqueness across different key derivations.
    salt := []byte("s3eg-mpu-state-v1")
    return hkdf.Extract(sha256.New, []byte(password), salt)
}
```

**Rationale:** HKDF-Extract is sufficient here because the input
(password) is already a high-entropy secret. A fixed salt is safe as
long as it is unique relative to other salts used in the same binary;
`"s3eg-mpu-state-v1"` is distinct from any salt used in the engine's
KDF or IV derivation.

### ValkeyStateStore Extension

```go
// EncryptState seals a plaintext JSON blob with AES-256-GCM.
// Returns a byte slice in the versioned ciphertext format.
// Nonce is crypto/rand 96 bits; probability of collision is negligible.
func (s *ValkeyStateStore) EncryptState(plaintext []byte) ([]byte, error)

// DecryptState opens a ciphertext blob.
// If decryption fails with an AEAD tag error, returns ErrStateDecryptFailed
// so the caller can attempt legacy plaintext fallback.
func (s *ValkeyStateStore) DecryptState(ciphertext []byte) ([]byte, error)
```

### Configuration

```yaml
multipart_state:
  valkey:
    # ... existing fields ...
    # Optional: dedicated password for at-rest encryption of multipart state.
    # If unset, falls back to the gateway's main encryption password
    # (ENCRYPTION_PASSWORD) with a distinct HKDF info string.
    encryption_password_env: "VALKEY_ENCRYPTION_PASSWORD"
    # Opt-out toggle for backwards compatibility. Default true in v1.0.
    encrypt_state: true
```

**Validation rules:**
- If `encrypt_state == true` and `encryption_password_env` is set but
  the env var is empty → fail closed at startup.
- If `encrypt_state == false` → log a deprecation warning at startup.

## 4. Work Breakdown

### Phase A — Encryption helper and store integration

- **A1.** Add `EncryptState` / `DecryptState` methods to
  `internal/mpu/state.go`.
  - Extend `ValkeyStateStore` struct with `stateKey []byte` and
    `encryptState bool` fields.
  - `EncryptState`: allocate `1+12+len(plaintext)+16` buffer, write
    version byte, random nonce, call `aesgcm.Seal` with `nil` additional
    data, return.
  - `DecryptState`: validate length ≥ `1+12+16`, read version byte
    (reject != `0x01`), split nonce/tag, call `aesgcm.Open`, return
    plaintext.
  - `Close`: call `zeroBytes(s.stateKey)` before closing the Redis
    client.

- **A2.** Wire encryption into `Create`, `Get`, `List` in
  `internal/mpu/state.go`.
  - `Create`: after `json.Marshal(state)`, if `s.encryptState` call
    `EncryptState` and store the ciphertext blob as the hash value.
  - `Get`: after `HGetAll`, if `s.encryptState` call `DecryptState` on
    the `meta` raw bytes; if decryption returns an AEAD error, attempt
    `json.Unmarshal` on the raw bytes as legacy plaintext. On success,
    log a one-time warning and increment the legacy-read metric.
  - `List`: same decrypt-then-fallback logic as `Get` for each meta
    field.

- **A3.** Update `NewValkeyStateStore` in `internal/mpu/state.go:115` to
  accept the derived key.
  - Change signature to accept an optional `stateKey []byte` (or derive
    it inside the constructor from the config). If `encryptState` is
    enabled and the key is nil/empty, return an error.
  - Cache the key in the struct; do not derive per-operation.

### Phase B — Configuration and validation

- **B1.** Extend `ValkeyConfig` in `internal/config/config.go:632`.
  - Add `EncryptionPasswordEnv string` with yaml tag
    `encryption_password_env` and env tag `VALKEY_ENCRYPTION_PASSWORD_ENV`.
  - Add `EncryptState *bool` with yaml tag `encrypt_state` and env tag
    `VALKEY_ENCRYPT_STATE`. Pointer type allows "unset → default true"
    semantics.

- **B2.** Add env-var loading in `internal/config/config.go`
  `loadFromEnv` (around line 830–1140).
  - Read `VALKEY_ENCRYPTION_PASSWORD_ENV` and `VALKEY_ENCRYPT_STATE`.

- **B3.** Add validation in `internal/config/config.go` `Validate()`
  (or create a `ValkeyConfig.Validate()` method).
  - If `EncryptState` is explicitly `false`, log a deprecation warning.
  - If `EncryptState` is true (or nil-default-true) and
    `EncryptionPasswordEnv` is non-empty but `os.Getenv` returns empty,
    return a descriptive error.

- **B4.** Update default config initialization in `LoadConfig`
  (`config.go:777`) so that `MultipartState.Valkey.EncryptState` defaults
    to `true`.

### Phase C — Metrics and observability

- **C1.** Add counters to `internal/metrics/metrics.go`.
  - `gateway_mpu_state_encrypted_writes_total` — incremented on every
    `Create` / `Get` / `List` path that successfully encrypts or
    decrypts a state blob.
  - `gateway_mpu_state_legacy_reads_total` — incremented on every
    fallback read of unencrypted legacy state.

- **C2.** Add helper methods on `*Metrics` for the new counters,
    following the pattern of `RecordMPUStateStoreOp`.

- **C3.** Instrument `internal/mpu/state.go` to call the new metrics
    helpers on encrypt/decrypt success and on legacy fallback.

- **C4.** Add a one-time `sync.Once` warning in `ValkeyStateStore`:
    on first legacy fallback read, log:
    `"Unencrypted Valkey state detected — enable valkey.encrypt_state=true"`.

### Phase D — Documentation updates

- **D1.** Update ADR 0009
  (`docs/adr/0009-encrypted-multipart-uploads.md:616`) "Security
  Considerations" section to state that at-rest encryption is now
  available and recommended in v1.0.

- **D2.** Create `docs/RUNBOOK.md` section "Valkey state at-rest
  encryption" covering:
  - How to set `VALKEY_ENCRYPTION_PASSWORD`.
  - Migration behaviour (lazy, no downtime).
  - How to verify encryption is active (scrape `gateway_mpu_state_encrypted_writes_total`).

- **D3.** Update deployment docs (e.g., `docs/DEPLOYMENT.md` if it
  exists, or `README.md`) to mention `VALKEY_ENCRYPTION_PASSWORD` in
  the secret-management checklist.

| Layer | Tests |
|---|---|
| Unit — round-trip | `TestStateStore_EncryptDecrypt_RoundTrip` — create state with `encryptState=true`, get it back, assert fields match. |
| Unit — non-determinism | `TestStateStore_Encrypt_NonceUnique` — encrypt same state twice, assert ciphertexts differ (new nonce per write). |
| Unit — tamper detection | `TestStateStore_Decrypt_TamperedCiphertext` — flip a byte in the ciphertext, assert `DecryptState` returns error. |
| Unit — legacy fallback | `TestStateStore_Get_LegacyPlaintextFallback` — write raw JSON via miniredis directly, then `Get` with encryption enabled; assert success and legacy metric increments. |
| Unit — HKDF isolation | `TestDeriveStateAEADKey_DifferentPasswords` — same literal password via `ENCRYPTION_PASSWORD` vs `VALKEY_ENCRYPTION_PASSWORD` must produce different keys because of distinct salts. |

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Adding encryption adds latency to sub-millisecond Valkey ops | AES-256-GCM on small JSON blobs (~few KiB) is negligible overhead; benchmark in CI to ensure p99 < 1 ms. |
| Legacy fallback path accidentally decrypts attacker-crafted plaintext as ciphertext | The fallback is only attempted when `DecryptState` returns an AEAD tag error, not on generic errors; an attacker who writes raw JSON cannot forge a valid AEAD tag without the key. |
| Operator forgets to back up `VALKEY_ENCRYPTION_PASSWORD` | Document in RUNBOOK that password loss = inability to read in-flight MPU state; DEK itself is safe in wrapped form, but upload metadata becomes unreadable. |
| `EncryptState` enabled mid-deployment causes mixed encrypted/unencrypted state | Lazy migration design handles this automatically; no operator action required. |
| `stateKey` cached in struct is vulnerable to memory-dump attacks | Same threat model as the DEK itself; mitigation is `zeroBytes()` on `Close()` and process-level memory protections (already standard). |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **Encrypt part records (`part:<N>` hash fields).** The issue's scope
   focuses on the `meta` field because it contains the most sensitive
   data (bucket, key, wrapped DEK envelope). Part records contain
   `enc_len`, `plain_len`, and `etag`, which are less sensitive.
   Encrypting them would require re-writing all hash fields on every
   `AppendPart` (no per-field encryption in the current design).
   Consider for v1.1 if threat model demands it.

2. **Key rotation for the Valkey encryption key.** Out of scope; the
   password can be rotated by restarting gateways with a new password
   and letting old state expire via TTL. No automated re-encryption
   of existing state is planned.

3. **HSM-backed Valkey encryption key.** Deferred to a future issue;
   would require a `KeyManager`-like interface for the state store.
