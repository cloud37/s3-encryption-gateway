# V1.0-CLI-2 ‚Äî Gateway-Aware Audit Tooling (`s3eg-cli`) + Remove Gateway-Bypassing Migration ‚Äî Implementation Plan

Status: Draft
Owner: DevEx
Priority: P3
Labels: `milestone:v1.0`, `area:ecosystem`, `area:ops`

> Scope clarified relative to the tracker entry at
> `docs/issues/v1.0-issues.md:275`. The tracker entry was rewritten during
> this planning session: the original "extend the migrate tool and rename it,
> retaining all migration functionality" was rejected. This plan instead
> **removes the gateway-bypassing migration write-path entirely** and turns the
> binary into a **pure read-only audit tool** (`s3eg-cli`) with three
> sub-commands (`inspect`, `verify-key`, `list-algorithm`). The supported
> re-encryption path becomes **GET-through-gateway ‚Üí PUT-through-gateway** using

> no-AAD recovery gap. This expands the stub tracker entry into a full

## 1. Context & Current State

### 1.1 The binary today is a gateway-bypassing migration tool

`cmd/migrate/main.go` builds the binary `s3eg-migrate`. Its `main` dispatches
on `os.Args` (`cmd/migrate/main.go:20-34`): the token `backfill-legacy-no-aad`
routes to `runBackfill()` (`:117`), everything else routes to `runMigrate()`
(`:36`).

`mustBuildDeps` (`cmd/migrate/main.go:178`) is the heart of the bypass: it
loads `gateway.yaml` via `config.LoadConfig` (`:179`), constructs an S3 client
pointed **directly at the backend store** via `s3.NewClient(&cfg.Backend)`
(`:185`), and rebuilds the full crypto stack locally via `buildEngines`
(`:191`, defined `:221`). `buildEngines` calls
`crypto.NewEngineWithChunking(...)` (`:238`) using the operator's password ‚Äî
i.e. a second, independent copy of the gateway's decryption machinery.

The actual rewrite happens in `internal/migrate`:
- `Migrator.Migrate` (`internal/migrate/migrator.go:75`) lists objects, spins
  up a worker pool (`:151-165`), classifies each via `ClassifyObject`
  (`internal/migrate/classifier.go:71`), and dispatches `migrateJob`s.
- `migrateObject` (`internal/migrate/worker.go:15`) performs the
  decrypt‚Üíre-encrypt‚Üí**rewrite** cycle: `GetObject` (`:23`),
  `SourceEngine.Decrypt` (`:30`), `TargetEngine.Encrypt` (`:37`),
  buffer-to-temp-file (`:44`), `PutObject` atomic overwrite (`:56`), optional
  `verifyObject` (`:62`), and best-effort fallback-companion `DeleteObject`
  (`:70`).
- `Migrator.BackfillLegacyNoAAD` (`internal/migrate/migrator.go:224`) is a
  metadata-only `CopyObject`-to-self that stamps `MetaLegacyNoAAD="true"`
  (`:292-294`).

This duplicates the gateway's crypto path and must track its evolution
forever ‚Äî the same liability that rejected V1.0-CLI-1 (per memory: "adding a
bespoke binary for generic operations adds maintenance burden without unique
value").

### 1.2 The gateway already exposes legacy objects

The gateway's `Decrypt` path reads every legacy class through fallbacks. In
particular `internal/crypto/engine.go:802-824` first tries AAD (current and
legacy pipe-delimited formats), then ‚Äî only when
`expandedMetadata[MetaLegacyNoAAD] == "true"` (`:818`) ‚Äî attempts a no-AAD
open. Because the gateway can already decrypt these objects, re-encryption
needs no bespoke binary: a `GET` through the gateway yields plaintext and a
`PUT` re-encrypts with the current parameters. Any S3 tool performs this.

### 1.3 The unique value that remains: backend-envelope inspection

What no S3 tool can do is read the **on-backend ciphertext envelope** the
gateway hides. The encryption parameters are stored as object metadata headers
defined in `internal/crypto/engine.go:35-71`:

| Constant | Header | Meaning |
|---|---|---|
| `MetaEncrypted` | `x-amz-meta-encrypted` | `"true"` if encrypted |
| `MetaAlgorithm` | `x-amz-meta-encryption-algorithm` | `AES256-GCM` / `ChaCha20-Poly1305` (`internal/crypto/algorithms.go:12,14`) |
| `MetaKeySalt` | `x-amz-meta-encryption-key-salt` | per-object salt (32 B, `engine.go:30`) |
| `MetaIV` | `x-amz-meta-encryption-iv` | nonce (12 B, `engine.go:31`) |
| `MetaWrappedKeyCiphertext` | `x-amz-meta-encryption-wrapped-key` | KMS-wrapped DEK |
| `MetaKMSKeyID` / `MetaKMSProvider` | `‚Ä¶-kms-id` / `‚Ä¶-kms-provider` | wrapping key identity |
| `MetaKeyVersion` | `x-amz-meta-encryption-key-version` | wrapping key version (`keymanager.go:84`) |
| `MetaLegacyNoAAD` | `x-amz-meta-enc-legacy-no-aad` | AAD scheme marker |
| `MetaKDFParams` | `x-amz-meta-encryption-kdf-params` | KDF algorithm + params |
| `MetaChunkedFormat`/`MetaFallbackMode`/`MetaIVDerivation` | (see `chunked.go:29`, `engine.go:60`, `chunked.go:33`) | format flags used by `ClassifyObject` |

`ClassifyObject` (`internal/migrate/classifier.go:71`) already maps these
headers to a human-readable migration class ‚Äî directly reusable by the audit
commands.

### 1.4 The no-AAD recovery gap

Removing `BackfillLegacyNoAAD` exposes a gap: objects encrypted before the
`MetaLegacyNoAAD` marker existed (and never backfilled) **fail to decrypt
through the gateway**, because the no-AAD fallback at
`internal/crypto/engine.go:818` is gated on the marker being `"true"`. Such
objects cannot be recovered by the GET‚ÜíPUT pattern. This plan closes the gap
with a fail-closed gateway config flag (¬ß3.4) rather than retaining the
bypass tool.

### 1.5 Blast radius (acknowledged, in scope)

`s3eg-migrate` is referenced by completed/verified issues and their runbooks:
V1.0-MAINT-1 (`docs/plans/V1.0-MAINT-1-plan.md`), V1.0-MGMT-2 (`--filter=kdf`,
`docs/issues/v0.8-issues.md:7`), V1.0-SEC-H03 (KDF re-seal,
`docs/plans/V1.0-SEC-H03-plan.md:561`), SEC-2/SEC-4 recovery
(`docs/issues/v0.7-issues.md:208`), and `docs/MIGRATION.md` in its entirety.
All must be rewritten to the GET‚ÜíPUT pattern (¬ß4 Phase F). The objective is to
eliminate a structurally unsafe (gateway-bypassing) maintenance liability while
preserving the genuinely-unique audit capability and a recovery story for every
object class.

## 2. Design Goals & Non-Goals

### Goals

1. **Read-only by construction** ‚Äî the `s3eg-cli` binary issues only
   `ListObjects`, `HeadObject`, and bounded ranged `GetObject` calls. No
   `PutObject`, `CopyObject`, or `DeleteObject` anywhere in the binary or its
   support package. A reviewer can `grep` the package and confirm zero write
   calls.
2. **Migration write-path fully removed** ‚Äî `Migrate`, `BackfillLegacyNoAAD`,
   `migrateObject`, `verifyObject`, the worker re-encrypt pool, and the
   state/dry-run/verify machinery are deleted. `ClassifyObject` and the
   `ObjectClass` taxonomy are retained (reused by audit commands).
3. **Three audit sub-commands** ‚Äî `inspect`, `verify-key`, `list-algorithm`,
   each with `--config`, `--log-level`, `--output text|json`.
4. **Binary renamed** ‚Äî `cmd/migrate` ‚Üí `cmd/cli`, output binary `s3eg-cli`;
   `s3eg-migrate` retained as a compatibility shim that prints a deprecation
   notice and points at `s3eg-cli` + the GET‚ÜíPUT pattern.
5. **No-AAD recovery gap closed** ‚Äî a new gateway config flag
   `encryption.allow_unmarked_no_aad_fallback` (default `false`, fail-closed)
   lets operators temporarily permit no-AAD decryption of unmarked objects so
   GET‚ÜíPUT can recover them. SEC-4 default behaviour is unchanged when the flag
   is `false`.
6. **Docs realigned** ‚Äî `docs/MIGRATION.md` rewritten around GET‚ÜíPUT;
   `docs/S3_CLI_TOOLS.md`, `README.md`, `CHANGELOG.md` updated; all stale
   `s3eg-migrate` runbooks corrected.
7. **Tested** ‚Äî unit tests (`-race`) for envelope parsing and each sub-command
   over a mock backend; MinIO integration tests behind `//go:build integration`.

### Non-Goals

1. **Generic put/get/presign** ‚Äî out of scope; covered by `awscli`/`s5cmd`/`mc`
   and by `docs/S3_CLI_TOOLS.md` (same rationale as the V1.0-CLI-1 rejection).
2. **External KMS unwrap (AWS/Vault) in `verify-key`** ‚Äî `verify-key` validates
   key-version metadata and attempts unwrap via the engine path available from
   `gateway.yaml`; full AWS/Vault unwrap is deferred until KMS adapters
   (V1.0-KMS-2 / V1.0-KMS-3) land. Out of scope here.
3. **In-place re-encryption / metadata-preserving migration** ‚Äî deliberately
   removed; GET‚ÜíPUT is the supported path. Edge cases that GET‚ÜíPUT cannot
   express (exact ETag preservation) are documented as accepted trade-offs.
4. **New crypto formats or algorithms** ‚Äî none introduced.

## 3. Target Interface / Semantics / Behaviour

### 3.1 CLI surface (`s3eg-cli`)

```
s3eg-cli inspect       <bucket> <key>            [--config F] [--log-level L] [--output text|json]
s3eg-cli verify-key    <bucket> <key>            [--key-version N] [--config F] [--log-level L] [--output text|json]
s3eg-cli list-algorithm <bucket>                 [--prefix P] [--workers N] [--config F] [--log-level L] [--output text|json]
```

Dispatch mirrors the existing pattern (`cmd/migrate/main.go:20-34`): inspect
`os.Args[1]` for the sub-command token, build a dedicated `flag.FlagSet` per
sub-command, and require the positional `<bucket>`/`<key>` arguments after flag
parsing. Unknown or missing sub-command prints usage and exits non-zero.
Rationale: keeps the zero-dependency stdlib `flag` style already used; no new
framework (Cobra) is introduced for three read-only commands.

### 3.2 Read-only backend access contract

A reduced client interface replaces `migrate.S3Client`
(`internal/migrate/migrator.go:17`). Only read methods are permitted:

```go
// AuditClient is the read-only backend access surface for s3eg-cli.
// Invariants:
//   - Implementations MUST NOT mutate backend state.
//   - ctx is honoured for cancellation on every call.
type AuditClient interface {
    HeadObject(ctx context.Context, bucket, key string, versionID *string) (map[string]string, error)
    GetObject(ctx context.Context, bucket, key string, versionID *string, rangeHeader *string) (io.ReadCloser, map[string]string, error)
    ListObjects(ctx context.Context, bucket, prefix string, opts s3.ListOptions) (s3.ListResult, error)
}
```

`s3.NewClient(&cfg.Backend)` (existing, `internal/s3/client.go:362`) satisfies
this. Rationale: small-interface guidance ‚Äî the audit tool advertises, in its
type, that it cannot write.

### 3.3 `inspect` output contract

`inspect` issues `HeadObject` for metadata and a **bounded** ranged
`GetObject` (`Range: bytes=0-N`, N‚âà64) for the ciphertext fingerprint. It never
reads the full body. Decoded fields:

```go
type EnvelopeReport struct {
    Bucket            string            `json:"bucket"`
    Key               string            `json:"key"`
    Encrypted         bool              `json:"encrypted"`
    Algorithm         string            `json:"algorithm,omitempty"`           // MetaAlgorithm
    Class             string            `json:"class"`                         // ClassToString(ClassifyObject(meta))
    AADScheme         string            `json:"aad_scheme"`                    // "v2-aad" | "v1-no-aad"
    KMSProvider       string            `json:"kms_provider,omitempty"`        // MetaKMSProvider
    KMSKeyID          string            `json:"kms_key_id,omitempty"`          // MetaKMSKeyID
    KeyVersion        string            `json:"key_version,omitempty"`         // MetaKeyVersion
    KDFParams         string            `json:"kdf_params,omitempty"`          // MetaKDFParams
    Chunked           bool              `json:"chunked"`                       // MetaChunkedFormat
    Fallback          bool              `json:"fallback"`                      // MetaFallbackMode
    SaltHex           string            `json:"salt_hex,omitempty"`            // decoded MetaKeySalt
    IVHex             string            `json:"iv_hex,omitempty"`              // decoded MetaIV, with length note
    CiphertextHeadHex string            `json:"ciphertext_head_hex,omitempty"` // first ‚â§32 bytes
    EncryptionHeaders map[string]string `json:"encryption_headers"`            // all x-amz-meta-encryption-*
}
```

`AADScheme` is derived: `"v1-no-aad"` iff `meta[MetaLegacyNoAAD]=="true"`,
else `"v2-aad"`. Text output is a key-aligned table; JSON output is
`json.MarshalIndent` of the struct. Error table:

| Condition | Exit | Message |
|---|---|---|
| object not found | 3 | `inspect: <bucket>/<key>: not found` |
| not encrypted | 0 | report with `encrypted=false` (informational) |
| metadata salt/IV un-decodable | 0 | field omitted + warn log; not fatal |

### 3.4 Gateway config flag ‚Äî no-AAD recovery

New field in `EncryptionConfig` (`internal/config/config.go:286`):

```go
// AllowUnmarkedNoAADFallback, when true, permits the gateway to attempt the
// no-AAD decryption fallback for objects that LACK the x-amz-meta-enc-legacy-no-aad
// marker. Default false (fail-closed): preserves the SEC-4 property that an
// attacker with backend write access cannot strip AAD by deleting the marker.
// Operators enable this only during a controlled re-encryption recovery window
// (GET-through-gateway -> PUT-through-gateway).
AllowUnmarkedNoAADFallback bool `yaml:"allow_unmarked_no_aad_fallback" env:"ENCRYPTION_ALLOW_UNMARKED_NO_AAD_FALLBACK"`
```

Engine wiring: add an unexported field `allowUnmarkedNoAAD bool` to `engine`
(`internal/crypto/engine.go:111`), set via a new option
`WithAllowUnmarkedNoAADFallback(bool)` in `engine_options.go`, and relax the
gate at `engine.go:818`:

```go
if expandedMetadata[MetaLegacyNoAAD] == "true" || e.allowUnmarkedNoAAD {
    if pt, err2 := gcm.Open(nil, iv, ciphertext, nil); err2 == nil {
        plaintext = pt
        openErr = nil
    }
}
```

Rationale: fail-closed default keeps the security invariant; the explicit
opt-in is auditable and time-boxed. `cmd/server/main.go` passes the flag into
the engine builder; the migrate-tool path is removed so only the gateway server
honours it.

### 3.5 `verify-key` semantics

Loads `HeadObject` metadata; reports the recorded `MetaKeyVersion`. If
`--key-version N` is supplied, compares against the recorded version and, when
an unwrap path is available from `gateway.yaml` (password/self-contained
engine), attempts a DEK unwrap to confirm; otherwise reports a metadata-only
match with a `verified: "metadata-only"` qualifier in JSON. No full body
decryption. Exit 0 on match, 4 on mismatch, 3 on not-found.

### 3.6 `list-algorithm` semantics

Paginated `ListObjects` + per-object `HeadObject` over a worker pool (default
`--workers 4`, reusing the channel-fan-out shape from
`internal/migrate/migrator.go:143-165` but **without** any write). Aggregates a
`map[algorithm]count` plus a `(plaintext)` bucket and a per-class breakdown.
Output is sorted descending by count with percentages. JSON output:

```json
{"bucket":"b","prefix":"","total":12600,
 "by_algorithm":{"AES256-GCM":12400,"ChaCha20-Poly1305":198,"(plaintext)":2},
 "by_class":{"modern":12598,"class_b_no_aad":2}}
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `allow_unmarked_no_aad_fallback` becomes a permanent AAD-bypass, weakening SEC-4 | Default `false`; documented as a time-boxed recovery window; regression test `TestDecrypt_UnmarkedNoAAD_FlagOff_FailsClosed`; promoted to DoD ¬ß8. |
| Removing the migration write-path strands objects the gateway cannot decrypt | The new flag closes the only known gap (pre-marker no-AAD); `inspect`/`list-algorithm` let operators enumerate affected objects before the window. |
| Existing release automation / operators expect `s3eg-migrate` binary | Keep the binary name as a deprecation shim that exits non-zero with guidance; CI still publishes it; transition documented in CHANGELOG. |
| Deleting `internal/migrate` breaks importers elsewhere | Two importers exist: `cmd/migrate/main.go:16` and `test/conformance/migration_test.go:23`. Move `ClassifyObject` to `internal/audit`, update both imports, and delete/port the conformance migration test (it exercises the removed write-path; convert to an `inspect`/`list-algorithm` audit conformance test or delete with rationale). |
| `inspect` ranged GET reads more than intended | Use an explicit `Range: bytes=0-63` header; cap the fingerprint at 32 decoded bytes; never call `io.ReadAll` on the full body. |
| Stale runbooks in verified issues mislead operators | Phase F adds forward-pointer notes; `docs/MIGRATION.md` fully rewritten; DoD requires all `s3eg-migrate` migration references corrected. |

## 12. O'Reilly Verification Gaps (2026-06-08)

> Added by plan-verifier after cross-referencing with O'Reilly authoritative sources.
> These gaps must be resolved before implementation begins.

### GAP-CLI2-1 ‚Äî HIGH ‚Äî Gateway password not specified to be stored as `[]byte` in `cmd/cli` ‚Äî string zeroization impossible

**Source:** *Serious Cryptography, 2nd Ed.* (Aumasson, No Starch Press, 2024), Ch. 3 ‚Äî residual key material in memory is a tangible threat.

**Description:** Phase C2 rewrites `cmd/cli/main.go` but does not specify how the gateway password from `cfg.Encryption.Password` is handled. If it is stored as a `string` at any point in `cmd/cli`, it cannot be zeroed (Go strings are immutable). The existing `cmd/migrate/main.go` already handles passwords as `[]byte`, establishing a correct precedent. Without an explicit specification, the implementor may introduce password leakage in `cmd/cli`.

**Required fix:** Add to Phase C2 specification: "The gateway password from `cfg.Encryption.Password` must be stored as `[]byte` throughout `cmd/cli/main.go` and never assigned to a `string`. After engine construction, zero the `[]byte` slice using `for i := range pwd { pwd[i] = 0 }`. If string-to-byte conversion is unavoidable, document that an `unsafe.Pointer` cast is required to zero the backing array."

### GAP-CLI2-3 ‚Äî LOW ‚Äî `AuditClient.GetObject` with `rangeHeader` parameter is not satisfied by `s3.NewClient` directly

**Source:** Go interface satisfaction ‚Äî `s3.Client.GetObject` has a different signature than `AuditClient.GetObject`.

**Description:** ¬ß3.2 specifies `AuditClient.GetObject(ctx, bucket, key, versionID *string, rangeHeader *string)` but ¬ß3.2 also claims "`s3.NewClient(&cfg.Backend)` satisfies this." The existing `s3.Client.GetObject` does not accept a `rangeHeader` parameter ‚Äî the two interfaces have different signatures. `s3.NewClient` does NOT satisfy `AuditClient` directly.

**Required fix:** Add to Phase A5/B1: "Create an `auditClientAdapter` struct in `internal/audit/client.go` that wraps `s3.Client` and implements `AuditClient`. It injects the Range header on `GetObject` when `rangeHeader` is non-nil. Correct ¬ß3.2's claim that `s3.NewClient` 'satisfies this' to 'satisfies this via `auditClientAdapter`.' Add `TestAuditClientAdapter_GetObjectWithRange_SetsHeader`."
