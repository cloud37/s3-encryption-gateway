## 1. Context & Current State

### Existing KeyManager Infrastructure

The `KeyManager` interface is defined in `internal/crypto/keymanager.go:24`.
It declares six methods: `Provider()`, `WrapKey()`, `UnwrapKey()`,
`ActiveKeyVersion()`, `HealthCheck()`, and `Close()`, each accepting
`context.Context`. The `KeyEnvelope` struct (`internal/crypto/keymanager.go:72`)
carries `KeyID`, `KeyVersion`, `Provider`, `Ciphertext []byte`, and
`CreatedAt time.Time`. Invariants require thread-safety, context propagation,
caller-owned DEK zeroization, no logging of plaintext, and idempotent `Close`.

The optional `RotatableKeyManager` extension
(`internal/crypto/keymanager.go:91`) adds `PrepareRotation` and
`PromoteActiveVersion`, used by the admin API rotation state machine.

Two production adapters ship today: `cosmian`
(`internal/crypto/keymanager_cosmian.go`) using KMIP protocol over TLS, and
`memory` (`internal/crypto/keymanager_memory.go`) using RFC 3394 AES key-wrap.
A `self_contained` adapter (V1.0-KMS-4) provides local AES-256-GCM and
RSA-OAEP wrapping. None of these call AWS KMS.

### Registry

`internal/crypto/keymanager_registry.go:69` registers `memory` and `hsm` in
`init()`. The `self_contained` adapter is registered from its own factory file.
The `cosmian` adapter is wired in `internal/api/crypto_factory.go`. A new
`aws` (and alias `aws-kms`) factory must follow the same `init()` pattern.

### Configuration

`internal/config/config.go:303` defines `KeyManagerConfig` with fields for
`Cosmian`, `Memory`, `SelfContained`, and commented-out stubs at line 311:
```
// AWS        AWSKMSConfig  `yaml:"aws"`
// Vault      VaultConfig   `yaml:"vault"`
```
The `KeyManagerConfig` provider comment at line 299 already lists
`"aws" or "aws-kms"` as a planned provider. `loadFromEnv` in the same file
handles explicit env-var overrides for each provider sub-struct; AWS KMS vars
must be wired there.

### AWS SDK v2 — Dependency Status

`go.mod` already imports `github.com/aws/aws-sdk-go-v2 v1.41.7`,
`github.com/aws/aws-sdk-go-v2/config v1.32.18`, and
`github.com/aws/aws-sdk-go-v2/credentials v1.19.17` for the S3 backend.
However, `github.com/aws/aws-sdk-go-v2/service/kms` is **not yet present** in
`go.mod` and must be added. All AWS credential resolution (IMDS, env vars,
shared credentials file, assumed roles) is already tested in the existing S3
backend; the KMS adapter can reuse the same `aws/config.LoadDefaultConfig`
entry point.

### Why the Existing State Is Insufficient

There is currently no way for the gateway to delegate DEK wrapping/unwrapping to
AWS KMS. Operators running on AWS must use either the password-based path
(PBKDF2/Argon2id — no external key custody) or the Cosmian KMIP adapter (a
separate server). Neither satisfies the AWS-native threat model where KMS key
policies, CloudTrail audit records, and IAM role bindings are the standard
control plane for key access. The issue (V1.0-KMS-2) targets this gap by
implementing a first-class AWS KMS adapter that is envelope-encryption aware:
the DEK is generated locally, then wrapped by AWS KMS (`kms:Encrypt`), and
unwrapped at read time (`kms:Decrypt`). AWS KMS never sees plaintext object
data.

## 2. Design Goals & Non-Goals

### Goals

1. **Interface compliance** — `awsKMSManager` satisfies `KeyManager` at
   compile time via `var _ KeyManager = (*awsKMSManager)(nil)`. The
   `ConformanceSuite` passes for all seven sub-tests.

2. **Envelope encryption via AWS KMS** — `WrapKey` calls `kms:Encrypt` to
   wrap the locally-generated DEK; `UnwrapKey` calls `kms:Decrypt`. AWS KMS
   never receives plaintext object content.

3. **Key alias and ARN support** — The adapter accepts key identifiers in all
   AWS-supported forms: key IDs (`mrk-...`), aliases (`alias/my-key`), and
   full ARNs. Identifier format is validated at construction time but not
   normalised (AWS KMS accepts all forms natively).

4. **Dual-read window for rotation** — The adapter stores a list of key
   references ordered by version. `UnwrapKey` first tries the envelope's
   declared `KeyVersion`; on `kms:Decrypt` failure it iterates over the
   remaining versions within the configured `dual_read_window`. This mirrors
   the `CosmianKMIPManager` multi-key fallback pattern
   (`internal/crypto/keymanager_cosmian.go:191`).

5. **Standard AWS credential chain** — Configuration exposes `region`,
   `endpoint` (for LocalStack / moto testing), and an optional
   `credentials_provider` enum (`"default"`, `"env"`, `"file"`, `"static"`).
   When `"default"` is set (or omitted), `aws/config.LoadDefaultConfig` is
   invoked, which honours environment variables, IMDS, and `~/.aws/credentials`
   automatically.

6. **Health check** — `HealthCheck` issues a `kms:DescribeKey` on the active
   key ID with a short deadline (default 5 s). A failed call surfaces
   `ErrProviderUnavailable` wrapped around the AWS error. This integrates with
   the existing `/healthz` readiness endpoint.

7. **Security hygiene** — Key ARNs are never logged at INFO or DEBUG level.
   Plaintext DEKs are zeroized immediately after the `kms:Encrypt` call
   returns. `ErrUnwrapFailed` is returned without embedding the raw AWS error
   message (which may contain key metadata) in user-visible responses.

8. **Registry integration** — The adapter registers under the names `"aws"`
   and `"aws-kms"` in `init()` via the standard `crypto.Register` call.

### Non-Goals

1. **KMS data-key generation (`kms:GenerateDataKey`)** — AWS KMS can generate
   DEKs directly; this plan intentionally does not use that path. The gateway
   always generates DEKs locally (via `crypto/rand`) and uses KMS solely for
   wrapping, matching the Cosmian and self_contained adapter model. Changing
   this would require engine changes; tracked separately.

2. **AWS Encryption SDK format** — The adapter uses the gateway's own
   `KeyEnvelope` wire format, not the AWS Encryption SDK message format. This
   is intentional: the gateway is the format owner.

3. **Multi-Region Key (MRK) automatic failover** — MRKs can be replicated
   across regions; the adapter does not automatically re-route to a replica on
   primary region failure. Operators must set endpoint explicitly for
   cross-region scenarios. Tracked as a follow-up.

4. **Custom key stores (XKS / CloudHSM)** — Calls are issued via the standard
   KMS endpoint; custom key store routing is out of scope for this plan.
   Transparent at the API level if the key ARN points to a custom key store.

5. **Vault Transit adapter (V1.0-KMS-3)** — Covered by a separate issue.

## 3. Target Interface / Semantics / Behaviour

### 3.1 `awsKMSManager`

```go
// awsKMSManager wraps/unwraps DEKs using AWS KMS kms:Encrypt / kms:Decrypt.
//
// Invariants:
//   - Thread-safe: all exported methods acquire m.mu (RLock for reads,
//     Lock for Close).
//   - Context propagation: ctx is passed directly to all AWS SDK calls;
//     cancellation and deadline propagate through the SDK's HTTP client.
//   - Zeroization: plaintext returned from UnwrapKey is owned by the caller.
//     Internal copies of the unwrapped DEK are zeroized with zeroBytes()
//     before the function returns.
//   - Idempotency: Close is idempotent; second call is a no-op (nil error).
//   - Nil envelope: UnwrapKey returns ErrInvalidEnvelope for nil or
//     zero-length Ciphertext envelopes.
//   - Post-close: WrapKey/UnwrapKey/ActiveKeyVersion/HealthCheck return
//     ErrProviderUnavailable after Close.
//   - No ARN logging: key ARNs and IDs are never logged at INFO or DEBUG.
//     Only the key version (integer) appears in structured log fields.
type awsKMSManager struct {
    mu           sync.RWMutex
    client       KMSClient             // interface, swappable in tests
    keys         []awsKeyRef           // ordered by version; index 0 is active
    dualReadWin  int                   // how many additional versions to try
    providerName string
    closed       bool
}

// awsKeyRef associates an AWS KMS key identifier with a gateway version integer.
type awsKeyRef struct {
    KeyID   string // alias, ARN, or key ID — accepted as-is by AWS KMS
    Version int
}
```

**`WrapKey(ctx, plaintext, metadata)`**

1. Check `ctx.Err()`; return wrapped error on cancellation.
2. `m.mu.RLock()` — check `closed → ErrProviderUnavailable`.
3. Retrieve `m.keys[0]` (active key reference).
4. Build `kms.EncryptInput{KeyId: &activeRef.KeyID, Plaintext: plaintext}`.
   Optionally set `EncryptionContext` from `metadata` map (non-nil keys only;
   see §3.3 for context key filtering).
5. Call `m.client.Encrypt(ctx, input)`.
6. On AWS error: return `fmt.Errorf("keymanager/aws: wrap: %w", err)`.
7. Return `&KeyEnvelope{KeyID: activeRef.KeyID, KeyVersion: activeRef.Version, Provider: m.providerName, Ciphertext: result.CiphertextBlob, CreatedAt: time.Now()}`.

**`UnwrapKey(ctx, envelope, metadata)`**

1. Check `ctx.Err()`.
2. Validate envelope not nil and `len(Ciphertext) > 0` (→ `ErrInvalidEnvelope`).
3. `m.mu.RLock()` — check `closed → ErrProviderUnavailable`.
4. Build `candidateVersions`: envelope's `KeyVersion` first, then up to
   `m.dualReadWin` other versions from `m.keys` in order.
5. For each candidate version, find the matching `awsKeyRef` from `m.keys`.
   If the version is absent: continue.
6. Build `kms.DecryptInput{CiphertextBlob: envelope.Ciphertext, KeyId: &ref.KeyID}`.
   Set `EncryptionContext` from `metadata` (same filtering as WrapKey).
7. Call `m.client.Decrypt(ctx, input)`.
8. On success: return `result.Plaintext` (caller owns; caller zeroizes).
9. On error: record `lastErr`; continue to next candidate.
10. On exhaustion: `return nil, fmt.Errorf("%w: all candidate versions failed", ErrUnwrapFailed)`.

**`ActiveKeyVersion(ctx)`** — RLock; return `m.keys[0].Version`.

**`HealthCheck(ctx)`**
1. RLock; check `closed`.
2. Build `kms.DescribeKeyInput{KeyId: &m.keys[0].KeyID}`.
3. Apply 5-second deadline from ctx if the caller's deadline is longer.
4. Call `m.client.DescribeKey(ctx, input)`.
5. On error: `return fmt.Errorf("%w: AWS KMS health check failed: %s", ErrProviderUnavailable, sanitiseAWSErr(err))`.
   `sanitiseAWSErr` strips key ARN/ID substrings from the error string for
   safe logging.

**`Close(ctx)`** — Lock; set `closed = true`; call `m.client = nil` (releases
the AWS SDK HTTP transport); return nil.

### 3.2 `KMSClient` Interface (for test injection)

```go
// KMSClient is the minimal AWS KMS API surface used by awsKMSManager.
// It is implemented by *kms.Client and by mockKMSClient in tests.
type KMSClient interface {
    Encrypt(ctx context.Context, params *kms.EncryptInput, optFns ...func(*kms.Options)) (*kms.EncryptOutput, error)
    Decrypt(ctx context.Context, params *kms.DecryptInput, optFns ...func(*kms.Options)) (*kms.DecryptOutput, error)
    DescribeKey(ctx context.Context, params *kms.DescribeKeyInput, optFns ...func(*kms.Options)) (*kms.DescribeKeyOutput, error)
}
```

**Rationale:** Defining a narrow interface over the concrete `*kms.Client`
follows the Go small-interface principle and allows unit tests to inject a
`mockKMSClient` without starting a real AWS endpoint. The interface is
unexported (lives only in `keymanager_aws.go`) to avoid leaking it into the
package API.

### 3.3 Encryption Context Handling

AWS KMS supports an `EncryptionContext` (`map[string]string`) that is bound to
the ciphertext and must be reproduced exactly at decrypt time. The adapter
passes a filtered subset of the `metadata` map as encryption context:

- Only keys with the prefix `"x-amz-meta-kms-ctx-"` are included (after
  stripping the prefix).
- The key version integer is always included as
  `"s3gw-key-version": strconv.Itoa(version)`.
- Keys with empty values are excluded.

This allows callers to bind ciphertext to specific object paths for
additional security, while keeping the context small and predictable.

**Rationale:** Requiring an exact context match prevents a wrapped DEK from
being replayed against a different object. The prefix filter avoids
accidentally forwarding all S3 headers as KMS context.

### 3.4 Go Config Struct

```go
// AWSKMSConfig captures settings for the AWS KMS adapter.
type AWSKMSConfig struct {
    // Region is the AWS region for the KMS endpoint (e.g. "us-east-1").
    // Required unless the default credential chain resolves a region.
    Region string `yaml:"region" env:"AWS_KMS_REGION"`

    // Endpoint overrides the KMS endpoint URL (e.g. for LocalStack or moto).
    // Leave empty to use the standard AWS regional endpoint.
    Endpoint string `yaml:"endpoint" env:"AWS_KMS_ENDPOINT"`

    // CredentialsProvider selects the credential resolution strategy.
    //   "default"  (default) — SDK default chain: env → ~/.aws → IMDS → ECS
    //   "env"                — AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY only
    //   "file"               — shared credentials file (profile selectable)
    //   "static"             — explicit access_key + secret_key fields below
    CredentialsProvider string `yaml:"credentials_provider" env:"AWS_KMS_CREDENTIALS_PROVIDER"`

    // StaticAccessKey / StaticSecretKey are used when CredentialsProvider == "static".
    // Prefer env-var injection over inline values.
    StaticAccessKey string `yaml:"static_access_key" env:"AWS_KMS_STATIC_ACCESS_KEY"`
    StaticSecretKey string `yaml:"static_secret_key" env:"AWS_KMS_STATIC_SECRET_KEY"`
    StaticSessionToken string `yaml:"static_session_token" env:"AWS_KMS_STATIC_SESSION_TOKEN"`

    // Profile selects the AWS named profile when CredentialsProvider == "file".
    Profile string `yaml:"profile" env:"AWS_KMS_PROFILE"`

    // Timeout is the per-call timeout applied to KMS API calls.
    // Default: 10s.
    Timeout time.Duration `yaml:"timeout" env:"AWS_KMS_TIMEOUT"`

    // HealthCheckTimeout is the timeout applied specifically to HealthCheck calls.
    // Default: 5s.
    HealthCheckTimeout time.Duration `yaml:"health_check_timeout" env:"AWS_KMS_HEALTH_CHECK_TIMEOUT"`

    // Keys is the ordered list of wrapping key references.
    // The first entry is the active key (used for WrapKey).
    // Additional entries are available for dual-read during rotation.
    // At least one entry is required.
    Keys []AWSKMSKeyRef `yaml:"keys"`
}

// AWSKMSKeyRef maps a KMS key identifier to a gateway version integer.
type AWSKMSKeyRef struct {
    // KeyID is the KMS key identifier: key ID, alias (alias/name), or ARN.
    KeyID   string `yaml:"key_id"`
    // Version is the gateway-local version integer for this key reference.
    Version int    `yaml:"version"`
}
```

**Validation rules:**
- `len(Keys) >= 1`.
- All `KeyID` values must be non-empty; versions must be unique positive integers.
- `CredentialsProvider` must be one of `"default"`, `"env"`, `"file"`,
  `"static"` (or empty, treated as `"default"`).
- When `CredentialsProvider == "static"`: `StaticAccessKey` and
  `StaticSecretKey` must both be non-empty.

**Rationale:** Mirroring the `CosmianConfig` nesting style
(`internal/config/config.go:374`) keeps the config layer consistent. Separating
`AWSKMSKeyRef` from `AWSKMSConfig` makes the YAML shape self-documenting and
allows future extensions (e.g., adding `EncryptionAlgorithm` per key).

```go
func (m *awsKMSManager) UnwrapKey(ctx context.Context, envelope *KeyEnvelope, metadata map[string]string) ([]byte, error) {
    if err := ctx.Err(); err != nil {
        return nil, fmt.Errorf("keymanager/aws: %w", err)
    }
    if envelope == nil || len(envelope.Ciphertext) == 0 {
        return nil, ErrInvalidEnvelope
    }
    m.mu.RLock()
    defer m.mu.RUnlock()
    if m.closed {
        return nil, ErrProviderUnavailable
    }

    // Build candidate list: declared version first, then fallback versions.
    candidates := m.candidateRefs(envelope.KeyVersion)
    encCtx := buildEncryptionContext(metadata, envelope.KeyVersion)

    var lastErr error
    for _, ref := range candidates {
        input := &kms.DecryptInput{
            CiphertextBlob:    envelope.Ciphertext,
            KeyId:             &ref.KeyID,
            EncryptionContext: encCtx,
        }
        out, err := m.client.Decrypt(ctx, input)
        if err != nil {
            lastErr = err
            continue
        }
        return out.Plaintext, nil
    }
    return nil, fmt.Errorf("%w: %s", ErrUnwrapFailed, sanitiseAWSErr(lastErr))
}

// candidateRefs returns key refs to try, with the envelope's declared version first.
func (m *awsKMSManager) candidateRefs(declaredVersion int) []awsKeyRef {
    out := make([]awsKeyRef, 0, m.dualReadWin+1)
    // Declared version first.
    for _, k := range m.keys {
        if k.Version == declaredVersion {
            out = append(out, k)
            break
        }
    }
    // Then remaining keys up to dual_read_window.
    for _, k := range m.keys {
        if k.Version == declaredVersion {
            continue
        }
        if len(out) > m.dualReadWin {
            break
        }
        out = append(out, k)
    }
    return out
}
```

**D4.** `docs/adr/0014-aws-kms-adapter.md` — **new**

ADR documenting:
- Context: operators running on AWS expect AWS KMS as the native key custody
  mechanism; no alternative currently exists in the gateway.
- Decision: implement a `KeyManager` adapter that uses `kms:Encrypt` for
  wrapping and `kms:Decrypt` for unwrapping. DEKs are generated locally; KMS
  never receives plaintext data.
- Alternatives considered: `kms:GenerateDataKey` (generates DEK inside KMS
  — deferred; would simplify key material handling but requires engine
  changes); AWS Encryption SDK format (incompatible with gateway's own
  `KeyEnvelope` format); direct PKCS#11 via CloudHSM (higher complexity,
  covered by HSM adapter).
- Consequences: KMS API availability is on the critical path for
  encryption/decryption; operators must provision IAM policies granting
  `kms:Encrypt`, `kms:Decrypt`, and `kms:DescribeKey` to the gateway role.
  CloudTrail provides a full audit trail of all key operations.
- Security properties: encryption context binds ciphertext to object identity;
  key ARNs are never logged; `ErrUnwrapFailed` is returned without leaking
  AWS error details to callers.

## 10. Follow-ups / Out-of-scope Items Surfaced During Planning

1. **`kms:GenerateDataKey` path** — Using AWS KMS to generate the DEK inside
   the HSM boundary (rather than locally) would harden key material handling
   but requires changes to the engine's `Encrypt` path. Tracked as a
   future enhancement; this plan uses local DEK generation to match all other
   adapters.

2. **Multi-Region Key automatic failover** — MRK replica routing on primary
   region failure requires detecting AWS regional endpoint errors and
   re-routing. Deferred; operators can work around this with `dual_read_window`
   and a secondary key entry pointing to the replica ARN.

3. **CloudTrail integration test** — Verifying that `kms:Encrypt`/`kms:Decrypt`
   events appear in CloudTrail would require an AWS account. Deferred to the
   integration test suite that requires manual AWS setup.

4. **Prometheus metrics for KMS call latency and error rate** — The existing
   `internal/metrics/metrics.go` has `kms_rotated_reads_total`; adding
   `kms_wrap_duration_seconds` and `kms_unwrap_errors_total` labelled by
   `provider=aws` is a clean follow-up that does not block this plan.

5. **`kms:GenerateDataKeyWithoutPlaintext` for pre-fetching** — A caching
   strategy that pre-fetches encrypted DEKs during idle periods could improve
   latency under KMS API throttling. Deferred.

## 12. O'Reilly Verification Gaps (2026-06-08)

> Added by plan-verifier after cross-referencing with O'Reilly authoritative sources.
> These gaps must be resolved before implementation begins.

### GAP-KMS2-1 — HIGH — EncryptionContext does not bind object identity, creating DEK portability risk

**Source:** AWS *Key Management Service Best Practices Whitepaper* (Reference 3 of this plan); *API Security in Action* (Madden, Manning 2021), Ch. 6 — ciphertext-binding security requirements.

**Description:** Plan §3.3 defines the encryption context as: filtered `x-amz-meta-kms-ctx-*` headers (optional, caller-supplied) plus `s3gw-key-version` (always present). The S3 object bucket and key path are never automatically included. This means a `KeyEnvelope.Ciphertext` produced for object A can be used to decrypt data under object B using the same KMS key — because the EncryptionContext still matches (it contains only the key version, not the path). An attacker with write access to object metadata could exploit this.

**Required fix:** Add two mandatory entries to `buildEncryptionContext`: `s3gw-bucket: <bucket>` and `s3gw-object-key: <object-key>`. These must be derived from the `metadata` map passed to `WrapKey`/`UnwrapKey` — the engine already passes object metadata at those call sites. Add unit test `TestAWSKMSManager_EncryptionContext_BucketKeyBinding` asserting these keys are present in every `kms:Encrypt` and `kms:Decrypt` call.

---

### GAP-KMS2-2 — MEDIUM — `TestAWSKMSManager_UnwrapKey_DualRead_WindowZero_NoFallback` absent from §6 test table

**Source:** Plan's own §7 Risks section references this test as required mitigation.

**Description:** The test `TestAWSKMSManager_UnwrapKey_DualRead_WindowZero_NoFallback` is mentioned in §7 Risks as a required mitigation for the dual-read window=0 boundary case, but does NOT appear in the formal §6 test strategy table. The DoD therefore cannot verify this boundary condition. The window=0 case is the default for operators who have not configured `dual_read_window`.

**Required fix:** Add `TestAWSKMSManager_UnwrapKey_DualRead_WindowZero_NoFallback` to the §6 test table under "Unit — dual-read window boundary" and add a corresponding DoD checkbox in §8.

---

### GAP-KMS2-3 — MEDIUM — `aws.EndpointResolverWithOptions` is deprecated in `aws-sdk-go-v2`

**Source:** `aws-sdk-go-v2` deprecation notices starting at v1.17; the plan's own DoD requires `golangci-lint run` to return no new errors.

**Description:** Phase C1 code uses `awsconfig.WithEndpointResolverWithOptions(aws.EndpointResolverWithOptionsFunc(...))` for the LocalStack/moto endpoint override. This API was deprecated at the plan's baseline of `v1.41.7`. Compiling with this API produces deprecation lint warnings, violating the DoD's `golangci-lint` clean requirement.

**Required fix:** Replace with the service-specific `EndpointResolverV2` pattern: implement `kms.EndpointResolverV2` and pass via `kms.WithEndpointResolverV2(...)` in client options, or use the legacy resolver with a `//nolint:staticcheck` comment and a TODO migration note. Document in §4 Phase C1 and add to the Risks table.
