## 1. Context & Current State

The gateway's backend layer is fully decoupled from the encryption layer via
the `s3.Client` interface defined at `internal/s3/client.go:35-61`. The
interface declares 18 methods covering PutObject, GetObject, DeleteObject,
HeadObject, ListObjects, multipart upload lifecycle, CopyObject,
UploadPartCopy, DeleteObjects, and Object Lock operations. The sole
production implementation, `s3Client`, wraps the AWS SDK v2
`github.com/aws/aws-sdk-go-v2/service/s3` client and is instantiated at
`internal/s3/client.go:170-175`.

The `ClientFactory` at `internal/s3/client.go:179-244` creates `s3Client`
instances from a `config.BackendConfig`. The `BackendConfig` struct at
`internal/config/config.go:63-76` already carries a `Provider` string field
(value `"aws"`, `"minio"`, `"hetzner"`, etc.) but this field currently has
no functional effect on which underlying SDK or transport is used — all
providers go through the single AWS SDK v2 S3 path. The gateway works
because GCS, Azure Blob, and other vendors offer their own S3-compatible
API surfaces; however, each has gaps or semantic differences that require
shim logic to paper over.

**GCS (Google Cloud Storage):** GCS exposes an S3-compatible XML API at
`storage.googleapis.com`. Authentication uses HMAC keys (access key /
secret key), not service accounts. Known caveats: object metadata keys must
be lowercase; ETags are quoted MD5 hashes and differ from the multipart
composite ETags that AWS produces; some Object Lock operations are not
supported; multipart upload part count is capped at 32 by some service
quotas on older projects; `CopyObject` response does not include
`LastModified` in some SDK code paths.

**Azure Blob Storage:** Azure exposes an S3-compatible API via the Azurite
emulator and production Azure Blob Storage "S3-compatible API" (preview as
of 2025). The production endpoint pattern is
`<account>.blob.core.windows.net`. Key caveats: the Azure S3 endpoint does
not implement all Object Lock APIs; content-type handling for
`application/octet-stream` may differ; some query parameters differ from
AWS; `x-amz-meta-*` keys are translated to blob metadata which has its own
size limits (8 KiB total); multipart part ETags are not quoted MD5 hashes
but rather block IDs.

Currently, `internal/config/config.go:63-76` defines `BackendConfig` with
only S3-compatible fields (`Endpoint`, `Region`, `AccessKey`, `SecretKey`,
`Provider`, `UseSSL`, `UsePathStyle`, `FilterMetadataKeys`, `Retry`). There
is no `Type` discriminator to select a non-S3 transport path, and no
Azure-specific or GCS-specific adaptation. The `ClientFactory` at `internal/s3/client.go:212-244` always
constructs an AWS SDK v2 S3 client; there is no factory switch for
alternative implementations.

The test provider registry at `test/provider/provider.go` demonstrates the
pattern: each backend is a `Provider` implementation with `Name()`,
`Start()`, `Capabilities()`, `CleanupPolicy()`, and `BackendConfig()` methods.
The MinIO provider at `test/provider/minio.go:60-93` starts a Testcontainers
container and returns an `Instance`; the test harness at
`test/harness/gateway.go:165-170` creates the `ClientFactory` from the
instance's `BackendConfig`.

**What is missing for V1.0-ECOSYS-1:**

1. A `backend.type` discriminator in `BackendConfig` to select the transport
   implementation at runtime.
2. GCS and Azure shim adapters: thin wrappers around the AWS SDK v2 S3
   client that apply backend-specific normalisation (metadata key casing,
   ETag handling, unsupported-operation stubs, provider-specific endpoint
   resolution).
3. A config sub-struct for Azure (`backend.azure`).
4. `NewBackendClient` factory function that dispatches on `backend.type`.
5. Test provider implementations for GCS (external) and Azure (Azurite
   Testcontainers).
6. Helm / config.yaml.example documentation with caveats.
7. A `docs/BACKENDS.md` reference page documenting known caveats per
   backend.

## 2. Design Goals & Non-Goals

### Goals

1. **Single dispatch point** — `NewBackendClient(cfg)` is the sole
   constructor; callers never reference `gcsClient` or `azureClient` directly.
   The rest of the gateway is unchanged.
2. **Minimal surface** — shim adapters implement only the delta: method
   overrides for operations that GCS or Azure handle differently. For
   everything else they delegate to the embedded `s3Client`.
3. **Fail-closed on unsupported ops** — shim methods that are absent on a
   backend return a proper S3 XML error body (`NotImplemented`) so the
   gateway never silently succeeds against a backend that ignored the
   request.
4. **Documented caveats** — every behavioural difference (ETag format,
   metadata limits, missing Object Lock, etc.) is documented in
   `docs/BACKENDS.md` under a per-backend table.
5. **Basic put/get/list interop demonstrated** — the Definition of Done
   requires a runnable integration smoke test for each backend showing
   a round-trip PUT → GET → LIST → DELETE cycle through the encryption
   gateway.
6. **No new required dependencies** — GCS and Azure shims route through the
   existing AWS SDK v2 S3 client (both providers' S3-compatible endpoints
   accept AWS SigV4 requests).

### Non-Goals

1. **Full S3 feature parity on GCS/Azure** — Object Lock, bucket versioning,
   and lifecycle APIs are not wired for these backends in this plan.
   Tracked as follow-ups.
2. **Native SDKs for GCS/Azure** — using `cloud.google.com/go/storage` or
   `github.com/Azure/azure-sdk-for-go` would add large dependencies and a
   second `io.ReadCloser` translation layer. This plan scopes to the S3 API
   surface each provider already exposes.
3. **Automated CI for cloud provider backends** — GCS and Azure integration
   tests require live credentials. They are gated by environment variables
   and documented as manual-run only. Azurite (Azure emulator) is the only
   automatically-started container.
4. **Hot-reload of `backend.type`** — the existing
   `internal/config/config.go:2109` immutability guard already prevents
   hot-reload of `backend.provider`. The same guard is extended to
   `backend.type`.

## 3. Target Interface / Semantics / Behaviour

### 3.1 `backend.type` Discriminator

```go
// BackendType selects the storage transport implementation.
//
// Invariants:
//   - "" (empty) and "s3" both resolve to the AWS SDK v2 S3 client (default).
//   - "gcs" resolves to the GCS S3-compatible shim.
//   - "azure" resolves to the Azure Blob S3-compatible shim.
//   - Any other value is rejected by config.Validate().
type BackendType string

const (
    BackendTypeS3    BackendType = "s3"
    BackendTypeGCS   BackendType = "gcs"
    BackendTypeAzure BackendType = "azure"
)
```

**Rationale:** A string discriminator in `BackendConfig` is the minimal
change that maintains backward compatibility (existing configs with no `type`
field continue to use the S3 path) while enabling a clean factory dispatch.

### 3.2 `NewBackendClient` Factory

```go
// NewBackendClient constructs the appropriate backend Client from cfg.
// Callers must not type-assert the returned Client; all behaviour is expressed
// through the Client interface.
//
// Invariants:
//   - Returns a non-nil Client on success.
//   - Returns an error if cfg.Type == "azure" and cfg.Azure.AccountName is empty.
//   - For "s3" and "gcs" the embedded ClientFactory handles credential
//     validation (access key and secret key must be non-empty).
//   - context.Context is not accepted because client construction is
//     synchronous; it is callers' responsibility to cancel with a deadline.
func NewBackendClient(cfg *config.BackendConfig, opts ...ClientFactoryOption) (Client, error)
```

The factory dispatches on `cfg.Type` (or `BackendTypeS3` when empty):

| `cfg.Type`     | Constructor called                               |
|---|---|
| `""` or `"s3"` | `NewClientFactory(cfg, opts...).GetClient()`   |
| `"gcs"`        | `newGCSClient(cfg, opts...)`                   |
| `"azure"`      | `newAzureClient(cfg, opts...)`                 |

**Rationale:** Keeping the factory function in the `internal/s3` package
preserves encapsulation; callers in `cmd/server/main.go` and
`test/harness/gateway.go` only need to swap `factory.GetClient()` for
`NewBackendClient(cfg)`.

### 3.3 GCS Shim (`gcsClient`)

`gcsClient` embeds an `*s3Client` and overrides the following:

| Method | Override rationale |
|---|---|
| `PutObject` | Force lowercase metadata key names before delegating to `s3Client.PutObject`. GCS's XML API silently drops user metadata keys that are not lowercase. |
| `GetObject` | Normalise returned metadata keys to lowercase; strip `x-goog-*` response headers if present. |
| `HeadObject` | Same normalisation as GetObject. |
| `CopyObject` | GCS's XML API does not return `LastModified` in the `CopyObjectResult` body; substitute `time.Now()` as the sentinel value to match the `s3Client` contract. |
| `CreateMultipartUpload`, `UploadPart`, `CompleteMultipartUpload` | Pass through to `s3Client` with a 32-part guard: if `partNumber > 32` return `s3.NewInvalidArgument("GCS S3-compatible API limits multipart uploads to 32 parts")`. |

The GCS endpoint is resolved from `cfg.Endpoint` (must be set to
`https://storage.googleapis.com`) or the shim returns an error. HMAC-key
credentials are passed as `AccessKey`/`SecretKey` in the standard fields.

### 3.4 Azure Shim (`azureClient`)

`azureClient` embeds an `*s3Client` and overrides:

| Method | Override rationale |
|---|---|
| `PutObject` | Enforce aggregate metadata size ≤ 8 KiB; reject with `InvalidArgument` if exceeded. Also sanitise metadata key characters (Azure allows only alphanumeric and `_`, `$`, `.`). |
| `GetObject` | Map Azure error code `BlobNotFound` → S3 `NoSuchKey` for consistent behaviour. |
| `HeadObject` | Same mapping. |
| Object Lock methods (`PutObjectRetention`, `GetObjectRetention`, etc.) | Return `s3.NewNotImplemented("Azure Blob Storage does not support S3 Object Lock")` for all six methods. |
| `PutObjectLockConfiguration`, `GetObjectLockConfiguration` | Same `NotImplemented` return. |

The Azure endpoint is resolved from `cfg.Endpoint` or assembled from
`cfg.Azure.AccountName` as
`https://<account>.blob.core.windows.net`.

`AzureConfig` sub-struct:

```go
// AzureConfig holds Azure-specific backend settings.
type AzureConfig struct {
    // AccountName is the Azure storage account name.
    // Used to construct the endpoint when cfg.Endpoint is not set.
    AccountName string `yaml:"account_name" env:"BACKEND_AZURE_ACCOUNT_NAME"`
}
```

### 3.5 Config Extension

`BackendConfig` gains two new fields (all backward-compatible):

```go
type BackendConfig struct {
    // ... existing fields unchanged ...

    // Type selects the backend transport: "s3" (default), "gcs", or "azure".
    // Empty string is treated as "s3".
    Type BackendType `yaml:"type" env:"BACKEND_TYPE"`

    // Azure holds Azure-specific backend settings.
    // Only consulted when Type == "azure".
    Azure AzureConfig `yaml:"azure"`
}
```

`config.Validate()` additions:

- Reject unknown `Type` values.
- When `Type == "azure"`: require either `Endpoint` or `Azure.AccountName`
  to be set.
- Extend the hot-reload immutability guard to cover `backend.type`.

### Phase B — GCS shim

- **B1.** Create `internal/s3/gcs_client.go`.
  - Nature: new
  - `gcsClient` struct embeds `*s3Client` (by pointer to allow override).
  - `newGCSClient(cfg *config.BackendConfig, opts ...ClientFactoryOption) (Client, error)`:
    - Normalise endpoint to `https://storage.googleapis.com` if not already
      set.
    - Construct the inner `s3Client` via `NewClientFactory(cfg, opts...).GetClient()`.
    - Return a `*gcsClient` wrapping the inner client.
  - Override `PutObject`: lowercase all metadata keys before delegating.
  - Override `GetObject`: lowercase all returned metadata keys.
  - Override `HeadObject`: same.
  - Override `CopyObject`: substitute `time.Now()` when
    `result.LastModified` is zero.
  - Override `CreateMultipartUpload` / `UploadPart` / `CompleteMultipartUpload`:
    part-count guard at 32; delegate otherwise.

```go
type gcsClient struct {
    inner Client
}

func (c *gcsClient) PutObject(ctx context.Context, bucket, key string, reader io.Reader, metadata map[string]string, contentLength *int64, tags string, lock *ObjectLockInput) error {
    return c.inner.PutObject(ctx, bucket, key, reader, lowercaseKeys(metadata), contentLength, tags, lock)
}

func lowercaseKeys(m map[string]string) map[string]string {
    out := make(map[string]string, len(m))
    for k, v := range m {
        out[strings.ToLower(k)] = v
    }
    return out
}
```

const azureMaxMetadataBytes = 8192

func (c *azureClient) PutObject(ctx context.Context, bucket, key string, reader io.Reader, metadata map[string]string, contentLength *int64, tags string, lock *ObjectLockInput) error {

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| GCS S3-compatible API deprecation or feature freeze | Shim is additive and thin; if GCS drops the S3 API, the shim file is deleted without touching the rest of the gateway. Document this risk in `docs/BACKENDS.md`. |
| Azure S3 API preview instability | Azure shim is scoped to the stable operations (put/get/list/delete); all Object Lock methods return `NotImplemented` rather than forwarding, preventing silent failures if Azure's API surface changes. |
| `NewBackendClient` breaks existing call sites that use `NewClientFactory` directly | `NewClientFactory` remains unchanged; `NewBackendClient` is an additive function. Only `cmd/server/main.go` and `test/harness/gateway.go` are changed to use it. All conformance tests automatically validate the new dispatch path. |
| Metadata size limit (Azure 8 KiB) may be exceeded by the encryption metadata the gateway writes | The encryption metadata (`x-amz-meta-encryption-*` fields) occupies ~600 bytes in compact-key mode. The Azure shim's 8 KiB guard leaves ~7.5 KiB for user metadata — sufficient in practice. Document the limit in `docs/BACKENDS.md`. |
| GCS multipart part limit (32 parts) caps encrypted object size | At default 5 MiB parts, 32 parts = 160 MiB max object size via multipart. At 64 MiB parts (gateway max part buffer), 32 parts = 2 GiB. Document in `docs/BACKENDS.md` with a recommendation to increase part size or use single-part upload for smaller objects. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **Object Lock passthrough on GCS** — GCS's S3 API does not fully implement
   Object Lock as of 2025. A follow-up can add `PutObjectRetention` /
   `GetObjectRetention` support once GCS stabilises their API surface. Tracked
   as a future ecosystem issue.
2. **Native Azure SDK path** — if Microsoft's S3-compatible API surface remains
   limited, a follow-up could implement an `azureBlobClient` backed by
   `github.com/Azure/azure-sdk-for-go` and the Blob Storage REST API
   directly. This would require translating between S3 semantics and Azure
   Blob semantics at every call site, which is substantially larger than this
   plan's shim approach.
3. **Backend type hot-reload** — currently blocked by the same immutability
   guard as `backend.provider`. A future plan could allow switching between
   same-type configurations after validating that the type is unchanged.
