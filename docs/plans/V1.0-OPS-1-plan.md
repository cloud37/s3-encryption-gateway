## 1. Context & Current State

### 1.1 Container Build Pipeline

The release workflow at `.github/workflows/helm.yml:144-164` currently builds
a single-arch `linux/amd64` image using a plain `docker build` command:

```
- name: Build Docker image
  run: docker build -t "${IMAGE_TAG}" .
```

`Dockerfile:33-42` hardcodes `GOARCH=amd64` in all `go build` invocations and
`Dockerfile.fips:37-48` does the same. Neither Dockerfile uses a `--platform`
`ARG`, making them incompatible with `docker buildx build --platform
linux/amd64,linux/arm64` without modification.

The `helm.yml:116-130` "Build binaries" step does cross-compile for
`linux/amd64`, `linux/arm64`, and `darwin/arm64` using bare `GOOS`/`GOARCH`
env-vars, so the Go cross-compilation logic is already proven and reusable in
the multi-arch Docker build.

No Docker Hub push step exists in `helm.yml`. The Trivy scan at `helm.yml:153-164`
references a locally-tagged `ghcr.io/${{ github.repository }}:${{ github.sha }}`
image that is never pushed to any registry — Trivy uses the local Docker daemon
via socket mount. The actual publish registry is Docker Hub
(`docker.io/cloud37io/s3-encryption-gateway`), where images are currently
released manually via `VERSION=0.9.0 make docker-all`. Multi-arch publication
requires an explicit `docker buildx build --push` step targeting `docker.io`
using `DOCKERHUB_USERNAME` / `DOCKERHUB_TOKEN` secrets provisioned in the
repository settings.

### 1.2 SBOM and Provenance Gaps

No SBOM generation exists anywhere in the repository. No `cosign` or SLSA
provenance steps exist. The `Makefile:354-375` `security-scan` and `trivy-scan`
targets are present (added by V1.0-OPS-2) but contain no SBOM or signing logic.

The GitHub Actions workflow runner at `helm.yml:56-60` already requests
`id-token: write` permission on the `release` job, which is the permission
required for `cosign` keyless signing via the GitHub OIDC provider — this
permission was added by V1.0-OPS-2 and does not need to change.

### 1.3 Artifact Hub Chart Metadata Gaps

`helm/s3-encryption-gateway/Chart.yaml:1-28` is missing the mandatory Artifact
Hub annotations:

- `artifacthub.io/category` — not present; required for correct taxonomy.
- `artifacthub.io/license` — not present; required.
- `artifacthub.io/prerelease` — not present.
- `artifacthub.io/changes` — not present; enables per-release changelog on the
  Hub page.
- `artifacthub.io/images` — not present; required for the Hub to display the
  published image references with their digests.
- `artifacthub.io/containsSecurityUpdates` — optional but good hygiene.

`helm/s3-encryption-gateway/README.md:1-10` already documents the Helm
repository URL (`https://cloud37.github.io/s3-encryption-gateway`) and the
`helm repo add` command. The chart is published via `chart-releaser-action`
in `helm.yml:100-107` which pushes a GitHub Pages `index.yaml`. Artifact Hub
ingests GitHub Pages Helm repositories automatically once the repository is
registered; the missing piece is the annotations in `Chart.yaml`.

### 1.4 helm.yml `release` Job — Current Sequence

The current `release` job sequence in `helm.yml`:
1. Checkout + Helm setup + Git config (`helm.yml:61-75`)
2. Add Helm repos + build dependencies (`helm.yml:77-81`)
3. Check if chart version already released (`helm.yml:83-98`)
4. Run chart-releaser (guarded by `already_released == 'false'`) (`helm.yml:100-107`)
5. Set up Go + Build binaries + Upload binaries (`helm.yml:110-141`)
6. Build Docker image (single-arch, local tag) (`helm.yml:143-151`)
7. Trivy container scan (`helm.yml:153-164`)
8. Copy chart README to gh-pages (`helm.yml:166-176`)

Steps 4–7 must be rearchitected: step 6 becomes a multi-arch `buildx` push to
GHCR; a new SBOM generation step follows; and a new `cosign` provenance step
follows the SBOM.

## 2. Design Goals & Non-Goals

### Goals

1. **Multi-arch images on Docker Hub** — every release automatically produces
   `docker.io/cloud37io/s3-encryption-gateway:<version>` as a multi-arch
   manifest list (`linux/amd64`, `linux/arm64`), replacing the current manual
   `VERSION=x.y.z make docker-all` step; the FIPS variant is published as
   `docker.io/cloud37io/s3-encryption-gateway:<version>-fips` (`linux/amd64`
   only, enforced by Go FIPS + crypto toolchain constraints).
2. **SBOM attached to release** — an SPDX-JSON SBOM generated by `syft` is
   uploaded as a GitHub release asset and attached as a cosign attestation on
   the Docker Hub image digest.
3. **SLSA provenance via cosign keyless signing** — the published Docker Hub
   image digest is attested with `cosign` using the GitHub OIDC token; the
   attestation is verifiable by any party with `cosign verify` and the
   Sigstore transparency log.
4. **Artifact Hub listing** — `helm/s3-encryption-gateway/Chart.yaml` contains
   all mandatory Artifact Hub annotations; the chart is discoverable and
   correctly categorised at `artifacthub.io`.
5. **Artifact Hub annotation CI gate** — `helm-test.yml` gains a job that
   fails if mandatory Artifact Hub annotations are missing or empty from
   `Chart.yaml`.
6. **Local developer parity** — `make sbom` and `make docker-buildx` targets
   let developers reproduce the multi-arch build and SBOM generation locally.
7. **Minimal long-lived secrets** — `cosign` uses keyless OIDC signing (no
   `COSIGN_PASSWORD` or private-key secrets). The only long-lived credentials
   required are `DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN` (a scoped Docker
   Hub access token, not the account password), provisioned as GitHub Actions
   repository secrets.

### Non-Goals

1. **Signing the Helm chart tarballs** — chart signing (GPG `--sign` in
   chart-releaser) is deferred; the `cosign` scope in this plan is the
   container image only.
2. **SLSA Build Level 3** — full hermetic reproducible builds are not targeted
   here; Level 2 provenance (cosign OIDC attestation) satisfies the DoD.
3. **GHCR publication** — Docker Hub (`docker.io`) is the only publish target;
   GHCR is used transiently for Trivy scanning only and is not a publish
   target. GHCR mirroring is a follow-up if desired.
4. **FIPS arm64** — `GOFIPS140=v1.0.0` + the Go standard library crypto module
   requires `linux/amd64`; `linux/arm64` FIPS support is a future item pending
   Go toolchain support.
5. **cosign bundle / Rekor transparency log pinning** — the default Sigstore
   public-good instance is used; private Rekor deployment is out of scope.

## 3. Target Interface / Semantics / Behaviour

### 3.1 Multi-arch Docker Build — `docker buildx` Invocation

The `Dockerfile` must accept a `TARGETPLATFORM` and `TARGETARCH` build arg
(populated automatically by `docker buildx`) so the Go cross-compile step
selects the correct `GOARCH`:

```dockerfile
# Build stage
FROM --platform=$BUILDPLATFORM golang:1.26-alpine AS builder

ARG BUILDPLATFORM
ARG TARGETPLATFORM
ARG TARGETARCH
ARG STRIP_SYMBOLS=true
ARG VERSION
ARG COMMIT

RUN apk add --no-cache git make

WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download
COPY . .

RUN if [ "${STRIP_SYMBOLS}" = "false" ]; then \
        LDFLAGS="-X main.version=${VERSION:-dev} -X main.commit=${COMMIT:-unknown}"; \
    else \
        LDFLAGS="-w -s -X main.version=${VERSION:-dev} -X main.commit=${COMMIT:-unknown}"; \
    fi && \
    CGO_ENABLED=0 GOOS=linux GOARCH=${TARGETARCH} go build \
    -trimpath -ldflags="${LDFLAGS}" \
    -o /bin/s3-encryption-gateway ./cmd/server && \
    CGO_ENABLED=0 GOOS=linux GOARCH=${TARGETARCH} go build \
    -trimpath -ldflags="${LDFLAGS}" \
    -o /bin/s3-encryption-gateway-healthcheck ./cmd/healthcheck
```

**Key change:** `GOARCH=amd64` → `GOARCH=${TARGETARCH}`. The `--platform=
$BUILDPLATFORM` on the builder stage ensures the builder itself always runs
natively (cross-compiling Go is pure cross-compile, not emulation).

The runtime stage (`FROM alpine:3.23`) does not need modification — Alpine
publishes `linux/amd64` and `linux/arm64` variants; `docker buildx` selects
the correct one automatically.

**FIPS constraint:** `Dockerfile.fips` retains hardcoded `GOARCH=amd64` and
`GOOS=linux` because `GOFIPS140=v1.0.0` is only validated on `linux/amd64`
by the Go team. No TARGETARCH substitution is applied to `Dockerfile.fips`.

### 3.2 GitHub Actions — `docker buildx` Step

The `helm.yml` "Build Docker image" step is replaced with:

```yaml
- name: Set up QEMU
  uses: docker/setup-qemu-action@v3

- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3

- name: Log in to Docker Hub
  uses: docker/login-action@v3
  with:
    username: ${{ secrets.DOCKERHUB_USERNAME }}
    password: ${{ secrets.DOCKERHUB_TOKEN }}

- name: Build and push multi-arch image
  id: build_image
  uses: docker/build-push-action@v6
  with:
    context: .
    file: Dockerfile
    platforms: linux/amd64,linux/arm64
    push: true
    tags: |
      cloud37io/s3-encryption-gateway:${{ steps.version_check.outputs.chart_version }}
      cloud37io/s3-encryption-gateway:latest
    build-args: |
      VERSION=${{ steps.version_check.outputs.chart_version }}
      COMMIT=${{ github.sha }}

- name: Build and push FIPS image (linux/amd64 only)
  id: build_fips_image
  uses: docker/build-push-action@v6
  with:
    context: .
    file: Dockerfile.fips
    platforms: linux/amd64
    push: true
    tags: |
      cloud37io/s3-encryption-gateway:${{ steps.version_check.outputs.chart_version }}-fips
    build-args: |
      VERSION=${{ steps.version_check.outputs.chart_version }}
      COMMIT=${{ github.sha }}
```

**Prerequisites:** Two GitHub Actions repository secrets must be provisioned
before this workflow can push:
- `DOCKERHUB_USERNAME` — the Docker Hub account name (e.g. `cloud37io`)
- `DOCKERHUB_TOKEN` — a Docker Hub access token (not the account password)
  scoped to read/write for the `cloud37io/s3-encryption-gateway` repository

**Rationale:** Using the `docker/build-push-action` action instead of raw
`docker buildx build` CLI commands provides automatic digest extraction via
`steps.build_image.outputs.digest` — this digest is required by `cosign` for
attestation and is the canonical reference for SBOM attachment. No `packages:
write` permission is needed; Docker Hub authentication is handled entirely by
the login step with the above secrets.

### 3.3 SBOM Generation — `syft`

An SBOM in SPDX-JSON format is generated from the built image:

```yaml
- name: Generate SBOM
  uses: anchore/sbom-action@v0
  with:
    image: cloud37io/s3-encryption-gateway:${{ steps.version_check.outputs.chart_version }}
    format: spdx-json
    output-file: sbom.spdx.json

- name: Upload SBOM to release
  run: |
    gh release upload "${{ steps.version_check.outputs.release_tag }}" \
      sbom.spdx.json \
      --repo "${{ github.repository }}" --clobber
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

The `anchore/sbom-action` action wraps `syft` and handles authentication to
GHCR automatically. It produces a standards-compliant SPDX 2.3 JSON document
listing all layers, packages, and their PURL identifiers.

### 3.4 cosign Keyless Attestation

```yaml
- name: Install cosign
  uses: sigstore/cosign-installer@v3

- name: Sign and attest image with cosign (keyless OIDC)
  run: |
    IMAGE_DIGEST="index.docker.io/cloud37io/s3-encryption-gateway@${{ steps.build_image.outputs.digest }}"
    cosign sign --yes "${IMAGE_DIGEST}"
    cosign attest --yes \
      --predicate sbom.spdx.json \
      --type spdxjson \
      "${IMAGE_DIGEST}"
  env:
    COSIGN_EXPERIMENTAL: "1"
```

**Verifiability contract:** Any third party can verify the attestation with:

```bash
cosign verify \
  --certificate-identity-regexp "https://github.com/cloud37/s3-encryption-gateway/.github/workflows/helm.yml" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  cloud37io/s3-encryption-gateway:<version>
```

**Rationale:** Keyless signing via GitHub OIDC (`id-token: write` already
present on `helm.yml:59`) requires no long-lived secrets. Sigstore's
public-good Rekor transparency log provides an immutable audit trail. The
`--yes` flag is required for non-interactive CI use.

### 3.5 Artifact Hub Annotations in `Chart.yaml`

The following annotations are added to `helm/s3-encryption-gateway/Chart.yaml`:

```yaml
annotations:
  artifacthub.io/category: security
  artifacthub.io/license: MIT
  artifacthub.io/prerelease: "false"
  artifacthub.io/signKey: |
    fingerprint: keyless
    url: https://github.com/cloud37/s3-encryption-gateway
  artifacthub.io/images: |
    - name: s3-encryption-gateway
      image: docker.io/cloud37io/s3-encryption-gateway:1.0.0
      whitelisted: true
    - name: s3-encryption-gateway-fips
      image: docker.io/cloud37io/s3-encryption-gateway:1.0.0-fips
      whitelisted: true
  artifacthub.io/changes: |
    - kind: added
      description: Multi-arch images (linux/amd64, linux/arm64) published to GHCR
    - kind: added
      description: SLSA provenance and SBOM attestations via cosign keyless signing
    - kind: added
      description: Artifact Hub listing with full metadata and image references
  artifacthub.io/containsSecurityUpdates: "false"
```

**Rationale:** Artifact Hub ingests the chart from the `gh-pages` branch
`index.yaml` (already populated by `chart-releaser`). The annotations are
the primary mechanism for Artifact Hub to display security metadata, category
taxonomy, image references with digest links, and the per-release changelog.
Without these annotations the chart is listed but appears incomplete; the
`category: security` classification is required for users browsing the
encryption/security category.

### 3.6 Artifact Hub CI Gate in `helm-test.yml`

A new `artifact-hub-check` job validates that the mandatory annotations are
present and non-empty:

```yaml
artifact-hub-check:
  name: Artifact Hub metadata check
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v6
    - name: Check mandatory Artifact Hub annotations
      run: |
        set -euo pipefail
        CHART="helm/s3-encryption-gateway/Chart.yaml"
        for key in \
          "artifacthub.io/category" \
          "artifacthub.io/license" \
          "artifacthub.io/images"; do
          if ! grep -q "${key}" "${CHART}"; then
            echo "ERROR: Missing mandatory Artifact Hub annotation: ${key}"
            exit 1
          fi
        done
        echo "All mandatory Artifact Hub annotations present."
```

**Rationale:** A lightweight grep-based check prevents accidental removal of
annotations during chart refactors. It runs on every PR touching `helm/**`
(matching the existing `on.pull_request.paths` filter in `helm-test.yml:10-15`).

### 3.7 `Makefile` Targets

Two new targets are added:

```makefile
# docker-buildx — build multi-arch image locally (requires Docker buildx)
# Usage: make docker-buildx VERSION=1.0.0
docker-buildx:
	@echo "Building multi-arch Docker image (linux/amd64, linux/arm64)..."
	@docker buildx build \
		--platform linux/amd64,linux/arm64 \
		--build-arg VERSION=$(VERSION) \
		--build-arg COMMIT=$(COMMIT) \
		-t $(IMAGE_NAME):$(IMAGE_TAG) \
		--load .
	@echo "Multi-arch image built: $(IMAGE_NAME):$(IMAGE_TAG)"

# sbom — generate SPDX-JSON SBOM for the local Docker image using syft
# Requires: syft installed (go install github.com/anchore/syft/cmd/syft@latest)
# Usage: make sbom IMAGE_NAME=cloud37io/s3-encryption-gateway IMAGE_TAG=latest
sbom:
	@echo "Generating SBOM for $(IMAGE_NAME):$(IMAGE_TAG)..."
	@syft $(IMAGE_NAME):$(IMAGE_TAG) -o spdx-json=sbom.spdx.json
	@echo "SBOM written to sbom.spdx.json"
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `linux/arm64` layer fails to build due to Alpine apk package availability differences | Pin base image to `alpine:3.23` (already in `Dockerfile:45`); `ca-certificates` and `tzdata` are available on both `amd64` and `arm64` Alpine. Validate in Phase A with a local `docker buildx build --platform linux/arm64 --load .` smoke test before merging. |
| `docker buildx build --push` fails on first run if `DOCKERHUB_USERNAME` / `DOCKERHUB_TOKEN` secrets are absent or the token lacks write permission | Provision both secrets before the first release run (Phase B1 prerequisite). Use a Docker Hub access token scoped to read/write for `cloud37io/s3-encryption-gateway` only. Test by running the login step manually: `docker login -u <user> --password-stdin <<< <token>`. |
| `steps.build_image.outputs.digest` is empty if `push: true` is not set or the buildx driver does not export digests | Use `docker/build-push-action@v6` (not raw CLI) which guarantees `outputs.digest` when `push: true`. Document in the workflow comment that this value must be non-empty before proceeding to cosign steps. |
| `DOCKERHUB_TOKEN` is a long-lived credential stored as a GitHub Actions secret | Use a Docker Hub **access token** (not the account password) scoped to read/write for `cloud37io/s3-encryption-gateway` only; rotate on a defined schedule (e.g. annually); document rotation procedure in `docs/releasing.md`. |
| Docker Hub rate-limits unauthenticated pulls of base images (`golang:1.26-alpine`) during parallel CI runs | The build stage login step authenticates pulls as well as pushes. If rate limits are hit on the runner's IP before the login step executes, move the `docker/login-action` step earlier in the job (before any `docker pull` or `go mod download` that might trigger implicit pulls). |
| cosign OIDC signing fails if the workflow job does not have `id-token: write` | `helm.yml:59` already has `id-token: write`. Confirm this permission is not removed during Phase B editing. |
| `anchore/sbom-action@v0` is a floating major-version ref that could introduce breaking changes | After initial integration, pin to a specific patch tag (e.g., `anchore/sbom-action@v0.20.0`); add a Renovate rule for `github-actions` managers to keep it updated. |
| Artifact Hub takes 24–48 hours to index a newly registered repository | Register the repository at `artifacthub.io` manually as a post-deploy operator step immediately after the first annotated chart release. Document this in `docs/releasing.md` (Phase F3). |
| `artifacthub.io/images` annotation contains a hardcoded version string that becomes stale after each release | Add a `RELEASING.md` checklist item to update the annotation version before tagging. A follow-up automation (sed/yq in a `make pre-release` target) can automate this; deferred to follow-up. |
| Multi-arch `buildx` push is slower than single-arch `docker build` (QEMU emulation for arm64 layer) | Use `--platform=linux/amd64,linux/arm64` with `docker/build-push-action` which uses the `docker-container` driver; Go cross-compilation avoids QEMU emulation for the build stage (`--platform=$BUILDPLATFORM`). The runtime stage uses pre-built Alpine arm64 layer — no emulation needed. Expected CI time increase: +2–4 min. |
| Trivy scan using GHCR image ref requires the image to be pushed before scanning | Phase B ordering ensures `docker/build-push-action` (push step) precedes the Trivy step. Guard both with the same `already_released == 'false'` condition. |

## 10. Follow-ups / Out-of-scope Items Surfaced During Planning

1. **Helm chart GPG signing** — `chart-releaser-action` supports `--sign`
   with a GPG key. This was explicitly excluded from scope (§2 Non-Goals item 1)
   because the cosign/OIDC model is the preferred modern approach; GPG chart
   signing is a separate follow-up if Artifact Hub's "signed chart" badge is
   desired.
2. **GHCR mirroring** — if publishing to GHCR in addition to Docker Hub is
   desired (e.g. for GitHub Packages discoverability), a secondary
   `docker/build-push-action` step targeting `ghcr.io/cloud37/s3-encryption-gateway`
   can be added alongside the Docker Hub push. Deferred; Docker Hub is the
   canonical registry for this project.
3. **Automated `Chart.yaml` version/annotation update** — a `make pre-release`
   target using `yq` to update `artifacthub.io/images` and
   `artifacthub.io/changes` before tagging would reduce manual error. Deferred
   to a tooling improvement issue.
4. **Renovate pinning for new actions** — `docker/setup-qemu-action`,
   `docker/setup-buildx-action`, `docker/login-action`, `docker/build-push-action`,
   `anchore/sbom-action`, and `sigstore/cosign-installer` should be pinned to
   specific SHA digests (not just semver tags) by Renovate after initial
   integration. This is a `renovate.json` configuration follow-up.
5. **SLSA Build Level 3 hermetic builds** — the current plan achieves SLSA
   Build Level 2 (provenance via OIDC attestation). Level 3 requires a hermetic
   build environment (no network access during build, reproducible outputs).
   Deferred pending Go toolchain support for fully hermetic Alpine builds.
