### 1.3 Missing metrics — gaps in monitoring coverage

The following observability gaps exist today that block a complete operational view:

| # | Gap | Impact |
|---|-----|--------|
| G1 | No KMS health status gauge per provider | Cannot alert when the KMS is degraded or unreachable |
| G2 | No config metadata at-rest encryption enabled gauge | Cannot confirm metadata encryption is active without reading logs |
| G3 | No TLS certificate expiry metric | Cannot alert before cert expiry causes client connection failures |
| G4 | No key rotation worker progress (objects_total, objects_reencrypted, objects_skipped) | Cannot track rotation batch progress |
| G5 | No KDF algorithm gauge (pbkdf2 vs argon2id) | Cannot verify the expected KDF is active post-upgrade |
| G6 | No admin-API request rate / error counter | Admin endpoints are invisible to Prometheus |
| G7 | No active MPU count gauge (in-flight multipart uploads) | Cannot detect MPU accumulation or Valkey saturation |
| G8 | No object size histogram for encrypted objects | Cannot assess encryption overhead relative to object size distribution |

These 8 metrics are specified in detail in §3 and implemented in Phase A.

### 1.4 Helm chart — current monitoring values

`helm/s3-encryption-gateway/values.yaml:757` defines:
- `serviceMonitor.enabled: false` — Prometheus Operator ServiceMonitor (opt-in)
- `podMonitor.enabled: false` — Prometheus Operator PodMonitor (opt-in)
- `metrics.port: 0` — dedicated metrics port (0 = disabled, uses data-plane port)

There are currently **no** Helm values or templates for:
- Grafana dashboard ConfigMap (`monitoring.grafana.dashboard.enabled`)
- PrometheusRule (`monitoring.prometheusRule.enabled`)

### 1.5 Why the existing state is insufficient

Despite a rich metric surface, operators have no turnkey way to answer the most
critical operational questions without building PromQL queries from scratch:
- "Is throughput healthy across all replicas right now?"
- "Are there encryption errors accumulating?"
- "Is the KMS reachable and rotating correctly?"
- "How is Valkey holding up under the MPU load?"

The gap is not in metrics but in packaged visualisation and alerting that ships
with the chart and can be opted into with two Helm values.

## 2. Design Goals & Non-Goals

### Goals

1. **All-metrics coverage** — Every one of the 47 existing Prometheus metrics
   plus the 8 new metrics (§3) must appear in at least one dashboard panel or
   alert rule.
2. **Replica-aware stacked visualisation** — All per-replica rate/throughput
   panels use `sum by (pod)` with a "Stack Normal" display mode, plus a
   combined total line, so operators see per-replica contribution AND fleet
   aggregate simultaneously.
3. **Opt-in via Helm values** — Dashboard ConfigMap and PrometheusRule are each
   independently opt-in (`monitoring.grafana.dashboard.enabled`,
   `monitoring.prometheusRule.enabled`). Both default to `false`. No breaking
   change to existing deployments.
4. **Local verification workflow** — A `dashboard.json.out` file (gitignored
   by the existing `*.out` rule in `.gitignore`) is generated by a helper script
   for loading into a remote Grafana instance during local verification.
5. **Stable, parameterised JSON model** — The dashboard is authored as a Go
   template in `helm/s3-encryption-gateway/templates/grafana-dashboard.yaml`
   and rendered by Helm; the raw panel JSON is also committed as
   `helm/s3-encryption-gateway/dashboards/s3-encryption-gateway.json` for
   direct import without the Helm chart.
6. **Alert completeness** — PrometheusRule covers the top-10 operational
   signals with appropriate `for` durations, severity labels, and runbook
   annotations.
7. **Schema validation** — New `monitoring` Helm values are added to
   `values.schema.json` under `monitoring.grafana` and `monitoring.prometheusRule`
   with draft-07 types and descriptions.

### Non-Goals

1. **Grafana Loki / log-based alerting** — Deferred; log-based panels require
   Loki datasource that may not be present in all deployments.
2. **OTel / tracing dashboards** — Jaeger/Tempo trace panels require OTLP
   collector setup; tracked separately.
3. **Multi-cluster aggregation** — Cross-cluster federation queries are
   out of scope; single-cluster view only.
4. **Auto-provisioning without Prometheus Operator** — We assume the
   Prometheus Operator CRDs are present when `monitoring.prometheusRule.enabled`
   is true; raw YAML examples are provided in `docs/OBSERVABILITY.md` for
   vanilla Prometheus setups.

## 3. Target Interface / Semantics / Behaviour

### 3.1 New metrics (Phase A)

Eight new metrics address the gaps identified in §1.3. They are implemented in
`internal/metrics/metrics.go` following the existing `promauto.With(reg)` factory
pattern.

#### G1 — KMS health status

```go
// kmsHealthy reports the result of the last KMS health check per provider.
// Value: 1 = healthy, 0 = unhealthy/unavailable.
// Labels: provider ∈ {cosmian, aws, vault, memory, self_contained}
kmsHealthy *prometheus.GaugeVec
// Name: "gateway_kms_healthy"
```

Updated by `internal/admin/health.go` on each health-check cycle (same
timer that drives `gateway_mpu_valkey_up`). Invariant: set once per
health-check interval; never nil after construction.

#### G2 — Metadata encryption status

```go
// metadataEncryptionEnabled reports whether encrypted-object-metadata
// (V1.0-CRYPTO-3) is active. Value: 1 = active, 0 = inactive.
metadataEncryptionEnabled prometheus.Gauge
// Name: "gateway_metadata_encryption_enabled"
```

Set at startup in `cmd/server/main.go` alongside `SetFIPSMode`.

#### G3 — TLS certificate expiry

```go
// tlsCertExpirySeconds reports seconds until TLS certificate expiry
// for the gateway's own serving certificate. Value: 0 when TLS is disabled.
// Labels: role ∈ {data_plane, admin, metrics}
tlsCertExpirySeconds *prometheus.GaugeVec
// Name: "gateway_tls_cert_expiry_seconds"
```

Updated by a background goroutine in `cmd/server/main.go` every 1 hour.

#### G4 — Key rotation worker progress

```go
// keyRotationObjectsTotal counts objects processed by the rotation worker.
// Labels: result ∈ {reencrypted, skipped_locked, skipped_plaintext, error}
keyRotationObjectsTotal *prometheus.CounterVec
// Name: "gateway_key_rotation_objects_total"
```

Incremented in `internal/admin/rotation_worker.go` (or equivalent). Distinct
from the existing `gateway_rotation_skipped_locked_total` which tracks only the
locked sub-case; this counter tracks the full rotation batch view.

#### G5 — KDF algorithm gauge

```go
// kdfAlgorithmActive reports the active KDF algorithm as a label.
// Value: always 1.0. Labels: algorithm ∈ {pbkdf2-sha256, argon2id}
kdfAlgorithmActive *prometheus.GaugeVec
// Name: "gateway_kdf_algorithm_active"
```

Set at startup from `cfg.Encryption.KDF.Algorithm`.

#### G6 — Admin API request metrics

```go
// adminAPIRequestsTotal counts all requests to the admin listener.
// Labels: method, path (sanitised), status_code
adminAPIRequestsTotal *prometheus.CounterVec
// Name: "gateway_admin_api_requests_total"

adminAPIRequestDuration *prometheus.HistogramVec
// Name: "gateway_admin_api_request_duration_seconds"
// Labels: method, path
```

Emitted from the admin server middleware in `internal/admin/server.go`.

#### G7 — Active MPU gauge

```go
// mpuActiveUploads reports the current count of in-flight multipart
// uploads tracked in Valkey. Updated on Create (+1) and
// Complete/Abort (-1) state store operations.
mpuActiveUploads prometheus.Gauge
// Name: "gateway_mpu_active_uploads"
```

Maintained as a gauge in `internal/mpu/state.go`.

#### G8 — Encrypted object size histogram

```go
// encryptedObjectBytes is a histogram of plaintext object sizes seen
// by the encryption engine at Encrypt() time.
// Buckets: 1KB, 16KB, 256KB, 1MB, 16MB, 64MB, 256MB, 1GB
encryptedObjectBytes prometheus.Histogram
// Name: "gateway_encrypted_object_bytes"
// Buckets: []float64{1<<10, 16<<10, 256<<10, 1<<20, 16<<20, 64<<20, 256<<20, 1<<30}
```

Observed in `internal/crypto/engine.go` at the top of `Encrypt()`.

### 3.2 Grafana dashboard layout

The dashboard is a single JSON document with 8 rows (Grafana "collapsed row"
panel type) and a `$pod` template variable to filter by replica or show all.

| Row | Title | Panels |
|-----|-------|--------|
| R1 | **Traffic Overview** | RPS (stacked by pod), p50/p95/p99 latency, error rate %, active connections |
| R2 | **Encryption Engine** | Encrypt/decrypt RPS (stacked), throughput bytes/s (stacked), encryption error rate, object size distribution, buffer pool hit ratio |
| R3 | **S3 Operations** | S3 ops/s by operation (stacked by pod), S3 error rate by error_type, backend retry rate, retry give-up rate, backoff latency heatmap |
| R4 | **KMS & Key Rotation** | KMS health status (state timeline), active key version, rotation ops/s, rotation duration heatmap, in-flight wraps gauge, rotated reads/s, rotation skipped locked, KDF algorithm (stat panel) |
| R5 | **Multipart Uploads** | Active MPU count (gauge), MPU success/error rate (stacked), parts/s (stacked), Valkey state store latency, Valkey up/insecure state timeline, manifest size distribution, legacy read rate |
| R6 | **Security & Config** | FIPS mode (stat), metadata encryption enabled (stat), HW acceleration (stat), Valkey insecure (stat), TLS cert expiry (gauge with threshold coloring), pprof request rate |
| R7 | **Admin API** | Admin API RPS (stacked by pod), admin API error rate, admin API latency p99 |
| R8 | **Runtime** | Goroutines (stacked by pod), heap alloc bytes (stacked), sys bytes (stacked), GC pause / memory pressure (derived from Go runtime metrics) |

**Stacking convention:** All rate/throughput panels use:
```json
{
  "fieldConfig": { "defaults": { "custom": { "stacking": { "mode": "normal", "group": "A" } } } },
  "options": { "tooltip": { "mode": "multi", "sort": "none" } }
}
```
A second series with `sum(...)` without `by (pod)` is added as a dashed-line
override in the same panel to show the fleet total.

**Template variables:**
- `$datasource` — Prometheus datasource UID (default: `${DS_PROMETHEUS}`)
- `$pod` — multi-value, sourced from `label_values(http_requests_total, pod)`, default `.*`
- `$namespace` — sourced from `label_values(http_requests_total, namespace)`
- `$interval` — fixed options `1m, 5m, 15m, 30m`

### 3.3 Helm values schema

New top-level `monitoring` stanza in `values.yaml`:

```yaml
monitoring:
  grafana:
    dashboard:
      # Set to true to create a ConfigMap containing the Grafana dashboard JSON.
      # The ConfigMap is labelled grafana_dashboard: "1" for sidecar auto-discovery.
      enabled: false
      # Namespace to create the ConfigMap in. Defaults to Release.Namespace.
      namespace: ""
      # Additional labels to add to the ConfigMap (e.g. grafana.sidecar.dashboards.label).
      labels: {}
      # Name of the Grafana datasource to use. Injected as the default datasource UID.
      datasource: "Prometheus"
  prometheusRule:
    # Set to true to create a PrometheusRule resource (requires Prometheus Operator).
    enabled: false
    # Namespace for the PrometheusRule. Defaults to Release.Namespace.
    namespace: ""
    # Additional labels to add to the PrometheusRule (e.g. release: kube-prometheus-stack).
    labels: {}
```

JSON Schema additions live under `properties.monitoring` in
`helm/s3-encryption-gateway/values.schema.json`.

### 3.4 PrometheusRule — top-10 alert rules

| Alert name | Expr (condensed) | For | Severity | Runbook |
|-----------|-----------------|-----|---------|---------|
| `S3GatewayHighErrorRate` | `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05` | 5m | critical | `RUNBOOK.md#high-error-rate` |
| `S3GatewayHighLatency` | `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 2` | 10m | warning | `RUNBOOK.md#high-latency` |
| `S3GatewayEncryptionErrors` | `increase(encryption_errors_total[5m]) > 0` | 1m | critical | `RUNBOOK.md#encryption-errors` |
| `S3GatewayKMSUnhealthy` | `gateway_kms_healthy == 0` | 2m | critical | `RUNBOOK.md#kms-unhealthy` |
| `S3GatewayValkeyDown` | `gateway_mpu_valkey_up == 0` | 2m | critical | `RUNBOOK.md#valkey-down` |
| `S3GatewayValkeyInsecure` | `gateway_mpu_valkey_insecure == 1` | 0m | warning | `RUNBOOK.md#valkey-insecure` |
| `S3GatewayTLSCertExpiringSoon` | `gateway_tls_cert_expiry_seconds < 604800` | 1h | warning | `RUNBOOK.md#tls-cert-expiry` |
| `S3GatewayTLSCertExpiryCritical` | `gateway_tls_cert_expiry_seconds < 172800` | 15m | critical | `RUNBOOK.md#tls-cert-expiry` |
| `S3GatewayHighRetryGiveUpRate` | `rate(s3_backend_retry_give_ups_total[5m]) > 0.1` | 5m | warning | `RUNBOOK.md#backend-retries` |
| `S3GatewayLegacyValkeyStateReads` | `increase(gateway_mpu_state_legacy_reads_total[1h]) > 0` | 5m | info | `RUNBOOK.md#valkey-legacy-state` |

**Rationale:** Alerts are defined in two severity tiers (critical / warning) using
the `severity` label convention established by kube-prometheus-stack. The `for`
durations are chosen to eliminate flap from transient spikes while remaining
actionable (critical < 5 min, warning < 15 min as a rule of thumb).

### 3.5 Local verification workflow

A script `hack/export-dashboard.sh` renders the Helm-templated ConfigMap,
extracts the JSON value, and writes it to `hack/dashboard.json.out` (gitignored
by `.gitignore`'s `*.out` rule). Operators then import `hack/dashboard.json.out`
directly into their Grafana UI for local verification before committing.

```bash
# Usage:
bash hack/export-dashboard.sh [DATASOURCE_UID]
# Output: hack/dashboard.json.out
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| New metrics increase scrape size and Prometheus TSDB cardinality | New metrics add at most ~12 label combinations (well within Prometheus defaults). `gateway_admin_api_requests_total` uses path sanitisation (same `sanitizePathLabel` function as HTTP metrics at `metrics.go:332`) to bound cardinality. |
| `tlsCertExpirySeconds` background goroutine leaks on shutdown | Use a context derived from the server's base context; goroutine exits when the context is cancelled (same pattern as `StartSystemMetricsCollector`). |
| Dashboard JSON becomes stale when metric names change | Dashboard is rendered from the chart's `dashboards/` directory; panel queries use exact metric names. A CI step validates JSON parse + `helm lint`. Alert rules are validated with `promtool check rules` in CI. |
| `hack/dashboard.json.out` accidentally committed | `.gitignore` has `*.out` (existing rule); plan additionally adds explicit `hack/dashboard.json.out` entry. |
| Grafana ConfigMap namespace mismatch (dashboard not discovered by sidecar) | `monitoring.grafana.dashboard.namespace` allows override; documented in `values.yaml` comment and README. |
| PrometheusRule labels don't match Prometheus Operator's `ruleSelector` | `monitoring.prometheusRule.labels` is a free-form map; documented with common example (`release: kube-prometheus-stack`). |
| `ObserveEncryptedObjectBytes` adds latency on the encryption hot path | The histogram `Observe` call is a single mutex-protected bucket increment; measured overhead < 50 ns per operation (consistent with existing `encryptionDuration` histogram). |
| TLS cert expiry metric not populated when TLS is disabled | Guard with `if cfg.TLS.Enabled { ... }` in main.go; when TLS is disabled the gauge remains at 0 (documented as "0 = TLS disabled"). |

All security-critical risks from the above are promoted to DoD in §8.

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **Loki log-based panels** — Several operational questions (e.g. "show me all
   KMS timeout log lines") require Loki datasource; a separate issue should track
   a Loki-backed row for log-derived panels once Loki is part of the reference stack.

2. **Jaeger / Tempo trace integration** — A "Traces" row linking to Jaeger exemplars
   (already supported in `metrics.go:getExemplar`) requires a trace datasource;
   deferred to a follow-up observability issue.

3. **Alert severity escalation via Alertmanager routes** — Documenting Alertmanager
   receiver routing (PagerDuty, Slack) is out of scope; the `RUNBOOK.md` playbooks
   intentionally leave escalation path customisation to the operator.

4. **`kms_rotated_reads_total` cardinality** — The `{key_version, active_version}`
   label combination can grow unbounded if many key versions are in rotation; a
   follow-up should cap the label cardinality or switch to a bucketed histogram.
   This is pre-existing; not introduced by this plan.

5. **Grafana Alerting vs. PrometheusRule** — We ship only PrometheusRule (evaluated
   by the Prometheus Operator). Grafana-native alerting is a separate concern; no
   Grafana Alert YAML is shipped.
