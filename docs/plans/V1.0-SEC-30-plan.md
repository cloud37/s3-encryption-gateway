Status: Draft (Rev 2 — envelope-encryption design per operator feedback)
Owner: Security

> Scope clarified relative to the security audit report produced on

> after multi-phase confidence filtering. This plan bundles all three into a
> single remediation issue because they share the same source (audit

> can be split across PRs if desired.
>
> **Rev 2 note:** The original draft (Rev 1) proposed PBKDF2-SHA256 600k for
> the state-encryption key. Operator feedback: this is architecturally
> inconsistent with the project's envelope-encryption direction and adds
> unnecessary KDF cost (even if one-time at startup). Rev 2 replaces the
> PBKDF2 derivation with a **random state DEK wrapped by the existing
> `KeyManager`** — the same envelope pattern used for object DEKs. This
> eliminates password-based key derivation from the state-encryption path
> entirely and reuses the KeyManager infrastructure the operator already
> configured.

## 1. Context & Current State

### 1.1 Finding A — `/metrics` path-prefix authentication bypass (HIGH)

`internal/api/auth_middleware.go:60` unconditionally skips authentication
for any request path satisfying `strings.HasPrefix(path, "/metrics")`, in
addition to exact-match skips for `/health`, `/ready`, `/live`, `/metrics`:

```go
path := r.URL.Path
if path == "/health" || path == "/ready" || path == "/live" || path == "/metrics" || strings.HasPrefix(path, "/metrics") {
    next.ServeHTTP(w, r)
    return
}
```

The gorilla/mux S3 subrouter (registered in `cmd/server/main.go` around the
`r.PathPrefix("/").Subrouter()` call) exposes `/{bucket:[^/]+}` and
`/{bucket:[^/]+}/{key:.+}` routes. The regex `[^/]+` matches
`metrics-attackerbucket` as a bucket name, so a request to
`/metrics-attackerbucket/key` bypasses `AuthMiddleware` and falls through to
the bucket handler. The handler then re-signs with the gateway's backend
credentials (`internal/api/utils.go` `forwardToBackend`) and proxies to S3.

The exact `/metrics` route (`cmd/server/main.go:1026`) only matches the
literal path, so any longer `/metrics-*` path is routed by the bucket
matcher. The result is a complete unauthenticated authentication bypass:
any network client can read, write, and enumerate objects on any backend
bucket the gateway's credentials can reach, in the default configuration
(`ProxiedBucket` unset) and against any bucket whose name starts with
`metrics` when `ProxiedBucket` is set.

The prefix check is the only defect. The exact-match checks for `/health`,
`/ready`, `/live`, and `/metrics` are intentional and correct — Kubernetes
probes and Prometheus scrapers need unauthenticated access to those
specific paths.

### 1.2 Finding B — Valkey state-encryption KDF uses HKDF-Extract on a password with no work factor (HIGH)

`internal/mpu/state.go:538` derives the 32-byte AES-256 key that protects
MPU state at rest in Valkey via a single `hkdf.Extract` pass:

```go
func deriveStateAEADKey(password string) []byte {
    // HKDF-Extract is sufficient here because the input (password) is already
    // a high-entropy secret. ...
    salt := []byte("s3eg-mpu-state-v1")
    extracted := hkdf.Extract(sha256.New, []byte(password), salt)
    key := make([]byte, 32)
    copy(key, extracted)
    return key
}
```

Three problems:

1. **Wrong primitive.** HKDF-Extract is designed for uniformly-random input
   keys (RFC 5869 §2.2). The input here is a human passphrase — the
   README, Helm chart, and RUNBOOK all describe
   `encryption_password` / `VALKEY_ENCRYPTION_PASSWORD` as a password, with
   examples like `"your-password"` and `"dev-password"`. The inline comment
   ("already a high-entropy secret") contradicts the documented usage.
2. **No work factor.** A single SHA-256 compression over the password offers
   no resistance to offline dictionary attack. An attacker who exfiltrates
   Valkey state ciphertext (ADR 0009 explicitly models this threat) can try
   billions of passwords per second on a GPU.
3. **Hardcoded cross-deployment salt.** `"s3eg-mpu-state-v1"` is identical
   in every installation. Two deployments sharing a password derive
   byte-identical keys, enabling precomputation/rainbow-table reuse across
   the entire fleet.

**Remediation approach (Rev 2):** Instead of replacing HKDF with PBKDF2
(which would add a one-time ~600ms KDF cost at startup and is
architecturally inconsistent with the project's envelope-encryption
direction), the state-encryption key will be a **random 32-byte state DEK**
generated once and **wrapped by the existing `KeyManager`** — the same
envelope pattern used for object DEKs. The wrapped state DEK is persisted
in Valkey. On restart, the KeyManager unwraps it. Per-call
`EncryptState`/`DecryptState` use the cached state DEK directly (AES-GCM,
microseconds). No password derivation in the state-encryption path.

The threat model in `docs/adr/0009-encrypted-multipart-uploads.md`
explicitly considers read-only Valkey exfiltration, so offline attack is
in scope.

### 1.3 Finding C — Silent plaintext fallback on Valkey state AEAD failure (MEDIUM)

`internal/mpu/state.go:368-383` (`Get`) and `:458-473` (`List`) silently
fall back to treating the raw ciphertext as plaintext JSON when
`DecryptState` returns an AEAD authentication failure:

```go
if s.encryptState {
    decrypted, err := s.DecryptState(metaBytes)
    if err != nil {
        // Decryption failed — try legacy plaintext fallback.
        s.legacyWarn.Do(func() { ...Warn("Unencrypted Valkey state detected...") })
        s.metrics.IncMPUStateLegacyReads()
        // Leave metaBytes as the raw value; unmarshal below will handle plaintext JSON.
    } else {
        metaBytes = decrypted
    }
}
```

This fails open. The README's "fail-closed guarantees" table promises the
gateway "refuses to silently degrade security". An attacker (or compromised
Valkey replica) able to write a forged `mpu:<uploadID>` record can inject
plaintext state with attacker-chosen `WrappedDEK`, `IVPrefixHex`,
`KMSKeyID`, and part records; the gateway trusts it as if it had been
authenticated. The existing test
`TestStateStore_Get_LegacyPlaintextFallback` (`state_encryption_test.go:124`)
explicitly asserts this fallback works, which codifies the fail-open
behaviour as intended.

The fallback exists for one-time migration from legacy plaintext state, but
it cannot distinguish "legacy plaintext written before encryption was
enabled" from "attacker-injected plaintext written after encryption was
enabled". It must be gated behind an explicit operator opt-in.

### 1.4 Why the existing state is insufficient

The three findings together undermine the gateway's two core security
boundaries: the authentication boundary (Finding A lets unauthenticated
clients reach the backend) and the MPU state integrity boundary (Findings
B+C let an attacker with Valkey read access offline-recover the password,
and an attacker with Valkey write access inject unauthenticated state). All
three are independently exploitable and each has a concrete attack path
documented in the audit report.

### 1.5 KeyManager availability at the call site

`cmd/server/main.go:445-517` constructs a `keyManager` that is **always
non-nil** by the time `NewValkeyStateStore` is called at line 771:

- When `cfg.Encryption.KeyManager.Enabled` is true: the configured provider
  (self_contained AES, RSA, or Cosmian) is built, optionally wrapped in a
  `FallbackKeyManager` with a `PasswordKeyManager` for legacy objects.
- When `cfg.Encryption.KeyManager.Enabled` is false (password-only mode): a
  `PasswordKeyManager` is constructed at line 513-517 and assigned to
  `keyManager`.

This means the envelope approach works in **all modes**: the state DEK is
wrapped by whatever KeyManager the operator configured. In password-only
mode, the `PasswordKeyManager` wraps it (PBKDF2 cost is one-time during
`WrapKey`/`UnwrapKey` at startup, not per Valkey call). In envelope mode,
wrapping is instant (AES key wrap or KMS call). The per-call
`EncryptState`/`DecryptState` always uses the cached random state DEK
directly — no KDF on the hot path.

## 2. Design Goals & Non-Goals

### Goals

1. **Exact-match auth bypass for system endpoints** — only the literal paths
   `/health`, `/ready`, `/live`, `/metrics` skip authentication; no
   path-prefix matching. A request to `/metrics-anything` requires valid
   SigV4/V2 credentials.
2. **No regression to probe/scrape functionality** — Kubernetes liveness,
   readiness, and health probes, and Prometheus scraping of `/metrics`,
   continue to work unauthenticated on their dedicated paths.
3. **Envelope-encrypted state key** — the state-encryption key is a random
   32-byte DEK wrapped by the existing `KeyManager`, consistent with the
   object DEK envelope pattern. No password-derived key in the
   state-encryption path.
4. **No KDF on the MPU hot path** — `EncryptState`/`DecryptState` use the
   cached state DEK directly (AES-GCM). KDF/KMS cost is incurred only at
   startup (`WrapKey`/`UnwrapKey`).
5. **Fail-closed state decryption** — when `encryptState` is enabled and
   `DecryptState` fails, `Get`/`List` return an error; they do NOT fall back
   to plaintext unmarshal.
6. **Explicit, opt-in legacy plaintext migration** — a config flag
   (`allow_legacy_plaintext_state`, default `false`) gates the one-time
   migration path, with a startup log warning and a metric.
7. **Backward compatibility for existing encrypted state** — deployments
   already running with the HKDF-derived key must continue to decrypt
   existing in-flight state after upgrade, via a V1 legacy key fallback
   during the 7-day TTL window.
8. **FIPS compliance preserved** — the state DEK is a random key wrapped by
   the KeyManager; AES-256-GCM is used for state AEAD. No argon2id in the
   state-encryption path.

### Non-Goals

1. **Per-credential bucket authorization.** The audit confirmed the default
   "any authenticated credential can access any backend bucket" behaviour
   is a documented v0.8 design decision (`docs/issues/v0.8-issues.md:70`),
   not a vulnerability. Out of scope.
2. **SigV4 body-hash verification.** The audit filtered this as a false
   positive (MitM out of scope per the README threat model; the gateway
   re-signs with backend credentials using the actual body hash). Out of
   scope.
3. **Replacing `InsecureSkipVerify` for Valkey/audit TLS.** Operator opt-in
   with startup warnings; application-layer AEAD protects the data. Out of
   scope (filtered as false positive).
4. **State-encryption key rotation tooling.** The envelope approach enables
   future rotation (generate a new state DEK, wrap it, re-encrypt state),
   but rotation tooling is deferred to a follow-up issue.
5. **Removing the `PasswordKeyManager` fallback.** In password-only mode the
   `PasswordKeyManager` still wraps the state DEK; this is acceptable
   because the PBKDF2 cost is one-time at startup, not per-call, and the
   state DEK itself is random (not password-derived).

## 3. Target Interface / Semantics / Behaviour

### 3.1 AuthMiddleware — exact-match system endpoints

`AuthMiddleware` skips authentication **only** for the exact paths
`/health`, `/ready`, `/live`, `/metrics`. The `strings.HasPrefix` call is
removed. No other behaviour changes.

```go
// System endpoints that bypass authentication. Kept as a set for O(1)
// lookup and to make the policy explicit.
var systemEndpoints = map[string]bool{
    "/health":  true,
    "/ready":   true,
    "/live":    true,
    "/metrics": true,
}

// isSystemEndpoint reports whether path is an unauthenticated system
// endpoint. It uses exact match only — prefix matching here would let
// /metrics-<anything> bypass auth and fall through to the /{bucket} route.
func isSystemEndpoint(path string) bool {
    return systemEndpoints[path]
}
```

In the middleware:
```go
if isSystemEndpoint(r.URL.Path) {
    next.ServeHTTP(w, r)
    return
}
```

**Rationale:** A map lookup makes the policy explicit and avoids accidental
prefix matches. Gorilla/mux `/{bucket:[^/]+}` would otherwise capture any
`/metrics-*` path as a bucket name.

### 3.2 Envelope-encrypted state key

The state-encryption key follows the same envelope pattern as object DEKs:

1. **State DEK**: a random 32-byte AES-256 key generated once via
   `io.ReadFull(rand.Reader, ...)`.
2. **Wrapping**: the state DEK is wrapped by the configured `KeyManager`
   (`keyManager.WrapKey(ctx, stateDEK, nil)` → `*KeyEnvelope`).
3. **Persistence**: the wrapped `KeyEnvelope` (JSON-encoded) is stored in
   Valkey under key `mpu:state-key-wrapped` (a plain Redis string, not
   encrypted — it is already wrapped by the KMS).
4. **Unwrapping on restart**: load the wrapped envelope from Valkey, call
   `keyManager.UnwrapKey(ctx, envelope, nil)` → plaintext state DEK, cache
   in memory.
5. **Per-call AEAD**: `EncryptState`/`DecryptState` use the cached state DEK
   directly via `aes.NewCipher(stateDEK)` + `cipher.NewGCM`. No KDF, no
   KMS call, no password derivation on the hot path.
6. **Zeroization**: the state DEK is zeroized in `Close()` via
   `zeroBytes()`.

```go
// ValkeyStateStore — revised key fields:
//   stateDEK       []byte          // random 32-byte AES-256 key (envelope DEK)
//   stateKeyV1     []byte          // legacy HKDF key for pre-upgrade state (nil if none)
//   keyManager     crypto.KeyManager // wraps/unwraps the state DEK
//
// Invariants:
//   - stateDEK is always 32 bytes when encryptState is true.
//   - stateDEK is zeroized in Close().
//   - stateKeyV1 is derived from legacyPassword (if non-empty) for
//     backward-compatible decrypt of pre-V1.0-SEC-30 state. It is
//     read-only (never used for new encryption).
//   - keyManager is required when encryptState is true; nil is rejected
//     at construction time (fail-closed).
```

**Backward compatibility (V1 legacy key):**

Existing deployments have state encrypted with
`HKDF-Extract(SHA256, password, "s3eg-mpu-state-v1")`. After upgrade, new
state is encrypted with the envelope state DEK. `DecryptState` tries the
envelope state DEK first, then falls back to the V1 HKDF key (if present)
for decrypting pre-upgrade state. Since MPU state has a 7-day TTL
(`ValkeyDefaultTTLSeconds`), all pre-upgrade state expires within 7 days.
After that, the V1 key is dead code and can be removed in a future version.

The V1 key is derived from `legacyPassword` (the gateway's
`encryption.password`) if non-empty. For brand-new deployments,
`legacyPassword` can be empty — no V1 fallback is configured.

```go
// DecryptState tries the envelope state DEK first, then the legacy V1
// HKDF key (if present) for backward-compatible decrypt of pre-upgrade
// state. Both paths return ErrStateDecryptFailed on failure; neither
// ever falls back to plaintext (that is gated by allowLegacyPlaintext
// at the Get/List call sites, not inside DecryptState).
func (s *ValkeyStateStore) DecryptState(ciphertext []byte) ([]byte, error) {
    // ... version/length checks (unchanged) ...
    if pt, err := tryOpen(s.stateDEK, ciphertext); err == nil {
        return pt, nil
    }
    if s.stateKeyV1 != nil {
        if pt, err := tryOpen(s.stateKeyV1, ciphertext); err == nil {
            return pt, nil
        }
    }
    return nil, fmt.Errorf("%w: aead open failed for all configured keys", ErrStateDecryptFailed)
}
```

**Constructor signature change:**

```go
// NewValkeyStateStore constructs a ValkeyStateStore.
//
// keyManager is required when encryptState is true (fail-closed if nil).
// It wraps/unwraps the random state DEK.
//
// legacyPassword is used ONLY to derive the V1 HKDF key for
// backward-compatible decrypt of pre-V1.0-SEC-30 state. Pass "" for
// brand-new deployments with no legacy state. The V1 key is read-only
// (never used for new encryption) and expires with the 7-day state TTL.
func NewValkeyStateStore(
    ctx context.Context,
    cfg config.ValkeyConfig,
    keyManager crypto.KeyManager,
    legacyPassword string,
) (*ValkeyStateStore, error)
```

**Rationale:** This reuses the `KeyManager` interface (defined at
`internal/crypto/keymanager.go:24`) and the `KeyEnvelope` struct (line 72)
that the operator already configured for object encryption. The state DEK
is a random key — no password derivation, no KDF cost on the hot path. The
`PasswordKeyManager` (always available in password-only mode per
`cmd/server/main.go:513-517`) wraps it with a one-time PBKDF2 cost at
startup; in envelope mode, wrapping is instant.

### 3.3 Fail-closed state decryption with opt-in legacy plaintext

`Get` and `List` fail closed when `encryptState` is enabled and
`DecryptState` returns an error. Plaintext fallback is gated behind a new
`allowLegacyPlaintext` field on `ValkeyStateStore`, set from a new config
flag.

```go
// Get — new behaviour:
if s.encryptState {
    decrypted, err := s.DecryptState(metaBytes)
    if err != nil {
        if !s.allowLegacyPlaintext {
            // Fail closed: do NOT treat AEAD failure as plaintext.
            return nil, fmt.Errorf("%w: state decryption failed for upload %q "+
                "(set allow_legacy_plaintext_state=true only if migrating from plaintext)", err, uploadID)
        }
        s.legacyWarn.Do(func() { ...Warn("...") })
        s.metrics.IncMPUStateLegacyReads()
        // Leave metaBytes as the raw value; unmarshal below will handle plaintext JSON.
    } else {
        metaBytes = decrypted
        s.metrics.IncMPUStateEncryptedWrites("get")
    }
}
```

New config field on `ValkeyConfig`:

```go
// AllowLegacyPlaintextState permits Get/List to fall back to plaintext JSON
// unmarshal when state AEAD decryption fails. Intended ONLY for one-time
// migration from a pre-encryption deployment. Defaults to false (fail-closed).
// V1.0-SEC-30.
AllowLegacyPlaintextState bool `yaml:"allow_legacy_plaintext_state" env:"VALKEY_ALLOW_LEGACY_PLAINTEXT_STATE"`
```

**Rationale:** The README's fail-closed table is the contract. Migration is
a real need but must be explicit, logged, and time-bounded by operator
action. The flag is a bool (not a window) because the migration is
one-time: enable it, restart, let `Get` decode legacy blobs, then disable
and restart again.

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Existing deployment upgrades and cannot decrypt pre-upgrade Valkey state because the V1 key was not derived | B4 derives the V1 key from `legacyPassword` (the gateway's `encryption.password`) when non-empty. B7's `TestStateStore_DecryptState_LegacyV1Ciphertext_OpensWithV1Key` locks this in. MPU state TTL is 7 days, so the V1 key is only needed during that window. |
| Wrapped state DEK lost (Valkey wiped) → state undecryptable | The wrapped DEK is stored in Valkey (`mpu:state-key-wrapped`). If Valkey is wiped, all in-flight MPU state is also lost (it lives in Valkey), so the wrapped DEK is recovered alongside the state it protects — both are gone together. For persistent backups, the wrapped DEK can be backed up as part of Valkey snapshots. ADR 0014 documents this. |
| `SET mpu:state-key-wrapped` race: two gateway replicas start simultaneously and both generate a DEK | B4 uses `SET NX` (atomic first-writer-wins). The second replica reads the stored wrapped envelope and unwraps it. Add test `TestNewValkeyStateStore_ConcurrentStateDEKGeneration` under `-race`. |
| Import cycle: `internal/mpu` → `internal/crypto` for `KeyManager`/`KeyEnvelope` | B1 documents the fallback: if `go build` reports a cycle, define a minimal `KeyManager` interface in `state.go` (structural typing makes `crypto.KeyManager` assignable to it). Verify during implementation. |
| Removing `strings.HasPrefix` breaks a non-S3 route mounted under `/metrics/...` | A2/A3 tests prove the exact-match behaviour. If a future sub-path is needed, it must be registered as an explicit route, not via prefix auth bypass. Document in ADR 0014. |
| Operator had `allow_legacy_plaintext_state` implicitly "on" (the old default) and relies on the fallback for legacy state | C is a **behaviour change**: the new default is fail-closed. D7 CHANGELOG calls this out under `Security` and `Changed`. D6 README updates the fail-closed table. Operators who need the migration path set the new flag explicitly. |
| `NewValkeyStateStore` signature change breaks external callers | The function is internal (`internal/mpu`); the only caller is `cmd/server/main.go:771` (B5). No external API surface. |
| Password-only mode: `PasswordKeyManager` wrapping the state DEK incurs PBKDF2 cost at startup | This is a one-time cost during `WrapKey` (first start) / `UnwrapKey` (restart), NOT per Valkey call. The state DEK is random — no password derivation on the hot path. Acceptable per operator feedback. |
