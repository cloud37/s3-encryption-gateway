## 1. Context & Current State

### 1.1 What V1.0-S3-2 Delivered (Baseline)

V1.0-S3-2 (`docs/plans/V1.0-S3-2-plan.md`) implemented all ~35 passthrough
handlers for bucket- and object-level subresources. Every handler body is a
single call to `h.handlePassthrough(…)` at `internal/api/handlers.go:4390–4630`,
which in turn calls `forwardToBackend` in `internal/api/utils.go:143` and
`copyProxyResponse` at `utils.go:124`.

The `handlePassthrough` helper re-signs the request with backend credentials
via the AWS SDK v4 signer when the incoming request carries no `Authorization`
header. When `Authorization` is present (i.e., a forwarded SigV4 request from
the client), the request body and all headers — including `x-amz-acl`,
`x-amz-grant-*`, CORS headers, and lifecycle headers — are preserved verbatim
and forwarded to the backend at `utils.go:176–178`.

Tagging, ACL subresource endpoints, lifecycle, CORS, policy, replication, and
all Tier 1–3 operations from V1.0-S3-2 are routed and forwarded. The S3
conformance suite in `test/conformance/suite_test.go:239–255` already registers
17 `S3Compat_*` cases for those operations.

### 1.2 Gaps Addressed by This Plan

**Gap 1 — Inline canned ACL on PutObject is silently dropped.**
`handlePutObject` calls `s3Client.PutObject(…, tagging, lockInput)` via the
AWS SDK at `internal/s3/client.go:461`. The SDK `PutObjectInput` struct
`internal/s3/client.go:435–446` does not populate `input.ACL` (the
`ObjectCannedACL` field). The `x-amz-acl` header from the original request is
therefore **not forwarded** to the backend — it is silently dropped. The same
applies to the fine-grained grant headers (`x-amz-grant-full-control`,
`x-amz-grant-read`, `x-amz-grant-read-acp`, `x-amz-grant-write-acp`).

This is a correctness gap: clients that expect an object to inherit a canned
ACL at write time (e.g., `public-read`, `private`) will receive 200 OK from
the gateway but find the ACL is not applied on the backend. The object still
exists unencrypted upstream of the gateway and is created with the backend's
default ACL instead.

**Gap 2 — Inline canned ACL on CreateMultipartUpload is silently dropped.**
`handleCreateMultipartUpload` constructs `metadata` from `x-amz-meta-*` headers
only (`handlers.go` around line 1700+). `x-amz-acl` and `x-amz-grant-*`
headers are not extracted and not passed to `s3Client.CreateMultipartUpload`.

**Gap 3 — Lifecycle response headers not validated or tested.**
AWS S3 populates `x-amz-expiration` and `x-amz-restore` headers on GetObject
and HeadObject responses when a lifecycle rule is active. These are backend
headers that `copyProxyResponse` (`utils.go:124–135`) forwards verbatim (because
they are not hop-by-hop headers). However, there are no unit or conformance
tests that assert these headers survive the passthrough path.

**Gap 4 — Feature matrix documentation is absent.**
`docs/S3_API_IMPLEMENTATION.md` covers the encryption-focused operations in
detail but has no table documenting which inline headers on PutObject/
CreateMultipartUpload/CopyObject are forwarded, blocked, or unhandled — and
which differ by provider (MinIO, Garage, AWS S3, Wasabi).

**Gap 5 — `CapBucketACL` / `CapObjectACL` deliberately excluded from MinIO
provider bitmap.**
The MinIO provider at `test/provider/minio.go:34–43` deliberately excludes
`CapBucketACL`, `CapObjectACL`, `CapBucketPolicy`, `CapBucketLifecycle`,
`CapBucketCors`, and `CapBucketEncryption` because the default MinIO test
container does not enable these features. This means the 6 corresponding
conformance test cases are permanently skipped in the PR gate. The plan
specifies additional unit-level tests that cover these paths without requiring a
compatible backend.

### 1.3 Current Behaviour of `forwardToBackend`

`internal/api/utils.go:172–178`: the proxy request is built with
`proxyReq.Header = r.Header.Clone()` followed by `proxyReq.Host = u.Host`.
This means **all** incoming request headers — including `x-amz-acl`,
`x-amz-grant-*`, CORS origin, `x-amz-storage-class`, `x-amz-server-side-encryption`,
and lifecycle-response headers — survive the `handlePassthrough` path verbatim.

The gap is exclusively in the **SDK-mediated path** (`handlePutObject` and
`handleCreateMultipartUpload`) where `forwardToBackend` is **not** used: these
handlers call the AWS SDK `PutObject` and `CreateMultipartUpload` directly
through `s3Client`, which builds a `PutObjectInput` struct. Only the fields
explicitly populated in that struct reach the backend.

## 2. Design Goals & Non-Goals

### Goals

1. **ACL inline header forwarding** — `x-amz-acl` and the five `x-amz-grant-*`
   headers on PutObject and CreateMultipartUpload must reach the backend.
2. **CopyObject ACL passthrough** — `x-amz-acl` on CopyObject (inline with the
   `x-amz-copy-source` header) must be forwarded to the backend.
3. **Lifecycle header survival** — `x-amz-expiration` and `x-amz-restore`
   response headers on GetObject/HeadObject must pass through
   `copyProxyResponse` / the SDK response path without stripping.
4. **Feature matrix document** — `docs/S3_API_IMPLEMENTATION.md` must contain a
   definitive table listing every S3 inline-header on PutObject/CopyObject and
   its gateway disposition (forwarded / extracted / silently dropped).
5. **Unit test coverage** — Every new or changed behaviour has at least one
   table-driven unit test covering the happy path and the drop/forward
   distinction (no Docker, no integration tags).
6. **Conformance matrix green** — The feature matrix referenced in the DoD
   (`docs/issues/v1.0-issues.md:132`) is populated and all P1 operations
   are green (pass or documented skip with reason).
7. **No silent drops** — After this plan, no S3-standard inline header that the
   gateway advertises as forwarded silently fails to reach the backend.

### Non-Goals

1. **Implementing ACL enforcement in the gateway** — ACL validation and
   enforcement is the backend's responsibility; the gateway is a transparent
   proxy. Deferred indefinitely (no tracking issue).
2. **Implementing server-side encryption validation** (`x-amz-server-side-encryption-*`)
   — The gateway performs client-side encryption independently of backend SSE.
   Inline SSE headers on PutObject are silently passed through to the backend
   (this is intentional and documented in V1.0-S3-2 §4.3). No change needed.
3. **Expanding MinIO provider capabilities** (`CapBucketACL` etc.) — MinIO's
   default test container does not support these features. Expanding the bitmap
   requires a different MinIO configuration (IAM + policies + Content-MD5).
   Deferred to V1.0-COMPAT-1.
4. **Adding new S3 operations** — All operations were covered by V1.0-S3-2.
   This plan adds tests and documentation only (plus the ACL fix).
5. **Lifecycle trigger simulation** — Verifying that a lifecycle rule actually
   causes expiration is an E2E concern requiring time manipulation. Deferred to
   nightly soak tests.

## 3. Target Interface / Semantics / Behaviour

### 3.1 Inline Header Passthrough Contract (PutObject)

The following table defines the **definitive passthrough contract** for inline
headers on PutObject after this plan is implemented. It is the source of truth
for `docs/S3_API_IMPLEMENTATION.md`.

| Header | Disposition | Mechanism | Notes |
|---|---|---|---|
| `x-amz-tagging` | **Forwarded** | Extracted, validated (`validateTags`), passed to SDK `PutObjectInput.Tagging` | `handlers.go:1443–1445`, `client.go:444–446` |
| `x-amz-acl` | **Forwarded** (after fix) | Extracted, mapped to `types.ObjectCannedACL`, passed to SDK `PutObjectInput.ACL` | Gap 1 — currently silently dropped |
| `x-amz-grant-full-control` | **Forwarded** (after fix) | Extracted, passed to SDK `PutObjectInput.GrantFullControl` | Gap 1 |
| `x-amz-grant-read` | **Forwarded** (after fix) | Extracted, passed to SDK `PutObjectInput.GrantRead` | Gap 1 |
| `x-amz-grant-read-acp` | **Forwarded** (after fix) | Extracted, passed to SDK `PutObjectInput.GrantReadACP` | Gap 1 |
| `x-amz-grant-write-acp` | **Forwarded** (after fix) | Extracted, passed to SDK `PutObjectInput.GrantWriteACP` | Gap 1 |
| `x-amz-storage-class` | **Forwarded** via metadata | Passed in the `metadata` map as a header; SDK does not natively use it — backend must accept it as a standard header forwarded by `forwardSignatureV4Request` | Provider quirk: MinIO ignores; AWS S3 respects |
| `x-amz-server-side-encryption` | **Passed through verbatim** | Not extracted by gateway; forwarded on re-signed request; backend applies its own SSE layer | Documented in V1.0-S3-2 §4.3 |
| `x-amz-object-lock-mode` | **Forwarded** | Extracted via `extractObjectLockInput`, passed to SDK `PutObjectInput.ObjectLockMode` | `object_lock.go:102` |
| `x-amz-object-lock-retain-until-date` | **Forwarded** | As above | `object_lock.go:120` |
| `x-amz-object-lock-legal-hold` | **Forwarded** | As above | `object_lock.go:131` |
| `x-amz-meta-*` | **Forwarded** | Extracted as user metadata map | `handlers.go:1465–1471` |
| `Content-Type` | **Forwarded** | SDK reads from request body content-type detection or explicit header | Standard |
| `Content-Encoding` | **Forwarded** | Standard header copied by SDK | Standard |
| `Cache-Control` | **Forwarded** | Standard header | Standard |

### 3.2 Inline Header Passthrough Contract (CreateMultipartUpload)

| Header | Disposition | Notes |
|---|---|---|
| `x-amz-acl` | **Forwarded** (after fix) | Gap 2 — currently silently dropped |
| `x-amz-grant-*` | **Forwarded** (after fix) | Gap 2 — currently silently dropped |
| `x-amz-tagging` | **NOT forwarded at CreateMultipartUpload** | Tags set at CompleteMultipartUpload time via `?tagging` subresource. AWS behaviour: tagging on CreateMultipartUpload is accepted but not applied until CompleteMultipartUpload. Gateway behaviour: the `x-amz-tagging` header on CreateMultipartUpload is not extracted; callers must use `PUT /{bucket}/{key}?tagging` after complete. **Document as known limitation** in `S3_API_IMPLEMENTATION.md`. |
| `x-amz-meta-*` | **Forwarded** | Extracted and passed to `s3Client.CreateMultipartUpload` |
| `x-amz-server-side-encryption` | **Passed through verbatim** | As per §3.1 |

### 3.3 Lifecycle Response Header Contract

When a backend applies a lifecycle rule to an object, it may return the
following response headers on GetObject and HeadObject responses:

| Header | Direction | Gateway Disposition |
|---|---|---|
| `x-amz-expiration` | Response | **Forwarded verbatim** by `copyProxyResponse` (not hop-by-hop) |
| `x-amz-restore` | Response | **Forwarded verbatim** by `copyProxyResponse` |
| `x-amz-delete-marker` | Response | **Forwarded verbatim** |

These headers originate from the backend. The gateway's `copyProxyResponse`
(`utils.go:124–135`) removes only the 8 hop-by-hop headers; `x-amz-*` headers
are preserved. No code change is required; conformance tests verify the
forwarding invariant.

**Note:** The SDK-mediated path for GetObject (`handleGetObject`) does forward
`x-amz-expiration` and `x-amz-restore` via the response header copy loop at
`handlers.go:634–643`. The copy loop explicitly checks only `x-amz-meta-*`
headers for the decryption case. Standard `x-amz-*` headers that do not start
with `x-amz-meta-` are always forwarded. ✅

### 3.4 Provider Quirks Matrix

| Feature | AWS S3 | MinIO (default) | Garage | Wasabi |
|---|---|---|---|---|
| `x-amz-acl` on PutObject | ✅ Full support | ⚠️ Default container requires IAM policy; use `private` or `public-read` | ⚠️ Limited; `private` and `public-read` accepted | ✅ Full support |
| `x-amz-grant-*` on PutObject | ✅ | ❌ Not supported in default container | ❌ | ✅ |
| `x-amz-tagging` on PutObject | ✅ | ✅ (`CapInlinePutTagging`) | ✅ | ✅ |
| `?tagging` subresource | ✅ | ✅ (`CapObjectTagging`) | ✅ | ✅ |
| `?acl` subresource (bucket) | ✅ | ⚠️ MinIO test container: not in default cap bitmap | ⚠️ Skip | ✅ |
| `?acl` subresource (object) | ✅ | ⚠️ Same | ⚠️ Skip | ✅ |
| `?lifecycle` subresource | ✅ | ⚠️ `CapBucketLifecycle` not in MinIO test container | ✅ Garage supports | ✅ |
| `x-amz-expiration` response | ✅ | Returned when lifecycle rule matches | ✅ | ✅ |
| `x-amz-storage-class` | ✅ | ⚠️ Ignored in most configs | ❌ | ✅ |

**Rationale:** This matrix drives which capability bits gate conformance tests,
and is the basis for the DoD "feature matrix green for priority features"
criterion.

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Changing `PutObject` / `CreateMultipartUpload` signatures breaks existing callers and mocks | Audit all call sites with `grep -n "PutObject\|CreateMultipartUpload"` before merging; add empty-string args to non-handler call sites (e.g., tests). Update mock implementations in `handlers_test.go`. |
| `types.ObjectCannedACL(cannedACL)` passes unvalidated strings to the SDK | The SDK will pass the value to the backend as-is; the backend will return an error for invalid ACL values. The gateway does not need to validate because it is a proxy and different backends support different ACL string sets. Add a note in the handler comment. |
| `x-amz-grant-*` headers are not supported by all backends (MinIO default, Garage) | These are already capability-gated via `CapBucketACL` / `CapObjectACL`. Unconditionally forwarding them on backends that ignore them is safe (no error from backend). |
| Lifecycle conformance test cannot force backend to set `x-amz-expiration` | Use `t.Logf` (not `t.Errorf`) for the header assertion to avoid false failures in local CI. The test verifies header forwarding path, not lifecycle evaluation. |
| `CapBucketACL` / `CapObjectACL` are excluded from MinIO provider bitmap, so ACL conformance tests are permanently skipped in PR gate | The gap is documented in §1.2 Gap 5. Unit tests (Phase C) fully cover the forwarding path without Docker. The conformance test will run against Wasabi (external provider) or a MinIO cluster with IAM enabled. |
| Upstream SDK `PutObjectInput.ACL` field type is `types.ObjectCannedACL` (a string type) | Casting with `types.ObjectCannedACL(cannedACL)` is the correct pattern; no invalid-cast risk. The SDK will reject unknown values at the HTTP level from the backend, not at compilation. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **`x-amz-tagging` on CreateMultipartUpload** — AWS S3 accepts the header at
   Create time but requires `?tagging` subresource to set tags on the assembled
   object. The gateway should document this and consider forwarding it for
   backends that respect it at Create time (e.g., Wasabi). Tracked as a
   documentation note only; implementation deferred.

2. **`CapBucketACL` / `CapObjectACL` in MinIO test container** — Enabling these
   capabilities requires a MinIO cluster with IAM policies configured. This
   would make ACL conformance tests run in the PR gate. Deferred to V1.0-COMPAT-1.

3. **`x-amz-storage-class` on PutObject** — The SDK `PutObjectInput.StorageClass`
   field is not populated. Storage class is a per-backend concept (not security-
   relevant); forwarding it is a nice-to-have deferred to a future maintenance
   issue.

4. **`x-amz-acl` on CopyObject** — The `handleCopyObject` re-encrypt path
   (`handlers.go:~4107`) calls `s3Client.PutObject` on the destination with an
   empty canned ACL. This means the destination object always gets the bucket
   default ACL. The source ACL is not copied. This is consistent with S3
   semantics (CopyObject does not copy ACLs by default; callers must set them
   separately). Document in `S3_API_IMPLEMENTATION.md`.

## 11. References

1. Amazon Web Services, *Amazon S3 security best practices*, 2024.
   <https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html>
   **Rationale for transparent-proxy design:** The gateway cannot enforce ACL
   policy at the proxy layer because (a) ACL values are backend-specific (AWS
   S3, MinIO, Garage, Wasabi differ in which canned ACLs and grant headers
   they accept), (b) the gateway has no knowledge of backend IAM policies or
   account-level settings, and (c) enforcing ACL validation at the gateway
   would break the transparency contract for clients that depend on
   backend-specific ACL behavior. ACL validation and enforcement is therefore
   the backend's responsibility.
