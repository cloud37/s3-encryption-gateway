## 1. Context & Current State

### 1.1 What exists today

The conformance suite (`test/conformance/`) is provider-agnostic: each test
function accepts a `provider.Instance` and is gated by capability bits declared
in `test/provider/capabilities.go:9–61`. Local providers (MinIO, Garage,
RustFS, SeaweedFS) spin up via Testcontainers; external providers (AWS S3,
Backblaze B2, Wasabi, Hetzner) are registered conditionally from environment
variables (`test/provider/aws.go:11–34`, `test/provider/b2.go`, etc.).

The `TestConformance` root (`test/conformance/suite_test.go:31`) iterates
`provider.All()` and runs every test case whose capability bit is satisfied.
The matrix guard (`test/conformance/matrix_guard_test.go:40`) enforces that
no test body hard-codes a provider name — capability bits must be used instead.

Existing coverage of SDK-adjacent features is limited to the AWS SDK v2
(used internally by the test harness itself). There are no tests for:
- `awscli` (`aws s3 cp`, `aws s3api put-object`, `aws s3api get-object`)
- `boto3` (Python)
- `s5cmd` (Go high-throughput client)
- `rclone` (multi-backend sync client)
- MinIO Python/JS client (`minio-py`)
- AWS SDK Go v1 (`github.com/aws/aws-sdk-go`)

There is no published compatibility document. The milestone acceptance criterion
(`docs/issues/v1.0-issues.md:14`) requires: "Compatibility certification matrix
completed for targeted SDKs/tools."

### 1.2 Why this is insufficient

The gateway is designed to be a transparent S3 proxy — any S3-compatible
SDK or tool that can reach it must be able to do basic put/get/delete without
knowing it is talking to a proxy that encrypts in-flight. Without a certified
matrix:

1. Users cannot determine whether their chosen client will work without
   reading source code.
2. Caveats (e.g., rclone's server-side copy semantics, or s5cmd's ETag
   expectations) are undocumented, leading to silent data corruption risks.
3. There is no automated regression gate; a future handler change could break
   a widely-used client without detection.

The existing conformance suite tests the gateway via the AWS SDK v2 Go client
exclusively. The SDK/tool matrix extends this pattern to cover 6 additional
clients across 3 languages and 2 CLI tools, executed against all local
providers in CI.

### 1.3 Scope of operations per tool

The matrix does not aim to exercise every S3 API operation per tool. It defines
a **smoke-test baseline**: the smallest set of operations that, if all passing,
certifies the tool can be used as a drop-in client against the gateway. Each
tool's smoke test must cover:

| Operation | Reason |
|---|---|
| `PutObject` (single, ≤ 5 MiB) | Basic write path; verifies auth + headers |
| `GetObject` (full read) | Round-trip correctness; verifies decrypt-on-read |
| `HeadObject` | Metadata visibility; verifies content-length accuracy |
| `DeleteObject` | Cleanup path |
| `ListObjectsV2` | Discovery; verifies key visibility and size accuracy |
| `PutObject` (multipart, ≥ 3 parts) | Gated on `CapMultipartUpload`; verifies MPU path |
| `CopyObject` | Gated on `CapMultipartCopy`; verifies server-side copy semantics |

Tools that natively expose only a subset of these operations (e.g., `awscli s3`
high-level commands don't directly expose `HeadObject`) will exercise the
available surface and note gaps as caveats in the published matrix.

## 2. Design Goals & Non-Goals

### Goals

1. **Automated matrix** — Every tool in the matrix has at least one Go
   conformance test (`//go:build conformance`) that spins up a real container,
   invokes the tool against the in-process gateway harness, and asserts
   round-trip correctness. No manual steps.
2. **Provider-agnostic** — Tool smoke tests follow the same `provider.Instance`
   pattern as existing conformance tests. They run against all local providers
   (MinIO, Garage, RustFS, SeaweedFS) in CI.
3. **Capability-gated** — New capability bits (`CapSDKAWSv2`, `CapCLIAWS`,
   `CapCLIS5cmd`, `CapCLIRclone`, etc.) allow providers to opt in or out per
   tool without modifying test bodies.
4. **Published document** — `docs/SDK_COMPATIBILITY.md` is the canonical matrix,
   generated from test results, listing each tool's version, pass/fail status per
   operation, and caveats. Updated as part of the release workflow.
5. **CI gate** — The `conformance.yml` workflow gains a `compat-matrix` job that
   runs `test-conformance-compat` on every PR to `main`, with local providers.
   External providers (AWS S3, B2) are checked nightly.
6. **Zero new external dependencies in the gateway binary** — Tool containers
   run via Testcontainers in the test binary only; no new `require` entries in
   `go.mod` for the gateway itself.
7. **Matrix guard compliance** — All new test functions use capability bits; no
   provider name literals appear in test bodies.

### Non-Goals

1. **Full API parity testing per tool** — Only the smoke-test baseline (§1.3)
   is automated. Exhaustive API coverage is tracked by V1.0-S3-1 and V1.0-S3-2.
2. **Windows or macOS CI** — The conformance suite runs only on `ubuntu-latest`.
   Windows/macOS caveats are documented manually if reported.
3. **AWS SDK v1 Go** — SDK v1 is EOL for security patches as of July 2025; it
   will not be added to the matrix. Callers should migrate to SDK v2.
4. **Performance benchmarking per tool** — Throughput numbers are tracked by
   V1.0-PERF-1. This plan only asserts functional correctness.
5. **KMS mode per tool** — Tool tests run in single-password (passphrase) mode
   only. KMS-mode compatibility is validated by the existing KMS conformance
   tests (`CapKMSIntegration`).

## 3. Target Interface / Semantics / Behaviour

### 3.1 Capability bits

Six new capability bits are added to `test/provider/capabilities.go`:

```go
// CapSDKAWSGoV2 indicates the provider is suitable for the AWS SDK Go v2
// smoke test. Set on all local providers and external AWS S3.
CapSDKAWSGoV2 Capabilities = 1 << 20

// CapSDKBoto3 indicates the provider is suitable for the boto3 (Python)
// smoke test. Requires Docker (Python container).
CapSDKBoto3 Capabilities = 1 << 21

// CapCLIAWSCLI indicates the provider is suitable for the awscli smoke test.
// Requires Docker (awscli container).
CapCLIAWSCLI Capabilities = 1 << 22

// CapCLIS5cmd indicates the provider is suitable for the s5cmd smoke test.
// Requires Docker (s5cmd container).
CapCLIS5cmd Capabilities = 1 << 23

// CapCLIRclone indicates the provider is suitable for the rclone smoke test.
// Requires Docker (rclone container).
CapCLIRclone Capabilities = 1 << 24

// CapSDKMinIOPy indicates the provider is suitable for the minio-py (Python)
// smoke test. Requires Docker (Python container).
CapSDKMinIOPy Capabilities = 1 << 25
```

**Rationale:** Following the existing pattern in `capabilities.go`, capability
bits are the only mechanism test bodies may use to differentiate provider
behaviour. This satisfies the matrix guard (`matrix_guard_test.go:66`) and
the project's open/closed principle for adding new providers.

### 3.2 Per-tool container interface

Each tool smoke test uses the same structural pattern:

```go
// sdkToolRunner is the common interface for launching an external SDK/CLI
// tool inside a Docker container against the gateway.
type sdkToolRunner interface {
    // Name returns the tool name for log output (e.g. "boto3", "awscli").
    Name() string
    // Image returns the Docker image tag to pull (e.g. "amazon/aws-cli:2.22.0").
    Image() string
    // Script returns the inline shell/Python script or command args to execute.
    // The caller injects gateway endpoint, credentials, and bucket as
    // environment variables before running the container.
    Script(env sdkTestEnv) string
    // AssertOutput inspects the container exit code and stdout/stderr.
    // Returns nil on success, a descriptive error on failure.
    AssertOutput(exitCode int, stdout, stderr string) error
}

// sdkTestEnv carries the gateway endpoint and credentials for injection into
// the container environment.
type sdkTestEnv struct {
    Endpoint  string // e.g. "http://172.17.0.1:8080" (gateway, not backend)
    Region    string
    AccessKey string
    SecretKey string
    Bucket    string
    Key       string // unique object key for this test run
}
```

**Rationale:** The interface isolates the tool-specific Docker image selection
and script content from the test orchestration loop. New tools are added by
implementing `sdkToolRunner` without touching the test harness.

### 3.3 Published matrix document schema

`docs/SDK_COMPATIBILITY.md` has the following schema:

```markdown
# SDK / Tool Compatibility Matrix

> **Generated:** YYYY-MM-DD (vX.Y release).
> Automated smoke tests run against MinIO (local) and AWS S3 (nightly).

## Matrix

| Tool | Version | PutObject | GetObject | HeadObject | DeleteObject | ListObjects | Multipart | CopyObject | Notes |
|---|---|---|---|---|---|---|---|---|---|
| AWS SDK Go v2 | v1.102 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | — |
| boto3 | 1.35 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | — |
| awscli | 2.22 | ✅ | ✅ | ⚠️ | ✅ | ✅ | ✅ | ✅ | HeadObject via `s3api` |
| s5cmd | 2.3 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ⚠️ | cp --source-version-id unsupported |
| rclone | 1.68 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ⚠️ | server-side copy requires --s3-copy-cutoff=0 |
| minio-py | 7.2 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | — |

## Caveats
[per-row caveats expanded here]
```

**Rationale:** A table-per-operation format gives operators an at-a-glance
view. Emoji status (`✅` pass, `⚠️` pass with caveat, `❌` fail, `➖` not
supported by tool) maps directly to the test assertion result, enabling
automated generation from test JSON output.

  func (r *boto3Runner) Name() string  { return "boto3" }
  func (r *boto3Runner) Image() string { return "python:3.13-slim" }
  func (r *boto3Runner) Script(env sdkTestEnv) string {
      return fmt.Sprintf(`python3 -c "
  import os, boto3
  s3 = boto3.client('s3',
      endpoint_url=os.environ['GATEWAY_ENDPOINT'],
      aws_access_key_id=os.environ['AWS_ACCESS_KEY_ID'],
      aws_secret_access_key=os.environ['AWS_SECRET_ACCESS_KEY'],
      region_name='us-east-1'
  )
  bucket = os.environ['GATEWAY_BUCKET']
  key    = os.environ['GATEWAY_KEY']
  # PutObject
  s3.put_object(Bucket=bucket, Key=key, Body=b'compat-test-data')
  # HeadObject
  h = s3.head_object(Bucket=bucket, Key=key)
  assert h['ContentLength'] == len(b'compat-test-data')
  # GetObject
  obj = s3.get_object(Bucket=bucket, Key=key)
  assert obj['Body'].read() == b'compat-test-data'
  # ListObjectsV2
  r = s3.list_objects_v2(Bucket=bucket, Prefix=key)
  assert any(o['Key'] == key for o in r.get('Contents', []))
  # DeleteObject
  s3.delete_object(Bucket=bucket, Key=key)
  print('boto3:OK')
  "`)
  }
  func (r *boto3Runner) AssertOutput(code int, out, _ string) error {
      if code != 0 { return fmt.Errorf("boto3 exited %d", code) }
      if !strings.Contains(out, "boto3:OK") { return fmt.Errorf("boto3: expected OK marker") }
      return nil
  }

  ```yaml
  compat-matrix:
    name: Compat Matrix / ${{ matrix.provider }}
    runs-on: ubuntu-latest
    timeout-minutes: 45
    needs: [unit]
    strategy:
      fail-fast: false
      matrix:
        provider: [minio, garage, rustfs, seaweedfs]
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-go@v6
        with:
          go-version-file: go.mod
      - name: SDK/Tool compat smoke tests / ${{ matrix.provider }} (-race)
        env:
          GATEWAY_TEST_SKIP_EXTERNAL: "1"
        run: |
          for p in minio garage rustfs seaweedfs; do
            if [[ "$p" != "${{ matrix.provider }}" ]]; then
              var="GATEWAY_TEST_SKIP_$(echo "$p" | tr '[:lower:]' '[:upper:]')"
              export "$var=1"
            fi
          done
          go test -count=1 -tags=conformance -race -v -timeout 30m \
            -run "TestConformance/${{ matrix.provider }}/Compat_" \
            ./test/conformance/...
  ```
- Add `compat-matrix` to the `needs` list of the existing
  `conformance-summary` job so the PR summary reflects compat results.

## 6. Test Strategy

| Layer | Tests |
|---|---|
| Conformance — AWS SDK Go v2 | `TestConformance/*/Compat_AWSGoV2` — asserts PutObject, GetObject (byte equality), HeadObject (ContentLength), ListObjectsV2 (key present + size accurate), DeleteObject; multipart gated on `CapMultipartUpload`; CopyObject gated on `CapMultipartCopy` |
| Conformance — boto3 | `TestConformance/*/Compat_Boto3` — Python 3.13-slim container; pip-installs boto3 inline; asserts `boto3:OK` marker in stdout and exit code 0 |
| Conformance — awscli | `TestConformance/*/Compat_AWSCLI` — amazon/aws-cli:2.22.0 container; uses `aws s3 cp`, `aws s3api head-object`, `aws s3 ls`, `aws s3 rm`; asserts exit code 0 and expected stdout patterns |
| Conformance — s5cmd | `TestConformance/*/Compat_S5cmd` — peak/s5cmd:v2.3.0 container; uses `s5cmd cp`, `s5cmd ls`, `s5cmd rm`; asserts exit 0 |
| Conformance — rclone | `TestConformance/*/Compat_Rclone` — rclone/rclone:1.68 container; inline rclone config injected as env; uses `rclone copyto`, `rclone ls`, `rclone deletefile`; `--s3-copy-cutoff=0` flag applied for CopyObject; asserts exit 0 |
| Conformance — minio-py | `TestConformance/*/Compat_MinIOPy` — python:3.13-slim container; pip-installs minio inline; asserts `minio-py:OK` marker in stdout and exit 0 |
| Unit — runner infrastructure | `TestRunToolContainer_ExitNonZero_ReturnsError` — inject a script that exits 1; assert `runToolContainer` returns an error |
| Unit — runner infrastructure | `TestRunToolContainer_MissingMarker_ReturnsError` — inject a script that exits 0 but omits the OK marker; assert `AssertOutput` returns error |
| Unit — capability bits | `TestCapabilities_NewBitsPresent` — assert that all 6 new capability bits can be set/unset independently without aliasing |
| Matrix guard | `TestConformance_NoProviderNameLiterals` (existing) — automatically validates that the 6 new test functions contain no provider-name literals; no new code required |

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Docker image pulls slow down CI | Pin exact image digests (e.g. `python:3.13-slim@sha256:...`) in `compat_runners.go`; Testcontainers caches layers between runs. Set `timeout-minutes: 45` in the CI job. |
| Tool container cannot reach the in-process gateway | The gateway harness listens on `0.0.0.0` (not `127.0.0.1`); pass the host's Docker bridge IP (`172.17.0.1` on Linux) as `GATEWAY_ENDPOINT`. Add `harness.StartGateway` option to bind on `0.0.0.0:0` and advertise the bridge address. |
| boto3/minio-py pip install latency | Use `python:3.13-slim` as base; cache `pip install` by pre-building a thin image in the same Docker network. If latency is unacceptable, publish a `ghcr.io/cloud37/test-boto3` image pinned to a specific boto3 version. |
| rclone server-side copy caveat silently passes without testing the caveat | Write a dedicated `testCompatSmoke_Rclone_CopyObject_NoCutoffFlag` negative case that runs without `--s3-copy-cutoff=0` and asserts the result; document the required flag in `SDK_COMPATIBILITY.md`. |
| s5cmd ETag expectations diverge from gateway-reported ETag | Gateway returns `x-amz-meta-encryption-original-etag` for encrypted objects. Run `s5cmd ls -e` (which prints ETags) and assert the ETag is non-empty; do not assert its specific value (it is the ciphertext ETag, not the plaintext MD5). Document this caveat. |
| External provider (AWS S3) compat tests consume cost in CI | Compat tests against external providers run only on `schedule` / `release` triggers (existing `conformance-external` pattern). The `compat-matrix` CI job explicitly sets `GATEWAY_TEST_SKIP_EXTERNAL=1`. |
| New capability bits alias with future bits | Bits 20–25 are assigned; document the next available bit (`1 << 26`) in a comment in `capabilities.go` to prevent accidental reuse. |
| Matrix guard fires on tool name literals in scripts | Tool scripts are inline strings in `compat_runners.go`, not test function bodies. The matrix guard only inspects test bodies (functions starting with `Test`). The script strings live in `Script()` methods on runner structs, which are not test functions and are therefore not subject to the guard. Verified by reading `matrix_guard_test.go:48`. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

- **AWS SDK v1 deprecation notice** — SDK v1 is excluded (§2 Non-Goals) because it
  reached security-patch EOL July 2025. A follow-up should add a note to
  `docs/SDK_COMPATIBILITY.md` pointing users to the SDK v2 migration guide.
- **AWS SDK Java v2 / AWS SDK JS v3** — Two widely-used SDKs not in the initial
  matrix. Deferred: they require JVM or Node.js Docker images and add significant
  CI time. Track as a separate issue in `v1.x`.
- **Presigned URL smoke test per tool** — awscli, boto3, and s5cmd all support
  generating and consuming presigned URLs. Deferred to V1.0-S3-1 which already
  covers presigned URL conformance.
- **CapBoto3Multipart / CapRcloneMultipart fine-grained bits** — Current design
  gates multipart on the existing `CapMultipartUpload`. If a specific tool fails
  multipart but passes single-object operations, a finer-grained bit may be
  needed. Defer until a failure is observed.

## 12. O'Reilly Verification Gaps (2026-06-08)

> Added by plan-verifier after cross-referencing with O'Reilly authoritative sources.
> Status as of 2026-06-09: all gaps resolved.

### GAP-COMPAT1-1 — ✅ RESOLVED — HIGH — `boto3Runner` and `minioPyRunner` scripts missing pip install step

**Source:** Python packaging documentation — `python:3.13-slim` does not include boto3 or minio packages.

**Description:** The `boto3Runner.Script()` code sketch (Phase B2) begins with `import os, boto3` but the specified Docker image `python:3.13-slim` does NOT have boto3 pre-installed. The script will fail immediately with `ModuleNotFoundError: No module named 'boto3'`. The same issue applies to `minioPyRunner`. The test strategy table and risk section acknowledge pip install latency but the actual code sketch omits the install step.

**Required fix:** Prepend the `boto3Runner` script body with `import subprocess; subprocess.run(['pip', 'install', 'boto3', '-q'], check=True)` before `import boto3`. Apply the same fix for `minioPyRunner` with the `minio` package. Alternatively, specify a pre-built image in the risk mitigation and document its build process.

---

### GAP-COMPAT1-2 — ✅ RESOLVED — HIGH — `rcloneRunner` script spec relies on unsupported "inline config via env var" approach

**Source:** rclone documentation — rclone accepts individual `--s3-*` flags via env vars but not a complete `[remote]` config stanza as a single env var.

**Description:** Phase B2 says "creates an rclone config inline and runs copyto / ls / deletefile" and the test strategy table says "inline rclone config injected as env." However, rclone cannot receive a full multi-line config block as a single environment variable. Without a concrete script body, the implementor may produce code that fails with "unknown remote" errors.

**Required fix:** Replace the vague description with a concrete `rcloneRunner.Script()` body that uses rclone's `--s3-*` flag form directly (e.g., `--s3-provider Other --s3-access-key-id $AWS_ACCESS_KEY_ID --s3-secret-access-key $AWS_SECRET_ACCESS_KEY --s3-endpoint $GATEWAY_ENDPOINT`) to avoid needing a config file. Provide a full code sketch at the same level of detail as `boto3Runner`.

---

### GAP-COMPAT1-3 — ✅ RESOLVED — MEDIUM — Capability bits 20–25 not verified as unoccupied

**Source:** Plan §7 Risks — risk entry partially acknowledges this but only documents the *next* available bit, not the existing occupancy.

**Description:** The plan assigns new capability constants at `1 << 20` through `1 << 25` (Phase A1). The plan acknowledges in the risk table: "Bits 20–25 are assigned; document the next available bit (`1 << 26`)." However, it does NOT audit whether bits 20–25 are already occupied by existing constants in `test/provider/capabilities.go:9–61` before assigning them. If any existing constant occupies bit 20–25, the new constants will silently alias, causing spurious test execution.

**Required fix:** Add an explicit pre-condition in Phase A1: reproduce the last-assigned bit value from current `capabilities.go` (e.g., "Existing constants end at `1 << 19`; bits 20–25 are confirmed free as of commit X") and add a `TestCapabilities_NewBitsPresent` assertion that verifies no two constants share a bit value across the entire `Capabilities` type.

---

### GAP-COMPAT1-4 — ✅ RESOLVED — MEDIUM — Docker bridge IP resolution is hard-coded without specifying dynamic resolution strategy

**Source:** Docker networking documentation — bridge IP `172.17.0.1` varies across platforms (macOS, rootless Docker, GitHub Actions).

**Description:** `newUniqueTestEnv` is specified as routing through `harness.StartGateway` but does not specify how the bridge IP is determined programmatically. The risk section mentions `172.17.0.1` but hard-coding this will cause test failures on macOS (Docker Desktop VM bridge), rootless Docker, and in some GitHub Actions runner environments.

**Required fix:** Specify in `newUniqueTestEnv` that the bridge IP must be resolved dynamically (e.g., via `testcontainers.DockerHostname(ctx)` or by querying the container's `NetworkSettings.Gateway`) rather than hard-coded. Add a note in Phase B1 and risk mitigation that `172.17.0.1` is only valid for default Linux Docker networking.
