## 2. Design Goals & Non-Goals

### Goals

1. **govulncheck as PR gate** — every pull request targeting `main` runs `govulncheck ./...`; any known CVE blocks merge.
2. **gosec as PR gate** — every pull request runs `gosec`; HIGH findings must be triaged before merge.
3. **Trivy blocks critical CVEs on release** — the release workflow (`helm.yml`) runs a Trivy container scan; `CRITICAL` severity exits with code 1; `HIGH` produces a non-blocking warning in the release summary.
4. **Local parity** — developers can reproduce every CI security check locally via `make` targets; no CI-only tooling.
5. **SECURITY.md accuracy** — the "Supported Versions" table reflects the v0.6.x + v1.0.x release lines authorised by the issue.
6. **No new Dependabot dependency** — Renovate (already active) covers automated dependency updates; `.github/dependabot.yml` remains commented out as-is.
7. **Minimal noise policy** — initial `gosec` run triaged; false positives suppressed with `#nosec G<rule> // justification`; only actionable findings block PRs.

### Non-Goals

1. **SBOM / provenance attestation** — tracked separately in V1.0-OPS-1 (Supply-chain & Publishing).
2. **Runtime intrusion detection** — out of scope for this issue; no Falco/eBPF work.
3. **Renovate configuration changes** — Renovate is already active and configured; no changes needed.
4. **Fixing pre-existing `gosec` findings** — this plan provisions only the infrastructure and an initial triage pass; suppression of confirmed false positives is part of the implementation. Deep remediation of genuine findings is a separate security-fix issue if needed.
5. **Branch protection rule changes** — documented as a post-merge operator step; not automated by this plan.

## 3. Target Interface / Semantics / Behaviour

### 3.1 New CI Workflow: `.github/workflows/security.yml`

A dedicated security workflow consolidates `govulncheck` and `gosec` into a single job that runs on every PR and every push to `main`.

```yaml
name: Security Scanning

on:
  pull_request:
    branches: [ main ]
  push:
    branches: [ main ]

jobs:
  security:
    name: Security scanning
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-go@v6
        with:
          go-version-file: go.mod

      - name: govulncheck
        run: go run golang.org/x/vuln/cmd/govulncheck@latest ./...

      - name: gosec
        uses: securego/gosec@master
        with:
          args: '-exclude=G104 ./...'
```

**Rationale:** A dedicated `security.yml` file keeps security concerns orthogonal to conformance and performance concerns, makes it easy to add new scanners, and allows branch protection to target `security / Security scanning` as a required check without touching the `conformance.yml` job graph.

### 3.2 Trivy Step in Release Workflow

The `helm.yml` release workflow gains a Trivy scan step after the Docker image is built (if applicable) or using the GHCR image ref. Because `helm.yml` does not currently build a container image (it builds Go binaries and releases a Helm chart), the Trivy step runs against the published GHCR image using `image-ref`:

```yaml
- name: Trivy container scan
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: ghcr.io/${{ github.repository }}:${{ github.sha }}
    severity: HIGH,CRITICAL
    exit-code: '1'
    ignore-unfixed: true
    format: table
```

**Placement:** After the `upload-binaries-to-release` step, guarded by `if: steps.version_check.outputs.already_released == 'false'` so it only runs on new releases.

**Severity policy:**
- `CRITICAL` — `exit-code: '1'` (blocks the release job)
- `HIGH` — `exit-code: '1'` as well (conservative for a security product); operators may downgrade this to `'0'` after an initial triage window.

**`ignore-unfixed: true`** — suppresses findings for vulnerabilities that have no fix available yet, reducing noise for genuine zero-day gaps that cannot be actioned.

### 3.3 Makefile Targets

Two new targets are added:

```makefile
# gosec — run static analysis security scan
gosec:
	@echo "Running gosec static analysis..."
	@go run github.com/securego/gosec/v2/cmd/gosec@latest -exclude=G104 ./...

# trivy-scan — scan the local Docker image for vulnerabilities
# Usage: make trivy-scan IMAGE=ghcr.io/cloud37io/s3-encryption-gateway:latest
trivy-scan:
	@echo "Running Trivy container image scan..."
	@docker run --rm \
		-v /var/run/docker.sock:/var/run/docker.sock \
		aquasec/trivy:latest image \
		--severity HIGH,CRITICAL \
		--ignore-unfixed \
		$(IMAGE_NAME):$(IMAGE_TAG)
```

The existing `security-scan` target is updated to invoke both `govulncheck` and `gosec`:

```makefile
security-scan:
	@echo "Running govulncheck..."
	@govulncheck ./...
	@echo "Running gosec..."
	@go run github.com/securego/gosec/v2/cmd/gosec@latest -exclude=G104 ./...
```

## 6. Test Strategy

Security scanning workflows are verified differently from application code — there are no unit tests for CI YAML. Verification uses the following approach:

| Layer | Tests |
|---|---|
| CI integration — govulncheck | `TestCI_GovulncheckBlocks` — manually: pin a known-vulnerable version of `golang.org/x/crypto` in a feature branch and confirm `security.yml` fails; restore to current version and confirm pass |
| CI integration — gosec | Confirm `security / Security scanning` appears as a required check in branch protection settings after merge |
| CI integration — Trivy | Trigger a test release tag on a feature branch; confirm Trivy step executes and produces a scan report in the release summary |
| Local parity — govulncheck | `make security-scan` exits 0 on current codebase |
| Local parity — gosec | `make gosec` exits 0 after triage (phase B2) |
| Local parity — Trivy | `make trivy-scan IMAGE=<local tag>` executes Trivy against local build |
| SECURITY.md accuracy | Manual review: confirm `1.0.x` and `0.6.x` appear in the Supported Versions table |
| gosec suppression audit | `docs/security/gosec-suppressions.md` lists every `#nosec` annotation with G-rule, file:line, and justification |

**Commands to verify locally:**

```bash
# govulncheck
go run golang.org/x/vuln/cmd/govulncheck@latest ./...

# gosec (after triage)
go run github.com/securego/gosec/v2/cmd/gosec@latest -exclude=G104 ./...

# Full security scan (both tools)
make security-scan

# Trivy on local image
make docker-build
make trivy-scan

# Vet passes on all build tags
go vet ./...
go vet -tags=fips ./...
go vet -tags=hsm ./...
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `govulncheck` reports a CVE in a transitive dependency with no fix yet | Use `ignore-unfixed: true` for Trivy; document that `govulncheck` blocks only on known-vulnerable packages with a published patch; add a workflow-level `continue-on-error: false` toggle that can be overridden with a documented exception |
| Initial `gosec` run surfaces dozens of HIGH findings, blocking all PRs immediately | Phase B1 (adding gosec to CI) is merged **after** phase B2 (triage) is complete; triage pass suppresses false positives before the gate is activated |
| `aquasecurity/trivy-action@master` is a floating ref — upstream breaking changes can fail the release pipeline | Pin to a specific tag (e.g., `aquasecurity/trivy-action@v0.30.0`) after initial integration; add Renovate config to auto-update this action pin |
| Trivy scan requires the GHCR image to be published before scanning; `helm.yml` may not push an image | Verify `helm.yml` release job: if it does not build/push a Docker image, add a `docker buildx build --push` step before the Trivy scan, or change to scan a locally-built image using `input: <tarball>` mode |
| `securego/gosec@master` floating ref can cause silent behaviour changes | Pin to a specific release tag after initial onboarding; update via Renovate |
| SECURITY.md version table becomes stale with future releases | Add a `RELEASING.md` checklist item: "Update SECURITY.md Supported Versions table before tagging" |

## 10. Follow-ups / Out-of-scope Items Surfaced During Planning

1. **SBOM and provenance (SLSA)** — the Trivy scan complements but does not replace SBOM generation. This is tracked in V1.0-OPS-1 (Supply-chain & Publishing).
2. **Renovate action pinning** — `.github/workflows/` action refs (e.g., `actions/checkout@v6`) are already pinned by major version. After Trivy and gosec actions are added, Renovate should be configured to manage their pins via `"matchManagers": ["github-actions"]`. This is a follow-up configuration addition to `renovate.json`.
3. **`gosec` remediation for real HIGH findings** — if the initial triage (phase B2) surfaces genuine security issues (not false positives), those are tracked as separate security fix issues rather than blocking this plan.
4. **Branch protection enforcement** — adding `security / Security scanning` to required status checks in repository settings is a manual operator action post-merge; it is not automatable via code changes.
